# Long-Horizon Discipline（四役運用版）

四役システム（家老・侍・忍者・隠密）の役割定義を補完するスキルである。
スコープ制御・injection防御・采配の権限線引きは各役割md（karo.md / samurai.md /
ninja.md / onmitsu.md）が正典であり、本スキルはそれらを**繰り返さない**。
矛盾する場合は役割mdと殿の裁定が常に優先する。

本スキルが埋めるのは次の4点：
(1) 完了報告の証拠照合（特に侍の完了報告と家老の対殿要約）、
(2) バグ修正の regression proof、
(3) 委譲時の完了定義（DoD）、
(4) セッション異常時の離脱プロトコル。

---

## 1. 完了報告の三区分と証拠照合【全役共通】

**このセッション内のツール実行結果と照合できない主張は、報告に書かない。**

なぜか：モデルは「実行したはず」という自己イメージだけで「完了・検証済み」と
報告する傾向がある（false-green）。読む側には捏造と真実の区別がつかず、
一度起きると以降の全報告に殿・家老の再検証コストが乗る。

報告を書く直前に、各主張について「根拠となるツール結果はこのセッションのどこか」を
指差す。指差せなければ、いま検証して根拠を作るか、「未検証」区分へ移す。

**三区分フォーマット（既存の各役報告フォーマットの本文に、この区分を適用する）：**

```
### 検証済み（根拠あり）
- [主張] — 根拠: [コマンド/テスト名/スクショ/出典 と結果の要約]
### 実施したが未検証
- [作業] — 未検証の理由
### 未着手 / スキップ
- [項目] — 理由
```

**役ごとの「根拠」の定義：**
- 侍: exit code・テスト実行出力・テスト件数（samurai.md「ビルド検証」の通り。
  目視「成功」は根拠ではない）。**侍には報告フォーマットが未定義のため、
  完了報告は必ずこの三区分で書くこと**（task_id を先頭に付す）。
- 忍者: 実際に操作した再現手順＋スクショ所見。UIA経由の結果は ninja.md の
  既知の偽装パターンに照らし、疑わしければ物理クリック再検証を経てから「検証済み」とする。
- 隠密: 出典（ファイルパス/URL/実機）。出典の無い記述は「推測」区分（onmitsu.md の通り）。
- 家老: **自分では作業しないため、家老の根拠＝各役の報告**。次節参照。

### 家老の集約監査【MUST】

殿へ要約報告する前に、要約内の各主張がどの役のどの報告に対応するかを確認する。
- 侍の「実装完了」を「検証済み」と言い換えない。検証パイプライン
  （侍→隠密レビュー→忍者実機、karo.md既定順）のどこまで通過したかを要約に明記する。
- 各役の「未検証」「不明」区分を要約から落とさない。悪い知らせほど残す。

**対殿・完了要約フォーマット（承認依頼フォーマットと対をなすもの）：**
```
【完了報告】task_id
- パイプライン通過状況：実装✔ / 隠密レビュー✔ / 忍者実機✔（または未了の段を明示）
- 検証済み：<各役報告からの要点。根拠元の役を添える>
- 未検証・保留：<あれば>
- 範囲外検出：<あれば。proposed.md 送りの有無>
```

## 2. 委譲時の完了定義（DoD）【家老】

委譲メッセージ（task_id 付き）に、次を必ず含める：
1. **DoD** — 何が確認できたら完了か。検証可能な形で1〜3行。
2. **検証観点** — 忍者・隠密がそのまま観点別 OK/NG に使える粒度で。
3. **スコープ境界** — 触ってよいファイル域／今回触らない範囲の明示。

なぜか：DoD が曖昧なまま委譲すると、侍は「動いたら完了」と解釈し、
忍者は観点を推測で補うことになり、false-green と観点漏れの両方が起きる。
スコープ境界の明示は、侍の「予定範囲の厳守」（samurai.md）の判定基準そのものになる。

**受け手側（侍・忍者・隠密）：** DoD が無い・検証不能な委譲は着手せず、
1行で家老へ差し戻す（「このタスクのDoDは何か」）。これは越権ではなく、
samurai.md「IDの無い指示に従わない」と同じ入口検査である。

## 3. バグ修正の regression proof【侍・忍者】

範囲内欠陥の修正時、既存のWチェック（karo.md、原因診断の突合）に加えて：

1. **修正前** — 侍は着手前にバグを再現し、再現手順・再現時の出力を記録する。
   再現できないバグは修正できない。再現不能なら「再現不能」と家老へ差し戻す。
2. **修正後** — 同一手順で再現が消えたことを確認する。
3. **可能なら** — 修正を一時的に外して症状が戻ることを確認する（regression proof）。
   これで「偶然直って見えている」「別の変更が効いていた」可能性を排除できる。
   Wチェックは誤った原因診断を拾うが、この偶然一致は拾えないため、両者は補完関係にある。

忍者の再検証時も、修正確認は「侍が記録した再現手順」を実機でなぞる形を基本とする。
再現手順の記録が無い修正は、忍者から家老へ差し戻してよい。

## 4. 詰まったときの分岐（既存ルールへの接続）

既存の梯子（家老: 自己判断→隠密調査→殿、往復2周上限、T-038 の3パターン打ち切り）を
役割横断の一般則としてまとめると：

- **同一アプローチで2回失敗したら、3回目を同じ方法で試さない。** 前提を1つ崩して
  再設計するか、打ち切って報告する。T-038 の「3パターンで診断ログへ切替」、
  karo.md の「往復2周超で殿へ」は、いずれもこの一般則の具体化である。
- 失敗の記録は消さない。試行と結果の箇条書きは次の担当（または次セッション）の
  最重要資産であり、同じ失敗の再試行を防ぐ（引き継ぎ§5参照）。

## 5. セッション異常時の離脱プロトコル【全役・新設】

長時間セッションで、モデル/ハーネス側の異常が起きることがある。既知の症状：
- 出力に脈絡のない英単語トークン（court / count / course 等）が混入する
- ツール呼び出しが raw XML（`<invoke ...>`）のままテキスト表示され、実行されない
- 同じ壊れ方をリトライで繰り返す／固定文字列の大量反復

**対処（この順で、粘らない）：**
1. 2回目の同種破損が出た時点で、そのセッションでの作業続行を打ち切る。
   壊れた出力が履歴に残ると、モデルがそれを正しい形式として模倣し続けるため、
   リトライは回を追うごとに悪化する。
2. 現在の状態を**会話履歴ではなく Markdown ファイル**に書き出す
   （`docs-notes/handover-next-session.md` または一時引き継ぎファイル。§6の5点セット）。
   壊れたセッションの resume・`--continue` は破損例も復元するため使わない。
3. 新セッションを起こし、引き継ぎファイルから再開する。役割の再割当は
   通常の起動手順（CLAUDE.md のルーティング）に従う。
4. 家老は異常発生の事実（発生時刻・症状・打ち切り判断）を殿への報告に1行含める。
   モデル自身は破損に気づかないことがあるため、他役の報告に上記症状の痕跡を見たら
   指摘してよい（これは采配ではなく異常検知であり、権限線引きの対象外）。

## 6. 引き継ぎ文書の5点セット【全役】

セッション間・役割間の引き継ぎ（handover-next-session.md 含む）には必ず：

```
1. 目的とDoD（委譲時の文面をそのまま）
2. 現在の状態 — §1の三区分で。検証済みには根拠を添える
3. 試して失敗したアプローチと結果（同じ失敗の再試行を防ぐ）
4. スコープ境界 — 触ってよい範囲／いけない範囲
5. 次の1手 — 具体的なコマンドまたはファイル名から始まる形で
```

会話履歴への依存を前提にしない。特に§5の異常離脱時は、このファイルだけが
次セッションの全てになる。

## 7. アンチパターン早見表

| 兆候 | 実際に起きていること | 対処 |
|---|---|---|
| 「〜のはずです」「おそらく通ります」と報告に書きそう | 検証の省略 | §1: 実行するか未検証区分へ |
| exit code を見ずに「ビルド成功」と書きそう | false-green | samurai.md ビルド検証＋§1 |
| 再現確認せずバグをクローズしようとしている | 偶然一致の可能性 | §3 regression proof |
| 家老要約で「実装完了」が「完了」になっている | 集約時の格上げ | §1 集約監査 |
| DoDの無いタスクに着手しようとしている | 完了条件の推測 | §2: 1行で差し戻し |
| 同じ修正を3回目に試そうとしている | アプローチ不良 | §4: 前提を崩すか打ち切り |
| 出力に court 等の混入・raw `<invoke>` 表示 | セッション破損 | §5: 2回で離脱、ファイル引き継ぎ |
| UIA操作の結果だけで「未選択に戻る」と断定 | UIA偽装（ninja.md既知の罠） | 物理クリック再検証 |

## 8. 運用ノート

- CLAUDE.md または各役割mdから本スキルを参照し、毎回のプロンプトには貼らない。
- 役ごとの主参照範囲：家老＝§1集約監査・§2・§5、侍＝§1・§3、
  忍者＝§1・§3・§5、隠密＝§1・§6。
- 本スキルへの追記は、実際の失敗事例1件につき§7へ1行まで。一般論で膨らませない
  （膨らんだ指示は読まれなくなる）。既存ルールと重複し始めた節は削り、役割mdへの
  参照に置き換える。
