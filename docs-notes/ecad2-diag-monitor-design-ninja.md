# 実機検証セッション常時ログ機構 設計案（忍者）

最終更新: 2026-07-13（家老よりの殿直接指示を受け設計検討、着手）

## 0. 経緯・依頼内容

殿より、過去2回のVSCodeクラッシュ（VSCode本体が落ち、その上で動くClaude Codeセッションも
巻き込まれた事象。Crashpad等の通常記録は一切残らず＝強制終了に近い急死とみられる）が、
いずれも**忍者の実機検証中、それもダブルクリックを連続で行っていた時期**に発生していたとの
指摘があった（確証はないが有力な手がかり）。5セッション+ecad2実機検証の同時稼働という
高負荷状況も背景にある。

殿の明示指示：「**以後、実機検証中は必ずログを記録すること**」。

本書は家老よりの委任（2026-07-13 10:36）を受け、常時ログ記録機構の設計案をまとめたもの。
**本書は設計検討のみであり、helpers.ps1・ninja.md等への実装反映は行っていない**
（家老裁量で制度化可否を判断いただく前提）。

## 1. 現状確認（実測、本設計にあたり追加調査）

- VSCode本体プロセス名: `Code`（現在14個、親子ツリーで多階層に展開）
- Claude Code CLIプロセス名: `claude`（現在13個）
- 監視対象プロセス名はこの2つで実務上十分と判断（実測ベース、推測ではない）
- `Get-Counter '\GPU Engine(*)\Utilization Percentage'` は本機で利用可能（実測確認済み）
- `Get-Counter '\Processor(_Total)\% Processor Time'` も利用可能
- 論理CPU数: 20
- 既存の類似機構: `$env:TEMP\ecad2-crash.log`（Ecad2.App自身の未処理例外ハンドラが記録、
  SKILL.md 6節記載）。これはアプリ内部クラッシュ用であり、**VSCode/Claude Code側の外部クラッシュは
  対象外**——本設計が埋めるべき空白はここ。

## 2. 設計方針（3層構成）

既存の `ecad2-ui-automation` スキルは「フォーカス非占有」を最優先方針としてきた（0節既存記載）。
本ログ機構もこの方針を踏襲し、**ログ取得自体が新たな負荷・干渉源にならないよう軽量に保つ**
（サンプリング間隔を短くしすぎない、GPU/CPUカウンタ取得はtry/catchで失敗を許容する等）。

### 層1: 常時リソースサンプラ（バックグラウンドプロセス）

`Start-Ecad2App` 相当の手法（`Start-Process -WindowStyle Hidden`）で、検証セッションと**別プロセス**
として監視スクリプトを起動する。PowerShellツールはコマンド間でシェル状態を保持しないため
（helpers.ps1冒頭のコメント通り）、`Start-Job` ではなく独立プロセスとして立てる必要がある
（`Start-Ecad2App`がdotnetプロセスを独立起動し複数ツール呼び出しをまたいで生存させているのと同じ要領）。

- サンプリング間隔: 2秒（暫定案、負荷とログ粒度のバランス。要調整）
- 記録内容（1行1サンプル、タイムスタンプ付き）:
  - CPU使用率合計（`% Processor Time`）
  - 利用可能メモリ（`Available MBytes`）
  - GPUエンジン使用率合計（`GPU Engine(*)\Utilization Percentage`、取得失敗時はNA表記でスキップ）
  - `Code`・`claude`・`Ecad2.App`・`dotnet` 各プロセスの合計ワーキングセット・プロセス数
- 出力先: `$env:TEMP\ecad2-diag-session-<起動時刻>.log`（セッションごとに新規ファイル、
  上書きしない＝過去セッションとの比較を可能にする）

### 層2: 操作履歴ログ（クリック・キー送信系ヘルパーのbefore/after記録）

`Invoke-Ecad2ScreenClick` / `Invoke-Ecad2CanvasClick` / `Invoke-Ecad2CanvasRightClick` /
`Invoke-Ecad2CtrlScroll` / `Send-Ecad2Keys`（＝グローバル入力＝`SetCursorPos`/`mouse_event`/
`keybd_event`/`SendKeys`を使う経路。SKILL.md 0節で「フォーカス・カーソル強奪注意」と明記されている
関数群と一致）呼び出しの直前・直後に、同一ログファイルへ1行ずつ記録する:

```
2026-07-13T19:40:12.345 BEFORE Invoke-Ecad2CanvasClick RelativeX=700 RelativeY=370
2026-07-13T19:40:12.520 AFTER  Invoke-Ecad2CanvasClick
```

**現行helpers.ps1にダブルクリック専用関数は存在しない**（`Ecad2Native.Click`は単発クリックのみ）。
殿の言う「ダブルクリックを連続で行っていた」がどの操作を指すか（同一箇所への`Invoke-Ecad2CanvasClick`
連続呼び出しか、別の操作か）は本書の時点では不明——**推測で断定しない**。層2はグローバル入力を
使う関数**全て**をbefore/after記録対象とすることで、この不明点を機構面でカバーする設計とした
（特定の呼び出しパターンだけを狙い撃ちせず、網羅的に記録する）。

### 層3: プロセス消失検知トリガ（クラッシュ疑い時の即時詳細スナップショット）

層1の監視ループ内で、`Code`・`claude`プロセスのPID一覧をサンプルごとに保持し、**前回サンプルに
存在したPIDが今回消えている**場合（かつ正常終了を示す目印がない場合）、通常の定期サンプルとは別に
即座に詳細スナップショットを追記する:

- 消失直前・直後の全プロセス一覧（`Get-Process | Sort-Object CPU -Descending`、上位20件程度）
- 消失PIDの一覧
- 直近の操作履歴ログ（層2）の末尾N行

これにより「次回VSCode/Claude Codeが巻き込まれて落ちた際、直ちに状況を記録できる体制」
（殿指示4点目）を満たす。通常の定期サンプル（層1）だけでは、クラッシュの正確な瞬間を
逃す可能性があるため、層3のイベント駆動記録で補完する。

## 3. 運用フロー（案）

1. `Start-Ecad2App` の直後に `Start-Ecad2DiagMonitor` を呼ぶ（検証セッション開始とログ開始を対にする）
2. 検証中、グローバル入力系ヘルパーは内部で自動的に層2ログを記録する（呼び出し側は意識不要）
3. 検証終了時、`Stop-Ecad2App` の前後で `Stop-Ecad2DiagMonitor` を呼ぶ
4. 万一VSCode/Claude Codeが落ちた場合、次回起動時にログファイル
   （`$env:TEMP\ecad2-diag-session-*.log`）を確認し、消失直前の層1定期サンプル推移・層3の
   即時スナップショット・層2の直近操作履歴を突合する

## 4. 未確定事項・要確認（推測で埋めず、確認待ちとして明記）

- サンプリング間隔2秒は暫定案。ログ肥大化と検知精度のトレードオフにつき、長時間セッションでの
  実測を経て調整が要るかもしれない
- 「ダブルクリックを連続で行っていた」の具体的操作内容は未特定（2節参照）。もし特定の検証タスクの
  心当たりがあれば、家老・殿からの追加情報を歓迎する
- GPUカウンタは環境依存（本機では利用可能と実測確認済みだが、他機での再現性は未確認）
- 監視プロセスをどこまでの粒度で追うか（`Code`・`claude`の子プロセス全てか、メインプロセスのみか）
  は実装時に要検討
- 本機構自体のオーバーヘッドが新たな不安定要因にならないか、導入後の実地検証が要る

## 5. 実装への反映（家老裁量、本書は着手前の設計案）

`helpers.ps1`への追加候補（関数名・シグネチャ案のみ、実装は侍/家老裁量）:

- `Start-Ecad2DiagMonitor` / `Stop-Ecad2DiagMonitor`（層1、独立プロセス起動/停止）
- `Write-Ecad2DiagLog`（層2、既存のグローバル入力系関数内部から呼ぶ内部ヘルパー）
