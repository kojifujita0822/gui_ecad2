## 1. 実装確認（`helpers.ps1` 該当箇所、再現実験前の静的確認）

```powershell
function Send-Ecad2Keys {
    param([Parameter(Mandatory)][string]$Keys)
    Set-Ecad2Foreground
    [System.Windows.Forms.SendKeys]::SendWait($Keys)
    Start-Sleep -Milliseconds 200
}
function Set-Ecad2Foreground {
    $proc = Get-Ecad2Process
    if (-not $proc) { throw "Ecad2.App process not found." }
    [Ecad2Native]::SetForegroundWindow($proc.MainWindowHandle) | Out-Null
    Start-Sleep -Milliseconds 300
}
```

**気づき（未確定・仮説）**:
- `Set-Ecad2Foreground` は常に `proc.MainWindowHandle`（＝Ecad2.Appのメインウィンドウ）を
  対象に `SetForegroundWindow` を呼ぶ。もし「開く」ダイアログ（OpenFileDialog、別トップレベル
  ウィンドウ）が表示中にこれを呼んだ場合、モーダルダイアログの親をアクティブ化しようとする形になる。
  Windowsの通常仕様ではモーダル子がアクティブな間は親を直接アクティブにできないはずだが、
  何らかの理由で成立してしまうとフォーカスの所在が不定になり得る。
- .NET の `SendKeys.SendWait` は対象ウィンドウへの送信に失敗する条件下で、内部的に
  ジャーナルフック（システム全体のキーボード入力をフックする方式）へフォールバックすることが
  知られている（.NET既知の制限）。ジャーナルフックが正しく解除されない場合、システム全体の
  キーボード入力が固まったように見える動作と整合する。
- 上記2点はいずれも**未検証の仮説**。実機再現で裏を取る。

## 3.5 コード側原因調査（Enter実験保留中、家老指示によるリスク低い追加調査）

- 対象: `src\Ecad2.App\MainWindow.xaml.cs` の `OpenButton_Click`（259-264行目）
  ```csharp
  private void OpenButton_Click(object sender, RoutedEventArgs e)
  {
      if (!ConfirmDiscardIfDirty()) return;
      var dialog = new Microsoft.Win32.OpenFileDialog { Filter = GcadFileFilter, DefaultExt = ".gcad" };
      if (dialog.ShowDialog(this) != true) return;
      ...
  ```
- **アプリ側実装に不備なし**: `ShowDialog(this)` と owner（MainWindow自身）を正しく渡す標準的な
  WPF実装。他の`ShowDialog()`呼び出し箇所（175/185/196行目）も同様の定型パターンで、特異な点なし。
- **原因は検証ヘルパー側にあると推定**（アプリのバグではない、という結論に傾く）:
  - `Microsoft.Win32.OpenFileDialog.ShowDialog(owner)` は内部でWin32のコモンダイアログを呼び出し、
    通常 `owner` ウィンドウを `EnableWindow(hwnd, FALSE)` で無効化してモーダル制約を実現する
    （標準的なWin32モーダルダイアログの実装パターン）。
  - `EnableWindow(FALSE)` は対象ウィンドウのマウス・キーボード**入力の受理**を無効化するが、
    `SetForegroundWindow` によるアクティブ化・最前面化そのものを妨げるわけではない（別レイヤーの
    API）。**通常のユーザー操作**（無効化されたメインウィンドウをクリックする等）では入力自体が
    無視されるためフォーカスが移ることはないが、`Set-Ecad2Foreground` ヘルパーが行う
    **直接の`SetForegroundWindow(メインウィンドウハンドル)`呼び出しは、この無効化状態を無視して
    フォアグラウンド化を成立させてしまう**（本セッションの実測: `GetForegroundWindow()`が
    実際にメインウィンドウを指した、で裏付け済み）。
  - 結果、「無効化されているのにフォアグラウンドかつアクティブなウィンドウ」というWindowsが
    通常のGUI操作では起こり得ない状態が人為的に作られる。この状態で`SendKeys.SendWait`が
    キーストロークを送ろうとすると、送信先（無効化されたメインウィンドウ）が実際には入力を
    処理できないため、.NET内部のフォールバック機構（対象が応答しない場合のジャーナルフック
    切替、既知の.NET制限）が働き、それがシステムワイドなキーボード入力凍結として観測された
    可能性がある。
  - **結論（暫定）**: P-056はEcad2.App本体の実装バグではなく、**検証ヘルパー
    `Send-Ecad2Keys`（内部で`Set-Ecad2Foreground`を無条件呼び出し）を、モーダルダイアログ表示中の
    メインウィンドウに対して使ったこと自体が引き金**と考えられる。通常のユーザー操作フロー
    （マウス・キーボードで自然に操作する場合）では、無効化されたウィンドウへの入力は最初から
    無視されるため、この状態には陥らないと推測される。
  - **再発防止の方向性（実装は行わず、方向性のみ記録）**: ダイアログ等のモーダル表示が想定される
    操作の前後では`Send-Ecad2Keys`（`Set-Ecad2Foreground`経由の強制フォアグラウンド化）を使わない、
    または`Set-Ecad2Foreground`側でEnumWindowsによる自プロセスの他ウィンドウ（モーダルダイアログ）
    存在チェックを挟み、あればそちらを優先してフォアグラウンド化する改修が考えられる。
