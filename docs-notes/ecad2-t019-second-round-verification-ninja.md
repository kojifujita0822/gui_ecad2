## 操作上の教訓（今回判明した罠、次回以降のため記録）

1. **`Send-Ecad2Keys`をモーダルダイアログ表示中に使うと危険**: 内部で`SetForegroundWindow`をメイン
   ウィンドウへ呼ぶため、開いているダイアログとの整合性が崩れる（キー入力が意図しない対象に届く／
   ダイアログが閉じる等）。SaveFileDialog・OpenFileDialog・要素配置ダイアログへの入力は、
   `Set-Ecad2Foreground`を経由しない直接`[System.Windows.Forms.SendKeys]::SendWait()`を使うこと。
2. **Win32ネイティブMessageBox（`確認`等）のボタンはUI AutomationでControlType.Paneとして現れ、
   InvokePattern等をサポートしない**。`Invoke-Ecad2Element`はFalseを返す。物理クリック
   （`Invoke-Ecad2ScreenClick`、ボタンの絶対座標中心）で対応する必要がある。
3. **モーダル子ウィンドウ（ダイアログ）は`Get-Ecad2Root`（メインウィンドウハンドル起点）のFindAllでは
   検出できない**。`GetForegroundWindow()`で現在アクティブなウィンドウハンドルを取得し、それを起点に
   `AutomationElement.FromHandle()`する必要がある。デスクトップRootElementからのProcessId検索も
   タイミングによっては検出できないことがあった（実際に存在するのに見つからないケースを経験）。
4. **WPFのTextBoxへ`ValuePattern.SetValue()`しただけでは、バインディング(`UpdateSourceTrigger=LostFocus`
   の場合)が即座に反映されないことがある**。配置ダイアログのデバイス名欄で、値設定直後にOKボタンを
   Invokeしても要素が配置されない事象が複数回発生。Tabキーでのフォーカス移動を挟んでも改善しないケースが
   あり、根本原因は未特定。**同一セルへの再配置を試みて「選択したセルには既に要素があります」の
   メッセージ有無で間接的に配置成否を判定する**フォールバックで対応した。
5. 上記4の影響で、配置した要素が機器表(DeviceTableGrid)に自動反映されるかは今回確認できていない
   （範囲外、今回の観点ではない）。
6. **同名プロセスが複数存在するとヘルパー関数一式が破綻する**: 切り分けのためgit worktree版
   Ecad2.Appを別途起動したところ、`Get-Process -Name "Ecad2.App"`が2件ヒットし配列を返すようになり、
   `Get-Ecad2Process`に依存する全ヘルパー（`Get-Ecad2Root`/`Get-Ecad2WindowRect`/`Set-Ecad2Foreground`等）
   が軒並み例外を投げた。worktree等で検証用に別インスタンスを起動する際は、作業完了後速やかに
   `Stop-Process`で個別終了し、単一プロセスの状態に戻すこと。
