# T-038 診断ログ連携運用メモ（起草・忍者）

UIA操作による実機検証が「UIA操作特有の限界なのか、実装側の問題なのか」の判別に難航した際、
忍者が一人で試行錯誤を重ねず、早期に家老・侍と連携して診断ログ注入へ切り替えるための運用案。
T-036観点6（デバイス名反映バグ）の実体験を基に起草する。殿承認後、`docs-notes/roles/ninja.md`
等への正式反映は家老が別途采配する。

## 背景（T-036観点6の顛末）

配置時の機器表反映（T-036本体）は観点1-5全OKだったが、観点6（プロパティパネル経由のデバイス名
編集）で、TextBox自体には入力値が反映されるのに機器表・キャンバス双方に反映されない現象を検出。
`ValuePattern.SetValue`／`SendKeys`+別要素クリック／`SendKeys`+Tabキー、と操作手法を3パターン
試したがすべて同一症状で、UIA操作固有の限界か実装側の問題か拙者からは断定できず「推測を断定
せず両論併記」の形で家老へ報告した。最終的に、侍が診断ログを仕込み、殿の実操作でログを実測した
結果、`UpdateSourceTrigger=LostFocus`（論理フォーカス）がキャンバス側の独立FocusScopeを跨ぐと
発火しないという構造的な原因が確定した（コミットab7cf7e）。忍者のUIA操作だけでは、この
「FocusScope跨ぎ」という内部構造の問題を外部観測だけで確定させることはできなかった。

## 1. 切替の判断基準

以下のいずれかに該当したら、それ以上UIA操作のパターンを増やさず診断ログ注入へ切り替える目安とする。

- **手法を変えても症状不変**: 異なる操作手法（`ValuePattern.SetValue`・`SendKeys`・物理クリック
  ・Tab/Enter等の異なるフォーカス移動手段）を3パターン程度試し、いずれも同一症状が続く場合。
  手法を増やしても新情報が出ない＝収穫逓減のサイン。
- **フォーカス/Binding/非同期処理が絡む症状**: UIAは外部から見えるプロパティ・イベントしか
  観測できない。`UpdateSourceTrigger`・`FocusScope`・`Dispatcher`経由の遅延処理等、内部の
  実装構造に起因する症状は、UIA操作の試行錯誤だけでは原理的に確定させにくい。
- **対照実験で明暗が分かれた場合はそこで打ち切ってよい**: 今回のように「経路Aは動く・経路Bは
  動かない」という対照結果が得られたら、それ自体が貴重な手がかり。無理に経路Bの原因を追わず、
  対照結果を添えてそのまま報告する方が効率的（拙者はこれを実践し、結果的に早い確定に寄与した）。
- **ninja.mdの既知の偽装パターンに類似**: UIA経由のInvoke/Select等が内部状態を不安定化させる
  既知の罠と類似した挙動（偽の成功/失敗、内部状態と表示の不一致）が疑われる場合。

## 2. 連携手順

1. **忍者が一行申告**: 現象・試行済みパターン・対照結果（動く/動かない）を簡潔に申告する。
   断定せず「UIA限界の疑い」「実装側の疑い」を両論併記し、詳細は検証記録ファイルのパスを添える
   （[[feedback_summarize_peer_messages]]の要約流儀を踏襲）。
2. **家老が侍へログ注入を采配**: 忍者の申告を基に、該当する処理経路（メソッド・イベント
   ハンドラ）を指定して侍へ診断ログの仕込みを依頼する。
3. **侍が診断ログを一時実装**: 下記「3. ログの標準形」に従い、対象箇所へ最小限のログ出力を
   仕込む。本実装とは別物であり、コミットに含めない運用とする（原因確定後に削除、本修正のみ
   コミットする）。
4. **操作者の使い分け**: まず忍者が従来のUIA操作で再現を試み、ログが発生するか確認する。
   - ログがUIA操作でも正常に発生する（＝UIA操作自体は問題なく現象を誘発できている）場合は、
     忍者がそのままログ回収まで完結できる。
   - UIA操作特有の何か（フォーカスの与え方、イベント発火順序等）が疑われる場合は、殿の実操作
     （物理キーボード・マウス）でのログ採取に切り替える。UIA操作と実操作の両方でログを比較
     できれば、「UIA操作固有の問題」か「実装の構造的問題」かをその場で切り分けられる。
5. **ログ回収・解析**: ログは加工・要約せず生データのまま報告する（[[feedback_diagnostic_log_escalation]]
   と同じ方針）。侍が実装側の理論と突き合わせ、隠密が独立に理論再検証するWチェック方式
   （T-021踏襲）が有効。
6. **原因確定後**: 侍が本修正を実装し、診断ログは削除。忍者が再検証して報告フォーマット通りに
   締める。

## 3. ログの標準形

- **パス**: `%TEMP%\ecad2-diag.log`（固定、複数回の試行を1ファイルに蓄積してタイムライン比較
  できるようにする）。
- **形式**: 追記式（既存内容を消さない）。各行はタイムスタンプ＋イベント名＋関連状態値。
  例: `[2026-07-05T18:00:00.123] DeviceNameBox.LostKeyboardFocus fired: OldFocus=DeviceNameBox, NewFocus=CanvasArea`
- **記録すべき箇所**:
  - **入口**: 対象メソッド・イベントハンドラの呼び出し自体（呼ばれたかどうかが最初の切り分け点）
  - **分岐**: if/elseやガード条件の判定結果（どちらへ分岐したか）
  - **状態遷移**: 値の変化前後（例: `DeviceName: old="M1" new="M1REN"`）
- **後始末**: 本実装確定後、侍がログ仕込みコードを削除する（恒久的な冗長ログ化を防ぐ）。

## 4. 実例（T-036観点6、原因確定までの実際の流れ）

1. 忍者がプロパティパネル経由のデバイス名編集を3パターン試行、いずれも機器表・キャンバス未反映。
   ダイアログ経由の入力（観点1-5）は正常という対照結果とともに、断定せず家老へ報告。
2. 家老が侍へ診断ログ注入を采配。
3. 侍が`DeviceNameBox`周辺（Binding更新・フォーカスイベント）へログを仕込む。
4. 殿の実操作でログを実測し、`UpdateSourceTrigger=LostFocus`がキャンバスの独立FocusScopeを
   跨ぐと発火しないという構造的原因が確定（隠密の理論的見立てとも一致）。
5. 侍が`UpdateSourceTrigger=Explicit`化＋`LostKeyboardFocus`・Enterキーでの明示`UpdateSource()`
   呼び出しへ修正（コミットab7cf7e）。
