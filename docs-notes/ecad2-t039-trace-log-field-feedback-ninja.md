# T-039 操作トレースログ基盤への現場意見（忍者）

隠密の実装方式比較検討に先立ち、UIA実機検証で実際に困った経験を基に率直な現場感を述べる。

## (1) 「これが記録されていれば即断できたのに」と感じた場面

- **T-036観点6（今回）**: プロパティパネルのデバイス名編集が反映されない現象。`ValuePattern.SetValue`
  ／`SendKeys`+Tab等、操作手法を3パターン変えても同一症状で、UIA操作固有の限界か実装側の問題か
  外部観測だけでは断定できなかった。フォーカス遷移(`LostFocus`/`LostKeyboardFocus`の発火有無・
  タイミング)とBinding更新のログさえあれば、その場でFocusScope跨ぎの構造問題と即断できたはず。
- **ninja.mdの既知の罠「UIA経由Invoke/SelectがClickハンドラを迂回する」**（T-016で複数ボタン
  同時ハイライトの実例あり）: UIA Invokeが本当にButtonのClickイベントを発火させたのか、それとも
  内部状態だけ変えて表示とズレたのか、Clickハンドラの発火有無がログにあれば、毎回「疑わしければ
  物理クリックで再検証」という二度手間を踏まずに済む。
- **モーダルダイアログ表示中の`Send-Ecad2Keys`の罠**（スキル0節に明記済み、今回もT-015検証で
  一度踏んだ）: キー入力が実際にどのウィンドウ/どの要素に届いたか(フォーカスを持つウィンドウ
  ハンドル)が記録されていれば、都度スキル文書を思い出す代わりにログで即座に「届いていない」と
  確認できる。
- **T-015での「行1(最上段)への配置で母線が描画されない」現象**: 深掘りしなかったが、
  `RedrawCanvas()`呼び出しの有無や描画範囲計算のログがあれば、範囲外の気づきとして留めるだけで
  なく、もう一段具体的な情報を添えて報告できた。

## (2) 記録内容の優先順位

実際に困った頻度・重大度が高い順:

1. **フォーカス遷移**（`FocusedElement`変化、`LostFocus`/`LostKeyboardFocus`の発火有無とタイミング）
   ——T-036を直撃。UIAでは原理的に観測しづらい領域。
2. **Binding/状態更新**（`PropertyChanged`発火、`UpdateSource`のタイミング）——ViewModelの値が
   「いつ」実際に変わったかは外部から見えない。
3. **Tool/SelectedCell等の主要状態遷移**——T-016のボタン多重ハイライト等、UIの見た目と内部状態の
   齟齬を疑う場面で頻出。
4. **コマンド実行(Clickハンドラ発火の有無)**——UIA Invoke迂回の罠の切り分けに直結。
5. **生のキー/マウス入力(どのウィンドウ/要素が受けたか)**——モーダル罠の切り分け。

キー/マウス入力そのものより、「その入力の結果、内部で何が起きた/起きなかったか」の方が優先度が
高い。UIAは前者(入力を送ったこと自体)は忍者側からも把握できるが、後者(内部の反応)は把握できない
ため。

## (3) 「あっても困る」もの

- **高頻度・低有用のイベントの垂れ流し**（マウス移動座標、毎フレームの描画呼び出し等）はログが
  埋もれてノイズになる。ユーザー操作1回に対応する単位でイベントを絞るべき。
- **ミリ秒精度でないタイムスタンプ**は前後関係の判断ができず実質無意味。
- **セッションを跨いだ無区切りの追記**: 検証のたびにアプリを再起動するため、区切りなく蓄積すると
  過去ログとの混線で読みにくい（今回のセッションだけでも4回前後再起動した）。
- **自由記述形式**: grep/検索しづらい。`event=... element=... old=... new=...`のようなkey=value
  形式なら機械的に追いやすい。
- **常時ON前提の設計**: 通常の観点別OK/NG確認（多くは正常動作の確認で原因調査を要さない）まで
  常にログが肥大化すると、本当に必要な時のログが埋もれる。

## (4) 起動方式・回収方法への注文

- **既定OFF、フラグで有効化**: `ecad2-ui-automation`スキルの`Start-Ecad2App`に`-DiagLog`のような
  スイッチを足し、必要な時だけログモードで起動できるとよい。常時記録は(3)のノイズ懸念と相反する。
- **セッション区切り**: アプリ起動ごとにログをクリアするか、明確な区切り線(起動時刻等)を入れて
  ほしい。今回のような複数回再起動を伴う検証でも迷わず追える形に。
- **既存の回収慣習に揃える**: `$env:TEMP\ecad2-ui-automation-std{out,err}.log`と同じ置き場所・
  読み出し感覚で`$env:TEMP\ecad2-diag.log`を扱えると、スキル側に`Get-Ecad2DiagLog`のような
  ヘルパーを1つ足すだけで済み、学習コストが低い。
- **検証記録への抜粋しやすさ**: タイムスタンプで範囲を区切って該当区間だけ引用できる形式だと、
  検証記録ファイルへの転記が楽になる。

## 総括

UIAは「何を送ったか」は分かるが「内部で何が起きたか」が原理的に見えない。今回T-036で3パターン
試して埋まらなかった溝も、フォーカス遷移とBinding更新さえ見えていれば1回の検証で埋まっていた
はず。優先度としては、まずフォーカス/Binding/状態遷移の3点を最小限のkey=value形式で記録できれば、
UIA検証の往復回数はかなり減らせる見込み。
