## 観点別 OK/NG

### 1. ツールバーボタンの表示と動作 — OK
- 1段目ツールバーに「行を追加 (Ctrl+Shift+Up)」「行を削除 (Ctrl+Shift+Down)」ボタンが
  他の標準ツールバーボタンと同一の`ToolBarButtonStyle`(46x42)で表示される。
  **「大型様式」という程の特別な大型化はされておらず、既存ツールバーボタン群と統一されたサイズ**
  （実装が標準スタイルを継承する設計のため、これは意図通りと判断）。
- クリックで行数が10→11に正しく増加することを確認。
- 無効時（下限/上限到達）は`Opacity=0.35`の半透明表示になることを確認（後述）。

### 2. Ctrl+Shift+Up/Downキー動作 — OK
Ctrl+Shift+Down（11→10）・Ctrl+Shift+Up（10→11）とも正しく動作。ボタンと同一のコマンドを
正しく起動できている。

### 3. 上限60/下限1クランプ — OK
- 上限: 59行→60行で「行を追加」ボタンがIsEnabled=True→Falseへ切替。60行を超える追加は
  キー送出しても発生しない。
- 下限: 2行→1行で「行を削除」ボタンがIsEnabled=True→Falseへ切替。視覚的にも1行のみの
  表示（行番号1のみ、グリッド線1本）を確認。

### 4. 最終行に要素がある状態での削除拒否 — OK（部品・ワイヤブレーク両方で確認）
- 最終行(行3)に部品(コイルM1)を配置した状態で削除を試行 →
  StatusMessageに「最終行に要素があるため削除できません」が表示され、行数は3のまま維持。
- コイル削除後、同じ行にワイヤブレーク（配線分断）を配置した状態でも同様に拒否を確認。
- **対照実験**: 最終行が空の状態（行4追加後）では削除が正常に成功し3行へ戻ることを確認
  （拒否条件が「要素の有無」に正しく紐づいていることの裏取り）。

### 5. 削除後のSelectedCellクランプ — OK
最終行(行3/列5)を選択中に行削除 → 選択セルが「行2/列5」へ正しく移動（列は維持）。
視覚的にも選択ハイライトがグリッド範囲内に収まっていることを確認（c28ec40の修正が
実機でも機能している）。

### 6. 既存操作の回帰 — OK
- 部品配置（コイル）・削除、ワイヤブレーク配置・削除は観点4の検証過程で正常動作を確認。
- シート切替: 2枚目のシートを追加すると独立して既定10行で作成され、1枚目（行数を
  変更済み）へ戻ると変更後の行数（2行）が正しく保持されたまま表示される。シート間で
  行数状態が独立していることを確認、回帰なし。

## 範囲外の気づき（修正不要・報告のみ）

- **DeleteRowCommandが成功した場合でも、直前に表示された「最終行に要素があるため
  削除できません」というStatusMessageがクリアされずに残り続ける。**
  再現手順: (1)最終行に要素がある状態で削除を試み拒否メッセージを出す (2)要素を除去する
  か行を追加するなどして最終行を空にする (3)削除を実行→実際には成功する（行数は減る）が、
  ステータスバーには古い拒否メッセージがそのまま表示され続け、あたかも今回も拒否された
  かのように見える。
  実害は小さいが、ユーザーが誤解する可能性がある表示上の不整合。DeleteRowCommand成功パス
  （`sheet.Grid.Rows--`以降）でStatusMessageをクリアする一文が無い設計上の抜けと見受けられる。
