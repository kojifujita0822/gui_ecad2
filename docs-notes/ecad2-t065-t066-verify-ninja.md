# T-065・T-066 実機検証記録（忍者）

- 対象コミット: `fd67ed9`（隠密静的レビュー指摘4件修正後、往復1周目）
- 検証環境: `Start-Ecad2App`（セカンダリモニタ）、都度シート追加→a接点配置で機器表に行を作成し検証

## 結論（観点別）

### ①機器表グリッドの「型式」列のみ編集可能 — OK

- 型式セルはダブルクリックで編集モード（白枠TextBox）に入ることを確認。
- 機器名セル・種別セルはダブルクリックしても行選択（青ハイライト）になるのみで、
  編集用TextBoxは出現しない。編集不可であることを確認済み。

### ②型式セル編集中の新規行追加・行削除不可 — OK

- 型式セル編集中（未確定）、グリッド最下部に新規行入力プレースホルダは表示されない
  （`CanUserAddRows="False"`の効果、DataItem数=1のまま）。
- 同状態でDeleteキーを送信しても行は削除されない（DataItem数変化なし）。

### ③型式セル編集中(未確定)にCtrl+Sを押しても入力値が保持され保存されること — **NG（重大・再現性あり）**

**型式セルの編集有無に関わらず、Ctrl+S押下（「名前を付けて保存」ダイアログ表示）のタイミングで、
直近配置した機器がDeviceTableGridから消失する。** キャンバス上の図形（シンボル）自体は残り続ける
ため、機器表とキャンバス描画のデータが乖離する（幽霊図形化）。

**再現手順**:
1. シート追加→a接点配置→機器名確定（例: X1）。機器表に1行反映を確認。
2. （ツールを選択ツールへ切り替えず）Ctrl+S送信。
3. 「名前を付けて保存」ダイアログが開く。**ダイアログを閉じる前の時点で**、メインウィンドウの
   DeviceTableGridを確認すると、直近配置した機器が既に消えている（ダイアログのクローズ方法
   ＝Esc／キャンセルボタンクリックとは無関係、ダイアログ表示自体のタイミングで発生）。
4. Ctrl+Zでも復元されない（1回のUndoでシート追加操作自体まで巻き戻ることがあった＝Undo履歴に
   機器配置操作が正しく積まれていない可能性も併せて示唆）。

**再現性**: 本セッションで3回試行し3回とも発生を確認。
- 1回目: 型式セルに値入力（未確定）→Ctrl+S→機器表が完全に空（1行→0行）。
- 2回目: 2件目のa接点配置直後（ツール未切替）→Ctrl+S→Esc→2件中1件（最後に置いた方）が消失。
- 3回目: 3件目のa接点配置直後（ツール未切替）→Ctrl+S送信直後・ダイアログ開いたまま未クローズの
  時点で確認→既に1件減少していることを確認（ダイアログクローズ操作より前に発生）。

**未確認・切り分け不足**:
- 「選択ツールに切り替えてからCtrl+S」の場合に再現するかは検証時間の制約で確認できていない
  （2回目・3回目はいずれも配置ツールがアクティブなままCtrl+Sを送っており、これが誘因の可能性は
  あるが未確定）。
- 実際に保存を完了させた場合（ファイル名入力→保存ボタン）に同現象が起きるかは、ダイアログ操作の
  UIA操作難航（後述）により最後まで確認しきれていない。
- 出力パネル・stdout/stderrログにエラー出力はなし（クラッシュではない）。

### ④型式セルを値を変えずEnter/Tabで抜けた場合、未保存マークが立たないこと — **未検証（保留）**

③の重大な副作用（保存操作で機器表データが乱れる）により、IsDirtyを正確にfalseへ戻す手段
（保存→再オープンでのクリーン状態確認）が信頼できず、切り分けが困難と判断し保留。
別途、③の原因究明後に再検証が必要。

### ⑤ドキュメント情報ダイアログ：8フィールド編集・保存・キャンセル — OK

- 8フィールド（社名/タイトル/図番/客先/設計者/製図者/検図者/日付）全て編集内容が反映される。
- OKで保存後、値が保持されていることを確認（ダイアログ再オープンで確認）。
- 保存後、MarkDirty連動を確認（「新規作成」実行時に未保存確認ダイアログ「はい/いいえ/キャンセル」
  が表示された。裏取りとしてキャンセルで復帰）。
- キャンセルボタンで変更が破棄されることを確認（社名欄に入力→キャンセル→再オープンで空のまま）。

### ⑥ダイアログにRevisions(改定履歴)欄が含まれていないこと — OK

- 8フィールドのみでRevisions欄は存在しないことを確認。

## 補足：UI Automation操作上の障害（実装バグではなく操作手法の問題と判明）

「図面(D)」メニュー→「ドキュメント情報(I)」の`InvokePattern.Invoke()`が当初「認識できない
エラーです」で失敗した件は、`IsEnabled="{Binding HasProject}"`によりメニュー項目が無効化されて
いたため（起動直後はDocument.Sheets=0でHasProject=false、殿裁定済みの既存仕様）。シート追加後は
UI Automation経由のInvokeが正常に成功した。実装バグではない。
