## T-069（右クリックメニュー）

### 観点1: 記入中ドラフト保持中のツールバーボタン押下→キャンセル+右クリックメニュー動作 — OK

縦コネクタ（PlaceConnector、制御回路シート）・自由線（PlaceLine、主回路シート）・画像挿入
（PlaceImage）の3種いずれも、ドラフト保持中に「自作パーツ」ボタンを押すと正しくキャンセルされ
（ツール表示がPlaceElementへ遷移）、直後の右クリックで行操作メニューが正常に表示された。旧
HasAnyDraft残留バグ（メニュー完全無反応）の再発なし。

### 観点2: 連続配置中の無関係要素右クリック→キャンセル→作業起点保持 — SelectedCellはOK、Tool.Modeに新規所見（要確認）

手順: a接点配置ツールで2要素配置（行2/列1・行4/列1）→ツール継続中に無関係要素（行2/列1）を右クリック
→メニュー表示→キャンセル、で検証。

- **SelectedCell（作業起点）自体は常に保持される**。Esc・メニュー外クリックいずれの経路でも、
  メニュー表示前のSelectedCellが変わらないことを確認（隠密レビュー書`ecad2-t069-fix4-review-onmitsu.md`
  の修正2「表示と実行の整合原則」通り）。
- **新規所見（NG疑い、要確認）**: `Send-Ecad2Keys`（SendKeys経由）でEscを送りメニューをキャンセル
  すると、Tool.ModeがPlaceElement→**Select**へ変化する。一方、メニュー外の空白領域を物理クリックで
  閉じた場合はTool.Mode=PlaceElementのまま維持される。対照実験で経路により結果が分かれた：
  - Esc（SendKeys経由）→ ツール:Select、選択セル保持（家老の指示文言「Tool.Mode=PlaceElement継続」
    の期待と不一致）
  - メニュー外クリック → ツール:PlaceElement継続、選択セル保持（期待通り）
  - コード確認（`MainWindow.xaml.cs:1177-1181`）: 層2ロジックとして「配置モード中にEscでSelectへ
    戻す。SelectedCellは保持し続けて別ツールで同じセルへ配置し直せるようにする」という**意図的な
    仕様**が存在する。これは通常のPlaceElementキャンセル（配置中に単にEscで取消す操作）向けの設計と
    見られるが、**右クリックメニューを閉じるだけのEscがこの層2ロジックにまで連動してしまうのは
    意図しない副作用の可能性がある**。
  - `Send-Ecad2Keys`は内部で`Set-Ecad2Foreground`（`SetForegroundWindow`）を呼んでおり、ContextMenu
    表示中にこれを挟むことでメニューが想定外に閉じ、その後Escがメインウィンドウの通常キャンセル
    ロジックへ落ちている可能性がある（UIA操作特有の限界の疑い）。ただし、実装側のEscハンドラが
    本当にContextMenu由来のEscと通常のEscを区別できていない可能性も排除できない。
  - **忍者の手元では実装バグかUIA限界か判別不能**。`ninja.md`「殿代行操作の運用」の対象
    （キーボードショートカット自体の検証）に該当する可能性があり、必要なら殿の実操作での
    裏取りをご検討いただきたい。

## T-064（画像挿入）

### 観点1: リサイズの最小サイズ制約+境界クランプ両立 — 保留・確認待ち

座標クリック方式でのリサイズハンドルのドラッグに苦戦した。粗いステップ（8分割）ではハンドルの
当たり判定を外し「サイズ変化なし」または「選択解除+別セル選択」という誤った結果が出たが、
細かいステップ（20分割、各40ms、LEFTDOWN後200ms待機）に変更したところ通常のリサイズは正常に
機能することを確認（座標クリック手法の精度問題であり、実装側の問題ではないと判断）。

この手法でページ境界を大きく超える位置へ右下ハンドルをドラッグしたところ、画像サイズは変化
しなかった（クランプが効いている可能性のある観測）。ただし、境界のすぐ外側を狙った再試行では
ズーム40%表示下でハンドルを外し「画像選択解除+別セル(行11/列21)選択」という別の失敗パターンに
なり、クランプによる結果か操作失敗かの切り分けが不十分なまま時間を要したため、**本観点は確定的な
判定に至らず保留**とする。可能なら侍による単体テストでの担保状況を確認するか、次回検証時に
座標特定の精度を上げて再検証されたい。

### 観点2: リサイズハンドルのアンカー反対側超え→軸反転追従 — OK

右下ハンドルを左上アンカーを大きく超えて左上方向へドラッグしたところ、画像は破綻せず新しい矩形
（旧アンカー位置が新しい右下、ドラッグ先が新しい左上）に正しく収まった。境界に関係なく軸反転は
正常に機能。

### 観点3: グリッド外上部余白帯（Y<15mm程度）での画像右クリック — OK

上部余白帯（行1より上）に画像を配置し右クリックしたところ、「削除」メニューが正しく表示された。

### 観点4: 矢印キー/Esc/Delete/右クリックの画像ドラフト対称性 — Esc/Delete/右クリックはOK、矢印キーに新規所見

- **Esc（配置待機中のドラフトキャンセル）**: OK。ツール:PlaceImage→Select、何も生成されない。
- **Delete（画像選択中）**: OK。正しく画像が削除された。
- **右クリック**: OK（観点1・観点3で確認済み、削除メニューが正しく表示）。
- **矢印キー**:
  - 画像挿入ドラフト保持中（未確定、マウス追従状態）: 押下しても無反応（ツール状態・メッセージとも
    変化なし）。縦コネクタ/自由線はグリッドセルベースで矢印キー調整に対応するが、画像挿入はマウス
    位置追従の自由配置設計（殿裁定「案A」）のため、そもそも矢印キー操作が定義されていない可能性が
    高く、これ自体は仕様と推測（バグとは断定しない）。
  - **新規所見（要確認）**: 画像選択中（確定済み画像をクリックして選択した状態）に矢印キー（右）を
    押すと、**画像選択が解除され、代わりにセル「行1/列1」が選択される**。画像の移動や他の意図した
    動作ではなく、既定のセルナビゲーション処理へフォールスルーしているように見える。「対称性」の
    観点から見ると、縦コネクタ/自由線が矢印キーで記入中の調整に使うのに対し、画像は選択中の矢印
    キーで前触れなく別セルへ飛ぶという挙動差があり、バグか仕様か忍者では判断できない。
