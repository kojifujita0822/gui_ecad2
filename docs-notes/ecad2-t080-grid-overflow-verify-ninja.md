# T-080 グリッドはみ出し疑問 実測記録（忍者）

- 位置づけ：グリッドはみ出しは殿御自身の目視で実在確定済み（P-060起票済み）。本記録はNG検出目的
  ではなく、症状の詳細記録が目的（家老指示）。
- 検証環境：起動中のEcad2.App（T-080実装途中、コミット前の作業ツリーでビルド・実機確認）

## 手順・結果

1. 新規シート「T080検証」を作成し、左端付近（列0）にa接点「LEFT」、母線直前の最右列にコイル
   「RIGHT」を配置。
2. ステータスバーの「選択セル」表示から、母線直前の最右列は**列19**（列39ではない）と判明。
3. PDFプレビュー・実出力（`t080gridtest.pdf`、PyMuPDFで実測）とも、LEFT・RIGHTいずれの要素も
   枠内に完全に収まっており、**クリップは一切発生しなかった**。
4. プレビュー画面と実出力PDFの見え方を比較し、要素位置・母線位置とも一致（差異なし）。
5. ズームは終始100%表示で、縮小変換はかかっていない（実出力PDFも同様、スケール調整なし）。

## なぜ「列39」ではなく「列19」だったか（コード上の食い違い、気づき）

- `Sheet.cs:39`の`Columns`既定値は**40**だが、実際のシート作成コード
  （`SheetNavigationViewModel.cs:107`・`MainWindowViewModel.cs:1660`、いずれも
  `new GridSpec { Rows = 10, Columns = 20 }`）で**20に上書き**されている。今回作成した
  「T080検証」シートも列数20で作られたため、最右列は列19（0-indexed）だった。

## 用紙幅とグリッド計算の関係（数値整理）

`DiagramRenderer.cs`の定数：`CellMm=9.0`、`MarginMm=20.0`、A4幅=210mm、A3幅=297mm。

- 母線位置（RightBusX）= `MarginMm + Columns * CellMm`
- diagram理論全幅 = `RightBusX + MarginMm`（右余白=MarginMm分を確保しようとする計算、
  `DiagramRenderer.cs:139`のrightExtra、要素なしの場合）

| Columns | 母線位置 | diagram理論全幅 | A4(210mm)との関係 | A3(297mm)との関係 |
|---|---|---|---|---|
| 20（実際の設定） | 200mm | 220mm | 母線まで10mm余裕、diagram理論幅は10mm超過 | 大幅に余裕あり |
| 40（Sheet.cs既定値、侍が疑問視した数値） | 380mm | 400mm | 170mm超過 | 83mm超過 |

**解釈**：現在の実装（Columns=20固定）では、母線自体の位置（200mm）はA4用紙幅（210mm）に
収まっており、これが実測でクリップが発生しなかった理由。ただし、diagram全体の理論幅計算
（220mm、母線の右側にさらに20mm分の余白を確保しようとする値）はA4幅を10mm超過しており、
「右母線の右側に領域を追加する」T-080（行コメント機能）が、まさにこの窮屈な右余白領域を
使う設計だとすると、現状でも10mmしか余裕がなく、行コメント表示に必要な幅次第では容易に
クリップへ転化しうる。侍が懸念していた「40列」という数値は現行コードのどこにも実際には
使われていないが、この右余白の窮屈さ自体は列数に関係なく既に存在する構造的な問題と見る。
