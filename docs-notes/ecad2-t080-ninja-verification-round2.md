## 観点(1) (a)切り分け: ダブルクリック不成立の原因調査

### 手順と実測

1. 制御回路シート、右母線(P24)右側の領域(スクリーン座標概ね2885,280)へ単クリック
   - ログ: `ClickCount=1 posMm=(212.33,26.20) RightBusX=204.50 HitTestRungCommentRow=0`
   - **着弾位置はヒット領域内(xMm=212.33 > RightBusX=204.50)、正しくヒットしている**
2. UIA合成ダブルクリック試行(間隔60ms・15ms指定の2パターン。ただしSet-Ecad2Foregroundのオーバーヘッドにより実測間隔は約770msまで伸びていた)
   - いずれも`ClickCount=1`のまま、2回とも別イベントとして記録。`ClickCount==2`に到達せず
3. 殿へ物理ダブルクリックを依頼(同一スクリーン座標2885,280)
   - 実測: 2件の`LadderCanvasHost_PreviewMouseLeftButtonUp`ログ、いずれも`ClickCount=1`(間隔約273ms)、着弾座標は完全同一(`posDip=(802.50,99.04)`)、`HitTestRungCommentRow=0`で両方ともヒット領域内

### 判定: NG(実装欠陥、確定)

着弾位置がヒット領域内であることを実測で確認した(隠密調査書の「ケース1」が的中)にもかかわらず、
**殿の物理ダブルクリックでもWPFの`ClickCount`が2に到達しない**。これはUIA/座標クリック合成の
限界ではなく、実装またはWPFのイベントルーティング側の問題であることが確定した。原因の特定は
コード読解(侍・隠密マター)に委ねる。

## 観点(2) (b)確認: フォーカスロスト経路

### 手順と実測

1. F2でエディタを開く → `OpenRungCommentEditor row=0 sheet=シート2 MainCircuit=False`
2. UIA経由でツールバーボタン(`"選択ツール (Esc)"`)をInvoke試行 → 例外「認識できないエラーです」(無効化)。ログに変化なし(`LostKeyboardFocus`不発火)
3. UIA経由でシートリスト項目をSelect試行 → 同様に例外、ログに変化なし
4. 殿へ物理クリックを依頼(キャンバス内の通常グリッド領域、スクリーン座標概ね2560,390)
   - 実測ログ:
     ```
     RungCommentBox_LostKeyboardFocus NewFocus=null _rungCommentEditingRow=0
     CommitRungCommentEditor restoreFocus=False _rungCommentEditingRow=0
     CloseRungCommentEditor restoreFocus=False _rungCommentEditingRow=0
     ```
   - UIAツリー上も`RungCommentBox`要素が消失(検出0件)。スクリーンショットでも選択セルの赤枠
     ハイライトのみでエディタ本体は表示されておらず、**実際に閉じていることを確認**
     (`docs-notes`添付なし、スクラッチパッド`t080-after-physical-click.png`参照可)
5. 再度F2でエディタを開き直し(フォーカス回復のため一度キャンバスをクリックしてから)、
   今度はUIA合成クリック(`mouse_event`、キャンバス内の別位置)を試行
   - ログに変化一切なし(`LadderCanvasHost`のUp/Downイベントも`LostKeyboardFocus`も発火せず)。
     エディタはUIAツリー上も開いたまま(検出1件)

### 判定: 無効(測定条件不備・窓外クリック)、再検証要

UIA合成クリック(`mouse_event`)では無効化された`LadderCanvasHost`領域に対してイベントが
一切発火せず(ヒットテストにすら到達しない)、エディタは開いたまま。

**【訂正・要再検証】** 手順4の「殿の物理クリック」について、実際には殿が**ecad2アプリ
ウィンドウの外(忍者への返信のため別ウィンドウ)をクリックした**ことが事後判明した(殿ご本人の
訂正申告)。つまりこれは「アプリ内の他要素へクリックしてフォーカスを移す」操作ではなく、
**ecad2ウィンドウ自体が非アクティブ化された**ケースであり、隠密調査書が検証しようとした
現象(`MainContentArea`無効化下での**アプリ内**フォーカス移動)とは別物である。ログ上
`RungCommentBox_LostKeyboardFocus(NewFocus=null)`→`Commit`→`Close`が発火し実際に閉じた
ことは事実として確定しているが、これは「ウィンドウ非アクティブ化に伴うフォーカスロスト」の
実測であり、「アプリ内クリックでのフォーカスロスト」の実測ではない。

前回round1(`docs-notes/ecad2-t080-ninja-verification-round1.md`追記部分)の「実機の物理
クリックでもエディタが閉じない。NG確定」との関係も、比較対象が異なる(round1がどの操作を
指していたか不明瞭)ため、現時点では矛盾と断定できない。

## 総括

- **(a)ダブルクリック不成立**: 実装欠陥として確定(着弾位置はヒット領域内なのに`ClickCount==2`
  に届かない)。隠密調査書の推奨切り分け(ケース1/ケース2)は**ケース1(未知の欠陥)**で確定。
- **(b)フォーカスロスト**: **再検証が必要**。今回得られたのは「ウィンドウ非アクティブ化では
  `LostKeyboardFocus`が発火し正しく閉じる」という実測のみで、隠密の主仮説(`MainContentArea`
  無効化下でのアプリ内フォーカス移動不能により不発火)の検証にはなっていない。UIA合成クリック
  はアプリ内クリックとして無反応だったが、こちらもUIA/座標クリック合成側の限界であり実装の
  問題ではない可能性が高い。**アプリ内(ecad2ウィンドウをアクティブに保ったまま)の無効化領域への
  物理クリック**が未実施のまま。

## 観点(2) (b)再検証(家老采配、条件を厳密化)

### 手順

1. ecad2ウィンドウをアクティブに保ったままF2でエディタを開く(直後にPowerShell呼び出しを
   挟むとフォーカスが失われる現象を確認したため、F2送信は単独のPowerShell呼び出しで完結させ、
   状態確認はBashの`tail`のみで行う運用に変更。詳細は次項「気づき」参照)
2. 殿へ「ecad2ウィンドウの内側、ツールバー『選択ツール』ボタンあたり(スクリーン座標概ね
   1990,120)を物理クリックしていただきたい。アプリ窓の外は絶対にクリックしない、この操作の
   前後は他ウィンドウへの切替も控える」旨を明記して依頼
3. 殿より「クリックした(窓内のみ)」の回答を得た

### 実測ログ

```
RungCommentBox_LostKeyboardFocus NewFocus=null _rungCommentEditingRow=0
CommitRungCommentEditor restoreFocus=False _rungCommentEditingRow=0
CloseRungCommentEditor restoreFocus=False _rungCommentEditingRow=0
```

UIAツリー上も`RungCommentBox`要素は消失(検出0件)。スクリーンショット(`t080-toolbar-click-result.png`)
でも選択セルの赤枠ハイライトのみでエディタ本体は表示されておらず、実際に閉じていることを確認。

### 判定: 隠密の主仮説は反証された(確定)

**アプリウィンドウをアクティブに保ったまま、ウィンドウ内側の無効化領域(ツールバー)を物理
クリックしても、`RungCommentBox`の`LostKeyboardFocus`は発火し(`NewFocus=null`)、正しく
コミット・クローズされる**ことを、測定条件を厳密化した上で確定した。`MainContentArea`が
`IsEnabled=false`で無効化されていても、WPFは(フォーカス移動先が存在しなくとも)現在
フォーカスを持つ要素からフォーカスを外す動作を行い、それに伴い`LostKeyboardFocus`ハンドラは
正常に発火する。隠密調査書の主仮説(無効化によりフォーカス移動不能→不発火)は、この実測で
反証された。

前回round1「実機の物理クリックでもエディタが閉じない。NG確定」との関係は依然不明瞭だが、
今回の測定条件下((1)ウィンドウ内クリック (2)ウィンドウのアクティブ状態維持 の両方を厳密に
満たした状態)では「正しく閉じる」が再現性をもって確定した。round1のNG判定は測定条件が
異なっていた可能性が高い(推測であり断定はしない)。

## 気づき(範囲外・自動化運用上の知見)

F2キー送信(`Send-Ecad2Keys`)を含むPowerShell呼び出しの直後に、UI Automationクエリ
(`Get-Ecad2Root`/`FindAll`等)を含む**別のPowerShell呼び出し**を実行すると、その呼び出し
自体が原因と見られる`RungCommentBox_LostKeyboardFocus(NewFocus=null)`が数百ms後に自動発火し、
意図せずエディタが閉じる現象を確認した(2026-07-12実測、本検証中に2回再現)。F2送信と状態
確認を同一PowerShell呼び出し内に収めるか、状態確認をBashの`tail`等(PowerShell以外)に
限定すると発生しなかった。原因はPowerShellツール呼び出し自体がバックグラウンドで何らかの
ウィンドウ操作を伴う可能性が疑われるが未特定。ecad2-ui-automationスキルへの既知の罠追記を
推奨する(家老・侍の判断に委ねる)。
