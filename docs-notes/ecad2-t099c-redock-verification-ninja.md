# T-099(c) 実機確認記録（忍者、2026-07-19）

## 検証観点と結果

### 観点(1) ドッキング時、上段ラベルが視覚上非表示か
**OK**。PrintWindow撮影で上段（旧`AnchorablePaneTitle`、Opacity=0適用済み）に「配置ツール」という
テキストは視覚的に見えず、下段のTabItem帯のみが表示されている状態を確認（スクリーンショット
`t099c-initial.png`）。

### 観点(2) フロート化→再ドッキングの一連動作が両方成立するか（最重要）

**フロート化は成立**：Collapse化時代の重大回帰（フロート化自体が不成立になる）は解消されたと判断
してよい。「配置ツール」ラベル（Opacity=0適用済み、UIA上はText要素として存在しヒットテスト生存）
を掴んでドラッグし、`EnumWindows`で新規ウィンドウ（配置ツールバーのフロートウィンドウ）が生成
されることを複数回確認した。

ただし**操作の再現性にはやや揺らぎがある**：通常速度ドラッグ（40ステップ/40ms間隔、距離300+350px）
では2試行中1回のみ成功、低速ドラッグ（80ステップ/30ms間隔、終点500ms静止）では2試行中2回成功。
これがUIA操作固有のタイミング起因か、実装側の判定閾値の際どさに起因するかは不明。

**再ドッキングに新たな疑義**：フロート化直後、UIA探索でメインウィンドウ側のドッキング候補領域
（`ControlType.Tab`要素）を実測したところ、**幅88px・高さ6pxまで縮小**していることを確認した
（T-099要件2の根本原因分析＝「フロート化でペインが空になるとDockingManager自身のActualWidthが
実測25px程度まで縮む」という予測と整合するが、`MinWidth="100"`を指定したはずの実測幅が88pxと
下回っている点は要確認）。

この極小領域を狙って再ドッキングを試行：
- 通常速度ドラッグ（40〜60ステップ）で2回試行 → いずれも再ドッキング不成立（フロートウィンドウの
  ままカーソル追従のみ、`EnumWindows`の可視ウィンドウ数は2のまま）
- 低速ドラッグ＋終点1秒静止（ドッキングガイド表示を待つ意図）で3回目試行 → **可視ウィンドウ数は
  1に戻り再ドッキングは成立したが、レイアウトが意図したものと異なる**。元の横長単一ツールバー
  （幅569px）へ統合されるのではなく、**幅44px程度の縦長の別タブとして脇に追加され、ボタン列が
  ほぼ全てオーバーフローメニューへ収納される異常な見た目**になった（スクリーンショット
  `t099c-shrunk-pane-crop.png`参照、拡大画像は縦長の小さいボックスに下向き矢印のみ見える状態）。

この状態は明らかに検証開始前と異なるレイアウトのため、アプリを再起動しクリーンな状態に戻した上で
再現性を確認しようとしたが、2周目はフロート化自体が通常速度ドラッグで1回不成立となり、低速
ドラッグでフロート化には成功したところで、UIA操作の限界判断基準（同一手法を3パターン試しても
症状不変）に達したと判断し、いったん試行を打ち切った。

**現時点の判定**：観点(2)は**部分的NG**。フロート化の重大回帰（Collapse化時の完全不成立）は解消
されたが、再ドッキングは(a)極小ヒットテスト領域が原因とみられる不成立が複数回発生し、(b)成立時も
意図しないレイアウト（縦長別タブ化）になる、という新たな問題が見られた。これがUIA操作特有の座標
精度限界か、実装側（MinWidth実測値の疑義、ドッキングガイド判定ロジック）の問題かは、忍者の実機
確認単独では切り分けが難しい。殿の実機目視・操作での確認、または侍による診断ログ計装での裏取りを
推奨する。

### 観点(3) Opacity=0のレイアウト占有スペース実測
UIA探索で、ドッキング時の上段領域（旧タイトルバー相当）が**ウィンドウ相対Y=104〜129（高さ約25px）
のスペースを占有し続けている**ことを実測した（`ControlType.Tab` 全体Bounds=絶対座標1974,144,
569,81・内部`ToolBar`Bounds=1978,169,561,43、両者の差分から上段高さ=25pxと算出）。見た目は透明
だが、省スペース化（T-099要件1の意図）とはトレードオフになる状態を維持している。数値は申し送り
通り、UI/UX的に許容範囲か殿・家老の判断材料とされたし。

Collapse化時代の「フロート化自体が不成立」という重大回帰は解消。ただし新たに「再ドッキングの
安定性・正確性」という別の問題が浮上しており、単純に「回帰なし・OK」とは言い切れない。
