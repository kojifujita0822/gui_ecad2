## 3. 試して失敗したアプローチと結果（今回の教訓）

- **目視でのテーマ・配色検証は誤読を招く（忍者・隠密2）**：忍者が2回（増分1層B・増分2メニュー
  背景色）にわたり目視で「変化なし=NG」と誤判定したが、いずれも既存スクリーンショット自体を
  画素実測すると正しく切り替わっていた。要因は(1)アクティブキャプション等「両テーマ共通で
  変わらないのが仕様」の配色を誤NG判定しやすい (2)対象領域が細い帯で見落としやすい、の2点。
  memory `ecad2_avalondock_vs2013_active_caption_by_design.md`・スキル追記済み。
- **UIA `ControlType.MenuBar`のFindFirstがOSタイトルバーのシステムメニューを誤って掴む罠
  （隠密2）**：侍の「メニュー効果なし」観測（診断ログで値は正しいが画面は白のまま）の真因。
  座標y≈18〜38がOSタイトルバー領域（y≦31、テーマ非依存で常時白系）と重なっていた。同一マシン
  5セッション体制ゆえ環境固有差異説は構造上成立せず、座標取り違え説が確定。スキル追記済み。
- **AvalonDock/WPFの「プロパティ値は正しいが実描画に反映されない」構造的制約に2回遭遇
  （侍）**：増分1層B（AvalonDock DockingManager.Theme設定）・増分7（Menu SubMenuBorderの
  TemplateBinding）いずれも、値操作だけでは解決せず、Theme変更後の明示的なレイアウト再構築や
  ControlTemplate自体の書き換えが必要という共通の壁に当たった。`DependencyPropertyHelper
  .GetValueSource`で値解決経路を特定するのが調査の近道（samurai.md追記済み）。
- **緊急実害時、侍が家老を経由せず殿と直接やり取りする場面が発生**：メニューが「白地に白文字で
  読めない」という実害への即応として、侍が家老の事前確認なしに殿の指摘へ対応（Light固定化）。
  今回は実害の緊急性ゆえ家老は事後承認したが、常態化すると台帳記録の一貫性確保が難しくなる
  懸念あり。次回以降も「結果だけは家老へ一報」を徹底する。
- **着手前調査の網羅性不足が増分の後追い発覚を招いた**：着手前調査（隠密）の色指定一覧が
  「明示的な色指定のgrep」に依っており、既定値のまま何も指定していないコントロール
  （ListBox/DataGrid中身等＝増分5、配置ツールバー2段目＝増分6）を拾えなかった。次回同種の
  計画立案では「明示指定の網羅」に加え「未指定＝既定値のまま残る箇所」も洗い出す視点を持つこと。
- **peerセッション消失時の役決め処理が脆い**：旧隠密（1x7t34ve）が突然list_peersから消え新隠密
  （nhxkvrpk）が着任した際、家老が「隠密2へ移れ」と誤指示し2往復の行き違いが発生。list_peers
  頼みのためセッション消失の即時検知ができない構造的な弱さ。
- **peerメッセージは「返信されたし」と明記しないと返ってこないことがある**：忍者への気づき
  収集依頼で、殿にご指摘いただくまで気づかなかった。軽い問い合わせでも明記が要る。
