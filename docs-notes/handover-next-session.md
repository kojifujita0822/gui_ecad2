侍・忍者・隠密から複数の実装案（画面配置・パネル構成・操作方式・見た目・キー割当・情報の
見せ方等）が提示されたとき、たとえ既存の裁定方針（design-brief等）の自然な延長に見えても、
**家老の裁量だけで選ばず、必ず殿に選択肢を提示して確認する。** 詳細・経緯は
`docs-notes/roles/karo.md`の同名節を参照。UI/UXと無関係な純粋な実装内部の技術選択
（変数名・内部アルゴリズム・段階分割の順序等）は引き続き家老裁量でよい。

**2026-07-04追記（確認のタイミングの使い分け）**：「必ず確認する」は常に実装前の抽象的選択とは限らない。
実装途中に発覚する細かい挙動の分岐（影響が小さく実装しないと判断しづらいもの）は、殿の指示により
「①家老が妥当な既定案を選び侍に実装→②忍者が実機検証→③検証済みの実際の挙動を殿に提示して確認」
という**実装→検証→確認のサイクル**で進めてよい。一方、構造的・後戻りしにくい分岐（データ形式・画面区画・
設計アプローチの転換等）は引き続き実装前に選択肢提示する。詳細は memory（feedback_route_design_decisions_to_user）参照。

## 本セッションで追加・更新した運用プロセス

いずれも `docs-notes/roles/*.md` に反映済み（`feature/t021-enter-placement`ブランチ上でコミット、mainマージ待ち）。

1. **バグ対応のWチェック（並行方式）**（karo.md）：忍者がバグ検出→侍(修正)+隠密(独立原因調査)を**並行**で走らせ、家老が突合。一致すれば確定、食い違えば再検討。
2. **検証パイプラインの既定順**（karo.md）：侍実装完了後は既定で**侍実装→隠密コードレビュー(静的)→忍者実機確認(動的)**の順。実機確認が最もコストが高いため、安いレビューで先にふるい落とし手戻りを減らす狙い。上記Wチェックとは別軸で併用する。
3. **隠密のコードレビューに`code-review`スキル併用を必須化**（onmitsu.md）：手動の観点確認に加え、組み込みの`code-review`スキルを必ず使う。effort levelは規模に応じて選択。`--fix`は使わない（実装は侍へ委譲）。
4. **忍者の検証手順にUI Automation偽結果対策を明記**（ninja.md）：Select/Invoke経由の操作は実装を迂回して偽の結果を生むことがあるため、疑わしい結果は物理クリック/実キーで必ず再検証。
5. **忍者・隠密のワークフローをrole.mdへ統合**（ninja.md/onmitsu.md）：別SOPファイル化はせず、既存role.mdに手順・制約・統一報告フォーマットを直接追記する方式を採用（侍のagent-sops方式とは異なる）。
