## 3. 試して失敗したアプローチと結果（今回の教訓）

1. **十字型オーバーレイ残留対策の2周不発→俯瞰評価で決着**（`memory:
   t110_increment1_moguratataki_overview_effect`）。1周目=侍が対象を取り違え（ecad2自前枠を
   Collapse化、AvalonDock標準枠には未対処）、2周目=隠密推奨APIが`internal interface`で
   コンパイル不可。家老がモグラ叩き検知を明示発動したところ、隠密が「表示させてから消す」枠を
   出て「そもそも表示させない」（`CanDock="False"`）という発想転換で一撃決着。
2. **API呼出提案時は「存在確認」だけでなく「アクセシビリティ確認」まで必要**（`memory:
   feedback_verify_before_acting_on_diagnosis`追記）。隠密が一次ソースで存在を確認したメソッドが
   `internal interface`経由で外部アセンブリから呼出不能だった。侍の実装前ビルドで発覚。
3. **タイムスタンプ観測は日付まで確認する**（`memory: onmitsu_environment_forensics_technique`）。
   忍者が「保存操作と同一秒で更新」と報告したが、実際は日付を跨いだ前日の痕跡だった。隠密の
   ファイルシステム実測（LastWriteTime日付込み・プロセスStartTime・バイナリビルド時刻の3点突合）
   で訂正。
4. **旧版永続化ファイルによるXAML修正の無音上書き**（新規パターン候補PR-25、
   `docs-notes/pattern-recurrence-log.md`）。案D実装後も、修正前に保存された`main-layout.xml`
   （新属性を含まない）のDeserializeがXAML側の修正を無音で消す罠が存在した。隠密が実ファイル
   検分で実害発生前に発見・防御コード追加（指2）。**永続化対象モデルのXAML初期値を変更した際は
   旧版ファイルとのround-tripを生き延びるか必ず問う**、という教訓。
5. **间欠的な動的事象は1回の再現不成功だけで「解消」と判断しない**（`memory:
   feedback_static_vs_dynamic_investigation`追記）。オーバーレイ残留は1回目の再検証で再現
   しなかったが、後に構造的原因が確定した。
