## 3. 試して失敗したアプローチと結果（今回の教訓）

1. **ダブルクリックによる再ドッキング仮説は不成立**——フロート窓は標準Window由来、ダブルクリックは
   最大化トグルとして機能するのみ。AvalonDockのContentDocking経路とは無関係だった。
2. **忍者の実マウスドラッグ合成はAvalonDockのOverlayWindow/DropTarget機構を一度も起動できず**——
   3回試行してもEnumWindows・CopyFromScreenとも十字型UI検出ゼロ。P-098（AvalonDock本家も実ドラッグ
   自動化を諦め内部API直接呼び出しで代替している旨）と符合する、UIA自動化の構造的限界の実例。
3. **peer ID変遷の謎、未解明のまま持ち越し**——侍が`cntyx58o`から`wzqc4njx`へ移行したように見え、
   家老が本人照会したところ殿より「wzqc4njxは本物の侍」との確認を得たが、なぜIDが変わったか（同一
   セッションの別接続か、何らかの再接続事象か）の技術的詳細は解明できていない。さらに`dk35lltu`と
   いう未名乗りIDも同時刻(23:00:06)に出現、正体不明のまま。**今回のセッション終了の引き金である
   「侍への指示が届かない」現象と関連する可能性を否定できない**——次回セッションで再発時は
   `list_peers`のID・PID・last seenを慎重に確認し、早期に異常を疑うこと。
4. **侍セッションの応答途絶、原因未特定**——T-104優先変更の指示（`wzqc4njx`宛）を最後に、`cntyx58o`・
   `wzqc4njx`どちらのIDからも一切の応答が得られなくなった。`list_peers`ではプロセス自体は生存表示
   （last seen更新あり）だったが、メッセージへの反応が完全に止まった。ハングアップ・出力破損等の
   可能性も排除できないが、確たる証拠は無いまま。次回起動時、`docs-notes/output-corruption-log.md`
