# 出力破損の発生記録簿

Claude Codeセッションの出力破損（raw `<invoke>` 表示・court等の異常トークン混入・
ツール呼び出し構文の崩れ）の発生を記録する台帳。**対策の効果を頻度の前後比較で検証する**
ことが目的のため、発生のたびに必ず1行追記する（検知した役が自著してよい）。

検知時の行動は従来どおり `long-horizon-discipline` スキル §5 の離脱プロトコルに従い、
**離脱の前に本記録簿へ1行残す**（記録が取れぬほど破損が激しい場合は復旧後に追記）。

## 記録様式

| # | 発生日時 | 検知した役 | モデル | 症状 | 文脈条件 | 離脱発動 | 備考 |
|---|---------|-----------|--------|------|---------|---------|------|

- **症状**: raw invoke表示／court等混入／その他（具体的に）
- **文脈条件**: セッション経過（起動直後・長期化・コンテキスト要約後か）・直前の作業内容・
  長い連続ターン中か等、後の頻度分析に効く情報を短く
- **離脱発動**: 有／無（無の場合は理由）
- **備考**: **同一ターミナル系列（/clear・再起動を挟んだ同じウィンドウ）での再発か否かを必ず書く**
  （下記「運用知見」の再発傾向を追跡するため）
- **モデル欄の記録流儀**: システムプロンプト環境情報のモデル表記は**ハーネス申告値の転記にすぎず、
  実態とずれうる**。実例=家老セッション（2026-07-10）は環境情報が「Sonnet 5 / claude-sonnet-5」
  だったが実態はFable 5（殿確認。同じプロンプト内のコミット署名指示は「Claude Fable 5」で矛盾していた）。
  記録時は「申告値」として書き、署名指示等との食い違いがあれば注記、確定は殿の確認に拠る

## 運用知見（殿の観察 2026-07-10）

**症状が一度出始めると、/clear後にも再発を繰り返す場合が多い。**「自覚できれば再発自体が
減少するかもしれない」（殿）——セッション自身が「この系列は破損が出やすい状態にある」と
自覚することが、再発低減に効く可能性がある。

これを踏まえた運用：
1. **破損発生後の再起動・/clear時**、新セッションは本記録簿を読み、直近の発生歴（特に自分と
   同一ターミナル系列・同一役の記録）を**明示的に自覚**してから作業に入る
2. 家老は発生歴のある役へ采配を再送する際、**直近の発生記録への警戒を一言添える**
3. 再発の有無は本記録簿の備考欄で系列追跡し、「自覚あり／なし」での再発率差が見えてきたら
   対策年表へ反映する
