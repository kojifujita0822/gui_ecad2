# 不具合パターン再発 台帳

「パターン再発検知」制度（`docs-notes/roles/onmitsu.md`・`docs-notes/roles/karo.md`参照）を、
記憶やその場のgrepという偶然に頼らず確実に運用するための台帳。2026-07-13、殿のご指摘で新設
（T-069往復3周目レビューで`CommitDeviceNameEdit()`呼び忘れ3回目を検出できたのは、たまたま
コード内コメントに過去の経緯が残っていたからに過ぎず、再現性が無いとの所感による）。

## 使い方

- **レビュー・調査で不具合を見つけたら、まず本台帳の「パターンの型」列と照合する**（記憶・grepは
  補助手段に留める）。一致する型があれば「パターン再発」として報告し、本台帳の該当行へ再発履歴
  （タスクID・日付）を追記する。
- 一致する型が無いが、過去の別タスクと根本原因が同じと判断できる場合は、**新規パターンとして
  本台帳へ追加**する（初出時点では「再発」ではなく「新規パターン候補」として記録し、2件目が
  出た時点で正式に「パターン」と確定してよい）。
- 追記は**検知した役が自著してよい**（隠密が主だが、忍者・侍が気づいた場合も可）。追記後は
  家老へ一報し、家老が制度化の要否を検討する（`karo.md`「パターン再発の制度化検討」参照）。
- 「制度化状況」列は、対処が完了したら更新する（pending→制度化済み等）。制度化済みでも再発する
  ようなら、その制度化が不十分だった証拠として扱い、再検討する。

## 運用ノート

- 台帳が肥大化してきたら、決着済み（制度化済みで再発なし）のパターンは要約のみ残し詳細を
  `docs/archive/`等へ移すことを検討する（現時点では2件のみのため未実施）。
- 「初出」の日付が特定できない場合（旧タスクでコード内コメント等からしか経緯を追えない場合）は
  無理に日付を推測せず、タスクIDと参照先のみ記す（不明は不明と明示する原則）。
