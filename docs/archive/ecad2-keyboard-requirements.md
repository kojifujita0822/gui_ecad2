# ecad2向け：GuiEcadキーボード操作の痛点と、キーボードファースト要件整理 — 2026-07-03（忍者）

> 家老依頼（ecad2 = C:\ECAD2、新規eCAD計画向け）。ecad2最重要要件「キーボードファースト操作」の
> 実現可否を左右する技術選定（隠密Part B）の評価基準として使う想定。**実装は行わない、調査・整理のみ**。
>
> 材料: `docs-notes/keyboard-input-fix-plan.md`（ラウンド1〜9）、`docs/known-issues.md`、
> `docs-notes/handover-next-session.md`、マウスレス配置（キーボード配置モード）非表示化の経緯、
> および忍者自身の実機検証（UI Automation・PowerShell操作）経験。

## (1) GuiEcadで果たせなかった/不安定だったキーボード操作 — 一覧と原因

GuiEcadのキーボード配置モード（マウスレス配置）は、9ラウンドにわたる修正の末、最終的に
**WinUI3自体の未解決の既知バグ**（[microsoft-ui-xaml Issue #6179](https://github.com/microsoft/microsoft-ui-xaml/issues/6179)、
Microsoft側 "not planned" ＝修正しない方針）に行き着き、根本解決できないまま**機能自体を非表示化**して
決着した（コード自体は削除せず残置、`docs/known-issues.md` に既知の制限事項として記録）。
以下、症状と原因を時系列・カテゴリ別に整理する。

### A. マウスレス配置（キーボード配置モード）関連

| # | 症状 | 原因 |
|---|---|---|
| A1 | Enterキーがボタンの既定動作（トグルOFF）に奪われる | `Canvas.Focus()` の同期呼び出しが、ボタンクリック後のWinUI既定フォーカス割り当てと競合し敗れる（フォーカス委譲のレース） |
| A2 | 矢印キー・数字キーが反応しない | Win2D `CanvasControl` がフォーカスを持つと `Page.OnKeyDown` が届かない既知の制約が、Canvasが実際にフォーカスを獲得したことで初めて発動 |
| A3 | （KeyboardAccelerator方式へ移行後）Escapeキーが配置モードを解除しない | 矢印キー・数字キーはグローバルな`KeyboardAccelerator`へ移行済みだったが、Escapeだけは旧来の`Page.OnKeyDown`に残っていた。通常のマウス操作でCanvasが日常的にフォーカスを持つ状態自体が既に不安定（A2と同根のより広い問題） |
| A4 | Spaceキー押しっぱなしパンが実装しづらい／離した瞬間の検知ができない | `KeyboardAccelerator`は「押下時の単発発火（Invoked）のみ」で、KeyUp相当のイベントを持たない。連続状態管理（押している間だけパン）と構造的に相性が悪い。最終的にSpaceパン自体を廃止し、PageUp/PageDownの離散スクロールに仕様変更して回避 |
| A5 | Enterで配置後、機器名を直接入力できない | 配置成功後にフォーカスを編集欄（機器名ボックス）へ渡す一手が実装から単純に抜けていた（設計自体は既存の枠組みと矛盾なく実現可能だった） |
| A6 | 機器名入力確定後、方向キーで移動しようとするとフォーカスが下部パネル（DRC/検索結果のTabView）に奪われる | インライン編集終了時（`Visibility=Collapsed`）にフォーカスをどこにも明示的に戻していなかった。WinUIの既定フォーカス委譲でTabViewItem等、**組み込みで矢印キーを消費するコントロール**にフォーカスが渡ってしまうと、以降の矢印キーがそちらに奪われる |
| A7 | 【最終的な真因】枠ラベル編集を開始した直後に別操作を挟むと、文字入力前に空ラベルで確定される | WinUI3自体の既知バグ（Issue #6179）。要素がPointerPressedを処理していても、対応するPointerReleased発生後にフォーカスがタブオーダー内の別要素へ**フレームワーク側で暗黙的に**移る。アプリコードのFocus()呼び出しを一切経由しないため、コード読解だけでは原因箇所が見つからない（実際に8ラウンドかかった） |

**A7への対策として試して失敗したもの**（ラウンド9）: `RootGrid.LosingFocus` イベントで
移動そのものを検知・`TryCancel()`で阻止する高度な割り込み方式を実機検証。イベント検知自体は
成功したが、**`TryCancel()`が「成功」を返しても直後にフレームワークが別経路で移動を強行し直す**
という、WinUI3内部の再試行に起因する新たな副作用（2回目以降のEnter配置が効かなくなる）が発覚し、
最終的に断念・再保留。**「移動を止める」方向のアプローチはこのフレームワークでは制御しきれなかった**、
という点が重要な教訓。

### B. ショートカットキー設定（キー再割当てダイアログ）関連

| # | 症状 | 原因 |
|---|---|---|
| B1 | キー捕捉ダイアログが開いても見えない（背面固定） | WinUI3は「同時に複数の`ContentDialog`を開けない」制約を持つ。設定ダイアログを開いたまま、その中の「変更」ボタンから別のダイアログでキー捕捉しようとする**ダイアログの入れ子構成**が、直列化ゲート（`SemaphoreSlim`）の設計と本質的に非対応だった。ダイアログというUI部品を2重に使う設計自体がフレームワークの制約と衝突した例 |
| B2 | 「変更」ボタン押下→キー押下しても無反応 | 対象ボタンを`IsEnabled=false`にした状態で`Focus()`を試みていた。**無効化されたコントロールはキーボードフォーカスを取得できない**という基本的なWinUI/UWP標準挙動に反する実装ミス |

**B1の解決**: 別ダイアログを開かず、既に開いている設定ダイアログ内で完結する「インライン捕捉」方式
（ボタン自身にFocusしKeyDownを一時アタッチ）へ変更して解消。**ダイアログを跨がない設計に倒すことで
フレームワーク制約を回避できた**、数少ない解決成功例。

### C. 横断的・構造的な問題（GuiEcad固有ではなく、WinUI3/類似フレームワーク一般に敷衍しうる教訓）

1. **フォーカス非依存でグローバルにキーを拾う公式APIが存在しない**（WinUI3。UWPの`CoreWindow.KeyDown`相当がなく、[Issue #3986](https://github.com/microsoft/microsoft-ui-xaml/issues/3986) "Gather keyboard input" はnot plannedでクローズ済み）。`KeyboardAccelerator`（グローバル・フォーカス非依存で解決される仕組み）が実用上の代替として機能したが、単発発火のみという制約（A4）を抱える。
2. **カスタム描画コントロール（Win2D CanvasControl）は既定でフォーカスを取得しない**（[Win2D Issue #686](https://github.com/Microsoft/Win2D/issues/686)）。`IsTabStop`明示設定等が必要で、対応を忘れると「クリックしたのにキーが効かない」という分かりにくい不具合を生む。
3. **フォーカスを持つコントロールの組み込みキー処理が、アプリ側のグローバルショートカットより優先される**（A6）。TabView等、矢印キーやTabキーを内部で消費する標準コントロールにフォーカスが渡ると、アプリのキー処理が届かなくなる。
4. **要素がVisual Treeから消える操作（`Visibility=Collapsed`、`Children.Clear()`等）は、既定のフォーカス委譲を誘発し、開発者の意図しない要素へフォーカスが移る**（A6、および調査中盤で疑われた`RefreshPropertiesPanel`のケース）。
5. **PointerReleased後の暗黙フォーカス移動という、アプリコードを経由しないフレームワーク内部の既知バグが存在し、Microsoft側は「仕様として直さない」方針を取りうる**（A7＝Issue #6179）。この種のバグは一次資料（GitHub Issue）に行き着くまで、コード読解ベースの調査だけでは8ラウンドかかった＝**診断コストが極めて高い**。
6. **フレームワークの「フォーカス移動を止める」割り込み系API（`LosingFocus`/`TryCancel`）は、検知はできても制御しきれない場合がある**（A7ラウンド9）。フレームワークが移動を強行し直す再試行が起き、対症療法が新たな副作用を生んだ。

### D. 忍者の実機検証（UI Automation経由の外部操作）から得た補足知見

今回のセッションで `guiecad-ui-automation` スキル整備のため実測した事実で、ecad2の
**テスト容易性・自動化容易性**の観点からも参考になる：

- **`SendInput`/`SendKeys`（Win32の低レベルキーボードイベント合成API）による外部プロセスからの
  キー送信が、このWinUI3アプリに一切届かない**（矢印キー・Escキーいずれも無反応、実測確認済み）。
  一方、マウスクリック・マウスホイールイベントは正常に機能する。実際のユーザーの物理キーボード入力
  とは別の話（ハードウェアレベルのメッセージ経路のため無関係と推定）だが、**外部からの自動UIテスト
  やアクセシビリティツール、リモート操作ツールがキーボード操作を模擬しづらい**という制約になる。
- Win2D `CanvasControl` の内部（記号・配線等）はUI Automationツリー上で**単一の画像要素**としてしか
  公開されず、内部要素は一切走査できない。キーボード操作の検証だけでなく、UI要素としての
  アクセシビリティ全般に影響する。

## (2) ecad2が「キーボードのみで作図完結」するために必須の操作要件（技術非依存）

GuiEcadの痛点を踏まえ、技術スタックに依存しない要件として整理する。**隠密Part Bの技術選定で
「キーボード/フォーカス堅牢性」を評価する際のチェックリストとして使える形**にした。

### R1. フォーカスモデルの一元性・予測可能性【最重要】
- 「現在どの要素がキー入力を受け取るか」が、アプリ側のコードで完全に把握・制御できること。
- フレームワークが**アプリコードを経由せず暗黙にフォーカスを移動させる**（GuiEcadのIssue #6179、
  Visual Tree消失時の既定委譲等）挙動を持たないか、持つ場合は**アプリ側で確実に上書き・無効化できる**こと。
- 要素がVisual Treeから消える／非表示になる操作（編集終了・パネル切替等）の際、フォーカスの移動先を
  必ずアプリ側が明示的に決定できること（フレームワークの既定委譲に委ねない）。

### R2. グローバルコマンドとローカルテキスト編集の分離
- 「ツール切替・要素移動等のグローバルショートカット」と「機器名・コメント等のテキスト編集」が、
  互いにキーストリームを奪い合わない設計であること。
- テキスト編集中は、数字・文字キーが確実にテキストへ入力され、同時に定義されたショートカット
  （例: 数字キー＝ツール切替）に横取りされないこと（GuiEcadでは一次資料ベースで「奪われない」と
  推定はしたが、100%の断定はできず実機確認が必須だった＝フレームワークの保証が弱い一例）。

### R3. 押下・解放の両方を検知できるキー入力機構（連続操作対応）
- 「押している間だけ有効」な操作（パン、連続移動等）を実装する場合、**KeyDown/KeyUp相当の両方を
  確実に検知できる**こと。単発発火（押した瞬間のみ）しか提供しないアクセラレータ機構しかない場合、
  ポーリング等の代替実装を強いられ、設計の一貫性が崩れる（GuiEcadのSpaceパンで発生、最終的に
  機能自体を離散スクロールへ変更して回避した）。

### R4. カスタム描画領域（キャンバス）への確実なキーボードフォーカス
- 作図領域（自前描画のキャンバス）が、クリック等の操作で**確実に・即座に**キーボードフォーカスを
  取得できる技術基盤であること。取得の可否が不確実だと、それに依存する全てのキー操作が土台から崩れる
  （GuiEcadの一連の不具合の多くがこの一点に起因する）。
- 理想的には、キャンバス内の論理要素（セル・記号等）がアクセシビリティツリー上でも意味のある単位で
  表現され、外部テストツールからも検証可能であること（Win2D CanvasControlは単一画像要素にしかならず
  この点で不利だった）。

### R5. パネル間フォーカスナビゲーションの制御可能性
- 複数パネル（作図領域・機器表・プロパティ・ナビツリー等）を持つUIで、Tab/Shift+Tab等による
  パネル間移動の順序・範囲をアプリ側で明示的に設計できること。
- 移動先のコントロールが、矢印キー等の**組み込みキー処理でアプリのグローバルショートカットを
  横取りしない**こと、または横取りする場合はその挙動を無効化・回避する手段が用意されていること。

### R6. 確定/取消（Enter/Esc）の一貫した意味論と到達性
- モード・コンテキスト（配置中／編集中／範囲選択中／ダイアログ表示中等）に応じてEnter/Escが何を
  意味するかが一貫しており、かつ**どのコンテキストでも確実にアプリへ届く**こと（GuiEcadでは
  Escapeがフォーカス状態依存で届かないケースがあった＝R1と表裏一体）。

### R7. モーダルUI（ダイアログ等）の入れ子・多重表示制約の把握
- 「ダイアログを開いたまま、その中から別の入力待ちUIを開く」構成が必要になる場面
  （例: 設定画面の中でキー再割当てを行う）を想定し、フレームワークのモーダルUIが多重・入れ子表示に
  対応しているか、対応していない場合はインライン化等の代替UIパターンで回避できることを事前に確認する。

### R8. コントロールの無効化とフォーカス可能性の分離
- 「一時的に操作を禁止したい」（多重クリック防止等）場合に、`IsEnabled=false`のような単純な無効化が
  キーボードフォーカスの取得可否にも影響する仕様かどうかを把握し、意図せずフォーカスを失わせる実装
  ミスを構造的に避けられる設計（フラグ管理での多重防止等）を選べること。

### R9. テスト容易性・自動化親和性
- 外部プロセス（テストツール・自動化スキル）からのキーボード操作シミュレーションが機能する
  （GuiEcadはSendInput等が届かず、UI Automation経由のボタン操作止まりだった）。
- アクセシビリティツリー（UI Automation等）が、キャンバス内の論理要素も含めて意味のある形で
  公開されている（キーボード操作の自動検証だけでなくスクリーンリーダー対応にも波及する）。

### R10. フレームワーク側の既知バグ・issueデータベースの成熟度
- 候補技術ごとに、フォーカス管理関連の未解決Issue（特にMicrosoft/開発元が "not planned" とした
  既知バグ）の有無・件数・深刻度を選定時に事前調査すること。GuiEcadのケースでは、この種の一次調査を
  もっと早い段階（実装後の不具合対応としてではなく選定段階）で行えていれば、9ラウンドの手戻りの
  多くを避けられた可能性がある。

## まとめ

GuiEcadの9ラウンドにわたる調査の核心は、**「フォーカスの所在をアプリ側が完全にコントロールできるか」
という一点（R1）に collapse する**。R2〜R9はすべてR1が満たされていれば実装しやすくなる派生要件、
またはR1が崩れた場合の被害範囲を狭める補助要件という位置づけで整理できる。技術選定（隠密Part B）
では、候補技術のフォーカス管理APIが「宣言的かつ確実」か、「暗黙・非決定的な委譲を持つか」を
最優先の判定軸にすることを提案する。
