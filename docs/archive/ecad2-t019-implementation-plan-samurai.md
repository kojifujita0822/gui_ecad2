# T-019 実装プラン（侍起草）

対象: ドキュメント管理（GcadSerializer Load/Save、新規/開く/保存のメニュー・ツールバー接続）
前提: main直での作業。GcadSerializerは実装済み（T-007でGuiEcadよりバイト単位同一移植、`docs/archive/ecad2-guiecad-code-survey-onmitsu.md` T-024節参照）。

## 現状（着手前の確認事項）

- `MainWindowViewModel.Document`は`{ get; }`のみで起動時ダミー生成の固定インスタンス。新規/開くで
  差し替えるには可変にする必要がある（純粋な技術的変更、UI/UX分岐ではない）。
- メニュー(`MenuBarArea`)・ツールバー双方に「新規」「開く」「上書き保存」の項目・ボタンは既に存在するが、
  Command/Click未接続の表示のみ（`MainWindow.xaml` 55-102行）。
- Dirty判定（未保存変更の検知）は現状ecad2に存在しない。
- `HasProject`(T-020)は`Document.Sheets.Count > 0`を見ており、「新規」でSheets=0にできれば
  自然に濃紺空状態へ連動する設計済み。

## 段階分割（標準Windows/WPF規約で完結する部分から）

1. **基盤整備**: `Document`を可変化（`LoadedFilePath`等の付随状態も追加）。UI未接続、挙動変化なし。
2. **保存/名前を付けて保存**: `Microsoft.Win32.SaveFileDialog`（拡張子`.gcad`）。パス保持済みなら
   上書き、未保持なら名前を付けて保存へフォールバック（標準的な挙動）。`GcadSerializer.Save`呼び出し。
   例外（I/Oエラー等）はMessageBoxで通知し、技術的例外文面をそのまま出さない（隠密推奨に沿う）。
3. **開く**: `Microsoft.Win32.OpenFileDialog`。`GcadSerializer.Load`で復元しDocumentへ差し替え、
   `CurrentSheetIndex`を0にリセット。スキーマ不一致等の例外もMessageBoxで通知に変換。
4. **新規**: 空ドキュメント（Sheets=0）へ差し替え。`CurrentSheet`等Sheets非空を前提にした既存の
   参照箇所（LadderCanvasHost.Drawへのバインド等）がSheets=0時に例外を起こさないよう防御を追加。
5. メニュー・ツールバー双方から2〜4のコマンドを呼べるようCommand/Clickを接続。

上記1〜5は標準WPF規約（Win32ダイアログ、MessageBoxでのエラー通知）の範囲で完結するため、
殿確認を待たずに進められる想定。

## UI/UX分岐点（保留・選択肢提示が必要）

1. **未保存確認フロー(Dirty判定)を今回のスコープに含めるか**: GuiEcad(T-024調査)はここが構造的
   欠陥だった領域。含める場合、判定基準（Undo履歴ベース／プロパティ変更通知ベース）とダイアログの
   ボタン構成（保存/破棄/キャンセルの3択が一般的）を要選択。含めない場合、今回は上書き確認なしで
   新規/開くが即座に現ドキュメントを破棄する（データ喪失リスクを残す暫定実装になる）。
2. **「新規」時の初期状態**: todo.md T-019備考の申し送りはSheets=0（濃紺空状態）を前提にしている
   ように読めるが、GX Works3等の実際の初期状態（即座に1シート生成 vs 空）との整合は未確認。
3. **保存先パスの既定値・記憶**: 前回開いたフォルダを記憶するか、毎回既定フォルダから始めるか。
4. **「最近使ったファイル」等メニュー拡張の要否**: 現行メニューには項目なし。今回は範囲外と判断。

殿は就寝中のため、1〜5の標準規約部分を先行実装し、上記分岐点は選択肢のまま保留する。
