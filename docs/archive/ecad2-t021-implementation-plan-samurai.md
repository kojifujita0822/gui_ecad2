## 1. 増分分割（案Xを検証可能な小単位へ）

家老想定の (i)〜(v) を踏襲しつつ、各増分の**変更範囲**と**検証観点**を付す。原則1増分1コミット・各増分で忍者検証。

### 増分(i)：選択セルEnterで1要素を配置
- **内容**：`Window_PreviewKeyDown`にEnterのcaseを追加。ガード＝`IsCanvasFocused() && Tool.Mode==PlaceElement && Tool.PartId!=null && SelectedCell!=null`。成立時、Tool.PartIdからエントリ解決し`TryPlaceElement(entry, Tool.IsOr)`を呼ぶ（クリック経路 `LadderCanvasHost_PreviewMouseLeftButtonUp` の判定を流用）。
- **変更範囲**：`MainWindow.xaml.cs`（Enter case追加、既存ヘルパ流用）。**新規ロジックほぼ無し**。
- **検証観点**：ツール選択中（例F5後にツールが保持される導線が要るか要確認、下記UI/UX分岐参照）にセル選択→Enterで配置ダイアログが開くか。ガード条件（キャンバス非フォーカス時／ツール未選択時／セル未選択時）でEnterが誤発火しないか。
- **リスク**：低〜中。ただし「アクティブなツール」の定義が現行フローと噛み合うか要検証（分岐A参照）。

### 増分(ii)：配置直後に浮動インライン入力欄へフォーカス委譲
- **内容**：現行`TryPlaceElement`は既に`ElementPlacementDialog`を`ShowDialog()`で開き、ダイアログ側`Loaded`で`DeviceNameBox.Focus()`済み（隠密調査 観点1）。案C（モーダル据え置き）ではこの委譲は**既に成立**。増分(ii)の実質作業は「Enter起点でもこの委譲が問題なく働くことの確認」に縮小する。
- **変更範囲**：原則コード変更なし（確認増分）。もし差異があれば`ElementPlacementDialog.xaml.cs`のフォーカス取得を微修正。
- **検証観点**：Enter配置直後、キーボードフォーカスがダイアログのデバイス名入力欄にあるか（T-023 Peer＋実キー入力で二重確認）。
- **リスク**：低（既存動作の追認）。

### 増分(iii)：命名確定後のフォーカス復帰（キャンバスへ）
- **内容**：`TryPlaceElement`のダイアログクローズ後（240行付近）に`Keyboard.Focus(LadderCanvasHost)`を明示追加し、暗黙委譲に委ねない（隠密の推奨・不明点1への対策）。OK確定時・キャンセル時の両方で復帰先を規定する。
- **変更範囲**：`MainWindow.xaml.cs`（`TryPlaceElement`末尾、`SelectDefaultButton_Click`等のEsc/選択解除経路とも整合を取る）。
- **検証観点**：命名Enter確定後、フォーカスがキャンバスに戻り、続けて矢印移動→Enter配置が「一気通貫」で回るか。連続配置（配置→命名→配置→命名…）でフォーカスが迷子にならないか。
- **リスク**：**中〜高**。GuiEcad A6（Visual Tree変化時の意図しないフォーカス委譲）と同種の地雷になり得る箇所。**最初にPoC検証すべき筆頭候補**（下記2参照）。

### 増分(iv)：Esc多段階4層
- **内容**：現行Escの「一括クリア」を、確定仕様の4層（内→外へ1層ずつ）に分解：
  1. テキスト編集中（ElementPlacementDialog表示中）→ Escで編集キャンセル（入力破棄）※ダイアログ内のEsc/キャンセル挙動として扱う
  2. 配置モード中（Tool.Mode==PlaceElement）→ Escで選択モードへ戻す（Tool=SelectDefault、SelectedCellは保持）
  3. 要素選択中（SelectedCell!=null）→ Escで選択解除（SelectedCell=null）
  4. 何もなし → 何もしない（キャンバスフォーカス維持）
- **変更範囲**：`MainWindow.xaml.cs`のEscape case（現在104-111の一括処理を段階判定に書き換え）。層1はダイアログ側の責務のため`ElementPlacementDialog`のEsc挙動確認も含む。
- **検証観点**：各層で1回のEscが1層だけ戻すか（現行は1回で全部クリアしてしまう）。層の順序（編集→配置→選択→無）が仕様通りか。StatusMessageの案内が各層で破綻しないか。
- **リスク**：中。現行の「Escで一括クリア」に依存した既存挙動（T-017実機検証で調整済みのStatusMessage残留対策 108-109行）を壊さないよう回帰注意。

### 増分(v)：矢印移動時のカーソル追従スクロール
- **内容**：`MoveSelectedCell`（156-170）でSelectedCell更新後、`LadderCanvasHost.BringIntoView(LadderCanvasHost.CellRectDip(newCell))`を呼び、親ScrollViewer（CanvasArea）を自動追従させる（隠密調査 観点4、既存`CellRectDip`流用）。パンはこの矢印追従＋既存ホイールスクロール（`CanvasArea_PreviewMouseWheel`）で満たす（quasimode不採用）。
- **変更範囲**：`MainWindow.xaml.cs`（`MoveSelectedCell`）。`CellRectDip`は既にinternal公開済み（T-023で選択ハイライトと共用）。
- **検証観点**：矢印で画面端までカーソル移動時、キャンバスが追従スクロールするか。**ズーム倍率≠100%**でも座標がずれないか（隠密の未検証点2）。
- **リスク**：中。ズーム時のBringIntoView座標計算（LayoutTransform考慮）が実機で正しいか未検証。

### 増分の依存関係
- (i)→(iii) は連鎖（配置できないとフォーカス復帰も検証不能）。(ii)は(i)の副次確認。
- (iv)Esc・(v)スクロールは(i)〜(iii)と**独立**して着手・検証可能。並行しても良いが、1増分1コミットは維持。

## 2. 最初にPoC/スパイクで検証すべき増分と最小PoC内容

**筆頭リスク＝増分(iii)「命名確定後のフォーカス復帰」**。理由：
- GuiEcadが9ラウンド格闘して断念した核心が「ダイアログ/Visual Tree変化後のフォーカスの所在制御」であり、増分(iii)がまさにその領域。
- 隠密の不明点1「ElementPlacementDialogが閉じた後どこにフォーカスが戻るか未検証」が未解決。
- (i)(ii)は既存経路の流用で確度が高く、(iv)(v)は独立かつWPF標準機能の範囲。**フォーカス復帰だけが「フレームワークの暗黙挙動に勝てるか」という質的リスク**を抱える。

### 最小PoC（スパイク）の内容
- **目的**：「選択セルEnter→配置→モーダルダイアログで命名→Enter確定→**キャンバスにフォーカスが確実に戻り**、続けてもう一度矢印→Enterで次の配置に入れる」という一気通貫ループが、フォーカスの迷子なしに回るかだけを最小構成で確認する。
- **実装範囲（PoC）**：増分(i)＋(iii)のみを最小結線（(ii)は既存動作、(iv)(v)は含めない）。`Keyboard.Focus(LadderCanvasHost)`のダイアログ後復帰を入れた版と入れない版の2通りを比較し、暗黙委譲任せで戻るのか明示Focusが必須なのかを判定する。
- **フォーカス所在の確認手段**：**T-023のAutomation Peer**でキャンバス/記号のフォーカス・選択状態を走査（`ecad2-ui-automation`スキル）。加えて隠密調査が警告する「UI Automation偽結果の罠」回避のため、忍者が**物理キー入力**でも同一シナリオを再現し、スクリーンショットでハイライト位置とフォーカスの一致を二重確認する。
- **判定基準**：連続3回の配置ループでフォーカスがキャンバスから外れない／ダイアログに毎回正しく入る、を満たせば案Xの土台成立とみなし本実装（各増分）へ進む。満たせない場合は`PreviewLostKeyboardFocus`のHandled化（隠密の一次情報、Focus Overview公式）を追加対策として投入し再スパイク。
- **PoCの扱い**：`poc/`は本実装と分離維持のルールがあるため、スパイクを`poc/`配下に置くか本体ブランチで最小コミット→検証後に増分実装へ発展させるかは**家老に確認したい**（下記の確認事項）。

## 4. UI/UX分岐になりそうな箇所（家老経由で殿へ確認したい候補）

以下は「どう作るか」でなく「どう見える・どう操作させるか」に触れるため、SOP Step2に従い侍・家老の裁量で決めず殿へ選択肢提示すべき候補として洗い出す（実装判断は保留）。

- **分岐A：Enter配置における「アクティブなツール」の意味**
  現行フローは F5等でツール選択→（クリック or キー）で配置後、`TryPlaceElement`末尾で`Tool=SelectDefault`にリセットする（243行）。案Xの「移動→Enter配置→命名→確定→また移動→Enter配置」を一気通貫にするには、**配置後もツールが保持されて連続配置できる**必要があるかもしれない。
  - 選択肢1：配置ごとにツールリセット（現行通り、1回配置ごとにツール再選択が要る）。
  - 選択肢2：配置後もツール保持（連続配置可、Escでツール解除）。GX Works3的な連続入力に近い。
  → キーボードファーストの一気通貫性に直結する操作感の分岐。**殿確認を推奨**。

- **分岐B：Esc層1（テキスト編集中）の「編集キャンセル」の粒度**
  案C（モーダルダイアログ据え置き）では、層1のEscはダイアログの入力破棄＝ダイアログを閉じる挙動になる。このとき「配置した要素も取り消す（配置前に戻す）」のか「要素は残しデバイス名だけ空で確定」なのかで挙動が変わる。
  → データの残り方に関わる操作感の分岐。**殿確認を推奨**。

- **分岐C（軽微・報告のみ）**：追従スクロールの「画面端」判定の閾値（端に何セル寄ったら送るか）。見た目の詰まり具合に関わるが、まずはBringIntoViewの標準挙動（見えるまで最小スクロール）で実装し、必要なら調整、で足りる可能性が高い。**まず標準で実装→忍者検証で違和感が出たら殿へ**、を提案。

- **非分岐（確認）**：モーダル非ネスト規約自体は案Cで現行のまま満たす（同時1枚）。真の非モーダル化はT-033で分離済みのため今回は分岐にしない。

## 6. 家老レビュー結果・殿裁定（2026-07-04）

- **分岐B：命名中Esc**（殿裁定）：**配置ごと取消**（置いた要素も削除する原子的キャンセル）。未命名の孤立要素を残さない。

- **Q3 増分着手順**（家老承認）：筆頭PoC(i+iii) → 本実装(i)(iii) →(iv)Esc・(v)スクロールは独立着手。

**次アクション**：侍が`poc/`で筆頭PoC（増分i+iii＝Enter配置→命名確定後のフォーカス復帰）に着手。忍者がT-023 Automation Peer＋物理キーで二重検証し、連続3回ループでフォーカス迷子なしを確認できたら、本体`src/`へ増分実装（各増分1コミット・各増分で忍者検証）。分岐A(保持)・B(取消)は本実装で織り込む。
