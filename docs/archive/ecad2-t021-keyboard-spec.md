## 実装・検証方針【殿指示 2026-07-04】

**T-021は実装を焦らず、検証に重きを置く。** GuiEcadが同領域（マウスレス配置）でIssue #6179等により9ラウンド
失敗した高リスク箇所のため。**プランを立ててから検証を挟み、それから実装してよい（殿指示 追補2026-07-04）。**

- **プラン先行→リスク検証→本実装**：隠密調査の後、まず実装プランを立てる。最もリスクの高い経路
  （選択セルEnter→要素配置→浮動インライン入力へのフォーカス委譲）は、本実装に入る前に小さなPoC/スパイクで
  成立性を検証してから進める。案X（フル）のゴールへ、案Z的な段階導入で着実に近づける。
- **増分分割で進める**：案Xを小さな検証可能単位に割り、各増分ごとに「侍実装→忍者検証→次へ」を回す。
  想定増分：(i) 選択セルEnterで1要素を配置 →(ii) 配置直後に浮動インライン入力欄へフォーカス委譲 →
  (iii) 命名確定後のフォーカス復帰（キャンバスへ） →(iv) Esc多段階4層 →(v) 矢印移動時のカーソル追従スクロール。
- **忍者検証を厚く**：T-023のAutomation Peerでセル/種別/選択/フォーカス所在が走査可能になったので、各ステップで
  「今どこにフォーカスがあるか」を確認する。疑わしい挙動は物理クリック/実キーで再検証（UI Automation偽結果の罠）。
- **拙速な一括投入をしない**：「一度に全部入れて動いた気になる」を避ける。検証優先で必要なだけ往復してよい
  （増分は計画的ステップであり、バグ修正の往復2周上限とは別枠。上限を超える修正往復が出たら家老が都度殿へ報告）。

## 論点2（核心）：Enter=配置確定 の意味論

**背景**：T-021の本丸。キーボードファースト方針の看板部分。

**選択肢**
- **案X（フル・キーボードファースト）**：選択セルにカーソルがある状態でEnter→アクティブなツールの
  要素をそのセルに配置し、続けて浮動インライン入力へフォーカス。マウス不要で
  「移動(矢印)→配置(Enter)→命名(入力)→確定(Enter)」が一気通貫。方針に忠実だが、
  GuiEcadが苦しんだ領域＝実装・検証を丁寧に要する（規模：中〜大）。
- **案Y（最小）**：Enterは浮動インライン入力の確定に留め、配置自体はマウスクリック主体のまま。
  安全だがキーボードファーストの看板は弱まる（規模：小）。
- **案Z（折衷）**：案Xを実装するが、まず「1要素の配置＋命名の一気通貫」だけに絞り、連続配置や
  複雑ケースは別タスクへ切り出す（規模：中）。← 家老推奨（GuiEcad失敗史を踏まえ段階導入）

## 論点3：Esc の多段階キャンセル レイヤー定義

**背景**：design-brief 原則2＝「Esc=1階層キャンセル、複数回Escで多段階に戻れる（AutoCAD方式）」。
現状はEscで配置ツール→選択モードに戻す1層のみ。多段階の「階層」を定義したい。

**家老の提案するレイヤー順（内→外）**
1. テキスト編集中（浮動インライン入力）→ Escで編集キャンセル（入力破棄）
2. 配置モード中（ツール選択中）→ Escで選択モードへ戻す
3. 要素選択中（選択セルあり）→ Escで選択解除
4. 何もなし → 何もしない（またはキャンバスフォーカス維持）

※ Escを押すたびに内側の階層から1つずつ外へ戻る。

## 論点4：モーダル非ネスト規約の明文化

**背景**：GuiEcadはダイアログ入れ子でフレームワーク制約に衝突（B1）。ecad2は浮動インライン＝非モーダルで
既に回避済み。これを設計規約として明文化する。

**規約案**：「ecad2はモーダルダイアログを入れ子にしない。入力待ちUIは浮動インライン（非モーダル）を
基本とし、やむを得ずダイアログを使う場合も同時に1枚まで」
