## 3. 過去教訓の反映（最重要）

### 3.1 GuiEcad（WinUI3）のキーボード配置失敗の教訓（`docs/ecad2-keyboard-requirements.md`）

GuiEcadのキーボード配置は9ラウンドの修正の末、フレームワーク既知バグに行き着き機能非表示化で
決着した過去がある。ecad2はWPF（同一バグの対象外）だが、**構造的な教訓はフレームワークを問わず
有効**であり、T-033はまさにその教訓が直撃する領域:

- **A5「配置後、機器名欄へのフォーカス委譲忘れ」**：非モーダル化で「配置確定→入力欄へフォーカス」
  の一手を明示的に書き損ねるリスクは現行モーダルより高い（モーダルは`Loaded`イベントで
  `DeviceNameBox.Focus()`を呼んでいるだけの単純な仕組みだが、非モーダルでは表示/非表示の
  タイミング管理ごと作り直しになる）。
- **A6・教訓C4「Visibility=Collapsed等のVisual Tree離脱操作が暗黙フォーカス委譲を誘発する」**：
  非モーダルバーを閉じる際（OK確定・キャンセルとも）、バー自身またはその子要素がフォーカスを
  持ったまま非表示化すると、意図しない要素へフォーカスが委譲されうる。**キャンバスへの
  `FocusCanvas()`明示呼び出しを、バーを閉じる**前**ではなく閉じた直後に確実に行う**必要がある
  （現行`TryPlaceElement`末尾の呼び出し位置をそのまま踏襲できるかは増分1で要検証）。
- **教訓C3「フォーカスを持つコントロールの組み込みキー処理がグローバルショートカットより優先」**：
  非モーダルバー表示中は、機器名`TextBox`がフォーカスを持つため、矢印キー等はテキスト編集に
  奪われて当然でよい（現行モーダルと同じ挙動）。問題は**バーを閉じた後**にキャンバス側の
  グローバルショートカット（F5-F8, Esc等）が正しく機能を取り戻すか。

### 3.2 T-021モグラ叩き構造の教訓（`docs/ecad2-t021-focus-design-consolidation-plan-onmitsu.md`）

- T-021では「フォーカス喪失のたびに個別ハンドラへ`FocusCanvas()`を後追いで足す」方式が3往復の
  モグラ叩きを生み、さらにEnter/Space非対称バグ（`LostKeyboardFocus`のNewFocus判定が原理的に
  不成立）という根深い罠に到達した経緯がある。**同じ轍を踏まぬため、T-033でも「個別イベントで
  都度FocusCanvas()を足す」場当たり実装ではなく、バーのライフサイクル（表示開始→確定/取消→
  非表示）の中で**フォーカス遷移点を最初から一箇所に集約**した設計にする**（T-021の教訓＝
  「入口を後から複数箇所に増やす」設計は破綻しやすい）。
- Enter/Spaceの非対称性が生じた根本原因は「一時的なフォーカス喪失」と「真の確定/取消」を
  `NewFocus`値だけで区別しようとしたこと。T-033のOK/キャンセルボタンでも同種の罠（ボタン押下時の
  一時的フォーカス移動と、バー全体を閉じる遷移を混同しない）に注意する。

### 3.3 非モーダル化特有の新規リスク（T-021には無かった論点）

- 現行は`ShowDialog()`の**同期呼び出し**が「OK/キャンセルの結果を受け取ってから配置処理・
  `FocusCanvas()`を呼ぶ」という`TryPlaceElement`の制御フローを支えている。非モーダル化すると
  この同期構造が失われ、**OK/キャンセルは非同期イベント（ボタンClick等）でしか受け取れなくなる**。
  `TryPlaceElement`は「バーを表示するだけ」で処理を終え、確定処理・`FocusCanvas()`・
  `RedrawCanvas()`はバー側のOK/キャンセルハンドラへ移設する構造変更が要る（範囲内の技術判断だが、
  影響箇所として明記）。
- **IsDefault/IsCancelの既定ボタン処理はWindowレベルの仕組み**（`Window`の`AcceptsReturn`とは
  別に、既定ボタン・取消ボタンの実装はWindow内部のコマンド束縛に依る）。非モーダル化の実装技術
  として`Popup`を使う場合、Popup自体は独立したWindow(HWND)を持つため、**IsDefault/IsCancelが
  同様に機能するかは未検証**。`Canvas`上への直接オーバーレイ（同一Window内の要素として重ねる）
  であれば同一Windowの機構がそのまま使えると見込まれるが、これも未検証（4.2のPoCで確認）。
- **モーダル非ネスト規約**（T-021で確認された既存規約＝モーダルは同時1枚）との整合：非モーダル
  バー表示中に、ファイル保存確認等の別モーダルを開く経路が現状あるか要確認（無ければ規約影響なし）。
- **UI Automation経由の操作性**（`ecad2-ui-automation`スキルが依拠）：現行はモーダル`Window`が
  独立したUIAツリーのルートになるため要素特定が明確。非モーダル化で同一Window内の要素になると、
  既存のUIA操作パターン（ウィンドウ切り替え前提の待ち合わせ等）に影響しうる。T-021でも
  「UIA Invoke経由の操作がPreview系ハンドラを迂回する恐れ」が実際に懸念点として挙がった前例あり。

## 4. 増分分割案

検証パイプライン既定順（侍実装→隠密静的レビュー→忍者実機確認）を各増分で回す。

### 増分1（最優先・最高リスク）：非モーダル化の骨格のみ

- 現行の縦積みレイアウト・現行の中央表示位置は**変えず**、`Window`の`ShowDialog()`から
  非モーダル表示（技術選定は侍裁量、4.2のPoCで検証してから決める）へ骨格だけ差し替える。
- `TryPlaceElement`の同期構造を崩す最大の変更点はここに集約し、フォーカス制御・Enter/Esc確定・
  キャンセル時の原子的取消（分岐B）が**非モーダル化後も成立する**ことを最優先で確認する。
- 見た目（位置・レイアウト）は増分2以降に送り、ここでは「モーダル→非モーダル」の構造変更単体の
  リスクを先に潰す。

### 増分2：位置・レイアウトの殿注文3点

- カーソル直下表示（`CellRectDip`＋座標変換）
- 小型横長化＋接点選択/機器名入力の横並び
- OK/キャンセルボタンの見切れ修正
- 画面端でのバー位置クランプ（右端・下端に近いセルで画面外へはみ出さないようにする、根拠は
  T-021增分(v)の「グリッド端クランプ」と同種の考慮）

### 増分3：拡張表示ボタンの追加（配置のみ）

- バー上に拡張表示ボタンを追加するが、押下時の詳細画面はT-028のまま未実装（今回はボタンを
  置くだけ、クリックしても無反応で構わない旨をXAML上のコメントで明記）。

## 4.2 PoC要否

**増分1（非モーダル化の骨格）は本実装前にPoC先行を推奨**。理由：

- T-021の教訓（フォーカス制御はコード推論だけで往復を重ねやすい高リスク領域）が直撃する
- 技術選定（Popup／同一Window内オーバーレイ等）そのものが未検証で、IsDefault/IsCancelの
  可否・Enter/Esc確定・キャンセル時のフォーカス委譲先が机上では断定できない
- `poc/`は実験場としてCLAUDE.mdで明示的に許容されており、T-021でも同様にPoC先行で成立させた実績あり

PoCで最低限確認したい項目（案、家老・殿確認の上で確定）：
1. 選択した非モーダル実装技術で、Enterキー確定・Escキー取消が機器名`TextBox`フォーカス中に効くか
2. OK/キャンセル後、キャンバスへのフォーカス復帰が確実に起きるか（暗黙委譲任せにせずFocusCanvas()相当を明示）
3. バー表示中に他のグローバルショートカット（F5-F8等）が誤って反応しないか（モーダルの背後入力
   ブロックが無くなることの影響確認）

## 5. 忍者検証観点（将来、各増分完了時に依頼する想定）

- Enter/Esc/マウスクリックの三経路すべてでOK確定・キャンセルが対称に動くこと（T-021のSpace非対称
  バグの再来がないか、UI Automation Invoke経由も含めて確認）

- 画面端セルでのバー位置がウィンドウ外にはみ出さないこと
- ズーム時（150%等）のバー位置ずれがないこと
- モーダル非ネスト規約の維持（他モーダルとの共存確認）
- 拡張表示ボタンは見た目のみで押下無反応でよい
