## 0. 事前調査で判明した受け入れ地形（結論のみ）

| 領域 | 状態 | 対応 |
|---|---|---|
| 境界ガード（`IsWithinGridBounds`等10箇所超） | `sheet.Grid.Rows`を都度動的参照、固定値キャッシュ無し | **既存流用可**（Rows値を変えるだけで自動追従） |
| 描画（`TotalRows`/`PageSize`/`PageCount`） | 同上、`Math.Max(Grid.Rows, maxRow+1)`で自動追従 | **既存流用可** |
| DRC・ネットリスト・永続化 | Grid.Rows/Columnsへの専用処理なし（永続化は汎用JSON POCO） | **既存流用可** |
| Undo機構 | 未実装が確定済み（T-051）。他操作と同じ「直接List操作+MarkDirty()」流儀 | **考慮不要** |
| 末尾行の加減算 | 相当機能なし | 新設要（シフト処理は不要） |
| 任意位置への行挿入・削除 | 相当機能なし。要素5種（後述）の行番号シフトが必要 | 新設要（最もリスクが高い） |
| 右クリックコンテキストメニュー | ecad2に前例が一切ない | 新設要（UI要素として初出） |
| シート設定ダイアログ（Grid.Rows数値入力＋Bus名編集） | `RenameDialog`/`AddSheetDialog`と同型の新設が必要 | 新設要（既存パラメータ渡し流儀は流用可） |

## 1. 増分の切り方と順序（リスク低→高）

高リスク領域（GuiEcadが躓いた経験のある種のUI操作＝キーボード配置同様の「座標・範囲・シフト」を扱う処理）は
実装速度より検証優先の原則（過去の家老裁定）に従い、**シフト処理を伴わない増分を先に固め、
最もリスクの高い任意位置挿入・削除を最後に回す**。各増分は独立にビルド・テスト・忍者実機確認が可能な粒度とする。

### 増分1：末尾行の追加・削除（ツールバー＋キーボードショートカット）

**DoD**: ツールバー「行＋/行－」・キーボードショートカット(Ctrl+Shift+Up/Down)経由でGrid.Rowsが
1増減し、上限60・下限1で正しくクランプされることをテストで実測確認（RED→GREEN）。全件テスト合格。

**内容**:
- `SheetNavigationViewModel`（またはより適切な置き場所、要検討）に末尾行追加・削除のコマンドを新設
  - 追加: `Grid.Rows + 1`（上限60でクランプ、60到達時は無効化）
  - 削除: `Grid.Rows - 1`（下限1でクランプ、1到達時は無効化。既存要素が削除される最終行に存在する場合の扱いは
    増分1の範囲では「削除しない」または「警告」のどちらにするか要確認——下記「開かれた論点」参照）
  - MarkDirty()呼び出しを既存流儀に揃える
- ツールバーへ「行＋」「行－」ボタン追加（`MainWindow.xaml`、配置は下記論点）
- キーボードショートカット新設（`MainWindow.xaml.cs:589`の`Window_PreviewKeyDown`へ追加）：
  - `Key.Up or Key.Down when Keyboard.Modifiers == (ModifierKeys.Control | ModifierKeys.Shift)`
  - 既存Ctrl+S/O/N同様、グローバル（キャンバスフォーカス不問）ショートカットとして追加

**要素シフト**: 不要（末尾操作のため）。

**RED先行証明**:
1. 上限クランプ: Rows=60の状態で追加コマンドを実行→Rows=60のままであることをテストで先に書き、
   クランプ未実装の状態でRow=61に増えてしまう（REDになる）ことを実測
2. 下限クランプ: Rows=1の状態で削除コマンドを実行→Rows=1のままであることを同様にRED証明
3. 通常加減算: Rows=10→追加→11、削除→10の単純ケース

**テスト観点**: 上限到達直前/到達/超過、下限到達直前/到達/超過（境界値分析の下限-1/下限/上限/上限+1）。

### 増分2：シート設定ダイアログの新設（Grid.Rows数値入力＋Bus.LeftName/RightName編集）

**DoD**: 新設ダイアログでGrid.Rows（1〜60）とBus.LeftName/RightNameを編集でき、入力値が
Sheetへ正しく反映されること、上限/下限外入力が拒否されることをテストで実測確認。全件テスト合格。

**内容**:
- `RenameDialog.xaml(.cs)`/`AddSheetDialog.xaml(.cs)`と同型の新規モーダルWindow
  （View側で`new`・`ShowDialog()`→結果を`ICommand.Execute(param)`で渡す既存流儀を踏襲）
- 入力項目: Grid.Rows（数値、Minimum=1/Maximum=60）、Bus.LeftName（文字列）、Bus.RightName（文字列）
  - GuiEcadにある「この母線名を新規シートの既定にする」チェックボックス相当を移植するかは
    殿裁定範囲外の追加機能ゆえ**移植しない**（殿裁定3点は行追加UI・既定値統一・母線名の正体確定のみ。
    既定値保存機構は範囲外、必要なら別途起票）
- ViewModel側に新規コマンド（仮称`UpdateSheetSettingsCommand`）、パラメータは専用DTO
  （`AddCommand`の`ValueTuple<string,bool>`方式に倣うか、3値以上のため専用recordにするかは実装時に判断）

**要素シフト**: 不要（ダイアログ経由でもGrid.Rows直接代入のみ、増分1と同じクランプ規則を共有）。

**RED先行証明**: ダイアログの数値入力バリデーション（Maximum=60超入力の拒否、Minimum=1未満の拒否）を
増分1のクランプロジックと共有する設計にすれば、二重実装を避けられる（要検討、下記論点参照）。

**テスト観点**: 境界値入力（0・1・60・61）、Bus名の空文字許容可否（GuiEcadは自由記述、空文字も許容と見られる
——要確認）。

### 増分3：任意位置への行挿入・削除（右クリックコンテキストメニュー）

**最もリスクが高い増分**。着手前にリスク検証（PoC）を挟むことを推奨。

**DoD**: 右クリックコンテキストメニュー経由で任意位置に行を挿入・削除でき、挿入/削除点より後ろの
5種類の要素（ElementInstance/VerticalConnector/WireBreak/GroupFrame/RungComment）のRowが
正しく一括シフトされることを型ごとにテストで実測確認。全件テスト合格。忍者実機確認必須
（GuiEcadでも複雑な既存実装だった領域ゆえ、単体テストのみで完了とせず実機での見た目確認を要する）。

**内容**:
- `ContextMenu`をecad2で初めて新設（キャンバス上での右クリック時表示）
- メニュー項目（GuiEcad文言踏襲、殿確認要）: 「行{N}の前に行を挿入」「末尾に行を追加」「行{N}を削除」
- 要素の行番号シフト処理（`RowOps.ShiftRows`相当）を新設。対象は事前調査で判明した5種:
  - `ElementInstance.Pos.Row`（`GridPos`のRow成分）
  - `VerticalConnector.TopRow`/`BottomRow`
  - `WireBreak.Row`
  - `GroupFrame.TopLeft.Row`／`GroupFrame.Height`（**GuiEcad方式踏襲で確定、家老裁定2026-07-10**。
    単純な「位置のみシフト」ではなく、挿入/削除点と枠の位置関係で次のとおり分岐する
    ——詳細は`docs/archive/ecad2-t055-increment3-precheck-onmitsu.md`§2.3参照:
    - **挿入**（`targetRow`の前に1行挿入）:
      - 枠の開始行が挿入点以降（`TopLeft.Row >= targetRow`）→ 枠ごと下へ1行シフト（位置のみ、Height不変）
      - 枠の開始行が挿入点より前だが範囲が挿入点にかかる（`TopLeft.Row + Height > targetRow`）→
        `Height++`（内部挿入、位置不変）
    - **削除**（`targetRow`を削除）:
      - 枠が削除行より完全に下（`TopLeft.Row > targetRow`）→ 位置のみ-1シフト
      - 削除対象行が枠の内部（`TopLeft.Row < targetRow`かつ`TopLeft.Row + Height - 1 >= targetRow`）→
        `Height--`（内部詰め、位置不変）
      - 枠の開始行そのものが削除対象（`TopLeft.Row == targetRow`）→ 枠ごと削除
        **※このケースは「削除対象行に要素がある場合」論点に該当し、殿確認待ちで実装保留（2026-07-10家老指示）**）
  - `Sheet.RungComments`の`RungComment.Row`
  - （`Sheet.Lines`の`CircuitLine.Row`は現状書き込み箇所が見当たらない未使用フィールドの可能性——
    移植対象に含めるかは要確認）
- 挿入点より後ろの行を持つ全要素のRowを+1（挿入）または-1（削除、詰め）する共通ロジックを新設
- 削除位置に要素が存在する場合の扱い（GuiEcadは「Undo可能な形で削除」だが、ecad2はUndo非対応のため
  単純に要素ごと削除するか、削除を拒否するかは殿確認要——下記論点参照）

**RED先行証明**:
1. 挿入点より後ろの要素が正しくシフトすること（挿入点=行3、行3以降の要素が全て+1、行0-2は不変）
2. 削除点より後ろの要素が正しく詰まること（同様に-1）
3. 削除位置に存在する要素の扱い（論点確定後に応じたテスト）
4. 5種類の要素型それぞれで独立に確認（一括シフトロジックの共通化は行うが、テストは型ごとに書く）

**テスト観点**: 先頭挿入・中間挿入・末尾挿入・下限（削除で1行になるケース）、複数要素が同一行に存在する場合の
一括シフト、GroupFrameの範囲外にまたがる挿入/削除（Frame内部への挿入でHeightを拡張すべきか、Frame境界外への
挿入としてHeightは不変か——要確認）。

## 2. 開かれた論点（殿確認が必要、UI/UX分岐【MUST】）

殿裁可の前に、以下の分岐点を確認いただきたい。侍の裁量では決めない。

1. **行＋/行－ボタンの配置**: ツールバーBand=0の「元に戻す」等と同型の大型ボタン（`ToolBarButtonStyle`）か、
   左パレットの「シート＋/－」と同型の小型ボタン対（`Width=28`）か。
2. **削除される最終行に要素が存在する場合の扱い**（増分1・増分3共通の論点）: 削除を拒否する（警告を出す）か、
   要素ごと削除するか。GuiEcadの挙動を確認した限りでは明記が無かった（隠密調査書に追加確認が要る旨の記載あり）。
3. **コンテキストメニューの文言・項目構成**（増分3）: GuiEcad文言（「行{N}の前に行を挿入」等）をそのまま
   踏襲するか、ecad2の既存UI文言トーンに合わせて調整するか。
4. **シート設定ダイアログの構成**（増分2）: Grid.Rows・Bus名を1つの新規ダイアログにまとめるか、
   既存の`RenameDialog`を拡張する形にするか（後者は「シート名変更」という既存の意味範囲を超えるため
   侍としては新規ダイアログを推奨するが、最終判断は殿へ委ねる）。
5. **増分の優先順序自体**: 本書はリスク低→高（末尾操作→ダイアログ→任意位置挿入）の順で提案しているが、
   殿の実用上の優先度（例えば「まず既定行数を10のままでも母線名だけ先に編集できれば急場をしのげる」等）が
   あれば増分2を先に回すことも可能。
6. **`Sheet.Lines`（`CircuitLine.Row`）を増分3のシフト対象に含めるか**: 現状書き込み箇所が見当たらない
   未使用フィールドの可能性がある。含めても実害はないが、含めない場合は「対象外とした理由」を
   コメントに残す形で明示的にスコープアウトする。
