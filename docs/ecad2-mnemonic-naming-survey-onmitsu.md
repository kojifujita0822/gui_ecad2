# ラダー接点ニーモニック表記の調査（隠密）

> 2026-07-03 隠密調査。家老依頼：T-026で内部ニーモニックにAND/NAND/OR/NORを便宜的に使ったが
> UI表示は「a接点」「b接点」「ORa接点」「ORb接点」のまま。今後のUI表示用語統一の判断材料として、
> GX Works3・公的標準・GuiEcadの表記実態を調査した。

## 結論

**GX Works3・GuiEcadともに「内部識別子は英語、UI表示は日本語ラダー用語（a接点/b接点等）」という
使い分けで一致している。**「a接点」「b接点」はJIS C 0617に由来する正式な日本語呼称であり、
AND/OR等の論理演算名を回路記号の表示名として単独採用するのは、PLC業界の慣行（ニーモニック命令
「接点命令」カテゴリ）との意味衝突リスクがあるため避けるべき。

## 根拠

### 1. GX Works3マニュアル（SH-081214）
- メニュー/ツールバー/機能名は一貫して**「a接点」「b接点」表記**（「A/B接点変更」機能も存在）。
  「2.4 メニュー一覧」章で`[回路記号][a接点]`、`[回路記号][b接点]`、`[回路記号][a接点OR]`、
  `[回路記号][b接点OR]`等の項目名を確認。
- 「AND」「OR」「NOT」は回路記号の名前としては使われておらず、**ニーモニック命令
  （LD/AND/ANI/OR/ORI等）**として「接点命令」カテゴリに分類される別概念。これとは別に
  「論理演算命令」カテゴリ（ワード単位ビット演算：BKAND/WOR等）も存在し、**AND/ORという語は
  両カテゴリで意味が異なる**（接点命令の"AND"＝直列接続、論理演算命令の"AND"＝ビットAND演算）。
- 「ノーマルオープン」「ノーマルクローズ」等の英語カタカナ表記は確認できず（LIKE検索の網羅性は
  完全ではないため断定はしない）。

### 2. JIS/IEC 61131-3等の公的標準
- JIS B 3503（IEC 61131-3のJIS版、IDT一致）はLD/FBD/IL/ST/SFC言語を規定するが、接点記号自体は
  **JIS C 0617（電気用図記号）に依拠**する構造。
- IEC 61131-3のLD言語では接点は"Normally Open (NO) contact"／"Normally Closed (NC) contact"と
  して規定。
- **「a接点」「b接点」はJIS C 0617に規定された正式呼称**（「メーク接点」「ブレーク接点」とも併存）。
  語源の一次資料は確認できず不明。

### 3. GuiEcadの表記実態
- 内部識別子（英語enum）: `ElementKind.ContactNO`, `ContactNC`, `Coil`, `PushButtonNO`,
  `TimerContactNO`等（`GuiEcad.Core\Model\Element.cs`）。
- UI表示（日本語）: `MainPage.xaml`で`Tag="ContactNO"`のRadioButton内`TextBlock.Text="a接点"`、
  `Tag="ContactNC"`は`"b接点"`。他「押釦NO」「押釦NC」「タイマNO」「タイマNC」「瞬時NO」
  「瞬時NC」「コイル」「表示灯」「端子」「セレクトSW」「サーマル(OL)」「非常停止」「三相モータ」
  等。**AND/OR/NOTのような論理演算名の画面露出はゼロ**。
- `Tag`文字列はXAML→enum変換の内部連携値でユーザー非表示（`MainPage.Tools.cs`の
  `Enum.TryParse<ElementKind>`）。ツールチップも設定箇所は全て日本語。`AutomationProperties.Name`
  は未使用、`AutomationId`のみ英語（UI Automation専用、非表示）。
- リソースファイル（.resw等）は0件、UI文字列はXAML/コードに日本語で直接ハードコード。

## 推奨案（ecad2への適用）

1. **内部識別子は英語enumのまま維持**（GuiEcadの`ElementKind`踏襲、T-026のAND/NAND/OR/NORも
   内部識別子としては問題なし）。
2. **UI表示は「a接点」「b接点」「ORa接点」「ORb接点」等の日本語ラダー用語で統一**（現状のまま
   が適切）。GX Works3・GuiEcadの両方でこの慣行が確立しており、日本の制御盤設計者に最も馴染む
   JIS C 0617由来の正式呼称でもある。
3. **「AND」「OR」を回路記号の表示名として単独採用することは避ける**。GX Works3の「論理演算
   命令」（ビット演算）カテゴリと意味が衝突するリスクがある。必要であれば「並列挿入」等の機能名の
   補助修飾語に留める。
4. GuiEcadの「押釦NO」のようにNO/NC略称を和文複合語に混在させる書式を維持するか、完全な日本語
   表記（例:「押釦a接点」）に統一するかは、現状踏襲でも問題ない実績があるため、既定路線として
   「押釦NO」形式の維持を推奨するが、最終的な文言統一はUI/UXブリーフでの一括レビュー時に確認する
   のが良い。

## 不明点
- 「a接点」「b接点」の語源（ドイツ語arbeiten由来説等）の一次資料は未確認
- IEC 61131-3規格書原文（有料）は未取得、条文番号までは未確認
- GX Works3マニュアルのLIKE検索は部分一致のため表記ゆれの完全網羅は保証できない
