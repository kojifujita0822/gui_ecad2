# P-012原因調査：行0の母線未描画・シンボル未描画+ラベル誤表示（隠密）

> 2026-07-06 隠密調査（診断調査、修正なし・往復に数えず）。家老依頼。
> 症状：(1) 行0（最上段）へ要素を配置すると母線（水平線）が未描画 (2) 行0で基準要素を配置すると
> シンボル自体が未描画＋別要素のラベルが誤表示（2026-07-06忍者観測、行0以外は正常）。
> 要素データ自体は正常（プロパティ・機器表反映OK）で描画層の問題と推測されている。

---

## 結論

**DiagramRenderer周りの静的調査では、「行0を特別扱いする分岐」「row=0でのみ破綻するオフバイワン計算」は発見できなかった（否定的結果）。座標計算はすべて行番号に対して線形で、Grid.Rows既定値22により通常のワークフローでは行数不足も起きない。座標を実際に計算した結果、唯一見つかった「行0関連」の記述（母線名ラベルとの近接コメント）も、現在の値では視覚的重なりが起きない距離が確保されており、原因として弱い（REFUTED寄り）。静的解析だけでは根本原因を特定できず、実機での動的検証（座標値の実測）が必要という結論に至った。**

---

## 調査した箇所（いずれも「問題なし」と判定）

### 1. 行番号・座標変換系（`GridGeometry.cs`）
`YRow(int row) => MarginMm + (row + 0.5) * CellMm`。row=0でも通常の正の値（既定値でMarginMm=20, Cell=9なら YRow(0)=24.5mm）。負値・ゼロ除算・特殊分岐なし。

### 2. `DiagramRenderer.Render()`（190-204行目）
`DrawRails`/`DrawRungWires`/`DrawConnectors`/`DrawElement`の呼び出しは全て`rowStart`/`rowEnd`/`InWindow(row)`によるページ範囲チェックのみで、row=0を特別扱いする分岐はない。`InWindow(row) => row >= rowStart && row < rowEnd`（rowStart=0なら0は正しく含まれる）。

### 3. 母線ノードの番兵座標（`NetlistBuilder.cs:31-32`）
```
int leftRail = Node(-1, -1);   // 番兵: 実セル行は 0 以上なので衝突しない
int rightRail = Node(-1, -2);
```
コメント通り、番兵座標はRow=-1で行0（Row=0）とは衝突しない設計。`AssignWireNumbers`の`kv.Key.Row < 0`除外処理も母線ノードのみを対象とし、行0の要素ノードには影響しない。

### 4. `LeftTerminator`/`RightTerminator`（373-392行目、母線描画の要）
`c.BottomRow == row`という条件でフィルタする設計。OR入力で行0が基準行（TopRow側）になるケースでは、`BottomRow`は新要素の行（行0より下）なので、行0自体の母線接続には影響しない（TopRow側は常に母線へ繋がる設計、P-003の修正コメントと整合）。

### 5. `MainWindowViewModel.PlaceElementAtSelectedCell`のOR/縦コネクタ生成ロジック（317-359行目）
`baseRow`は`el.Pos.Row < pos.Row`の`Max()`で決まる単純な計算。行0が基準行に選ばれても`TopRow=0`の`VerticalConnector`が生成されるだけで、特別扱いや0を「未設定」と誤認する分岐は見当たらない。

### 6. `WpfRenderer.cs`（座標変換の実装）
`PushTransform`/`DrawLine`/`DrawText`等はいずれも純粋な座標×係数の変換のみ。行番号を意識する実装ではない。

### 7. `PartDrawing.cs`/`SymbolGlyphs.cs`（記号本体の描画）
いずれもローカル座標系（`PushTransform`済み、原点=要素位置）のみを扱い、行番号は関知しない。

### 8. `MainWindow.xaml.cs`の`RedrawCanvas`/`TryPlaceElement`
`RedrawCanvas`はCurrentSheet/SelectedCell変更時に都度`LadderCanvasHost.Draw()`を呼ぶのみ。`TryPlaceElement`も行0固有の分岐なし。

## 唯一見つかった「行0」関連の記述と検算

`DiagramRenderer.DrawBusLabels`（237-243行目）にコメントあり：
```
// 第1ステップ（1行目の機器名ラベルは母線上端付近に出る）と離すため、
// ヘッダーは上端寄りに置いて余白を確保する。
double yLabel = Math.Max(2.5, _opt.MarginMm - 8.5);
```
これは開発者（侍）が過去に「行0（1行目）の機器名ラベルが母線名・電圧ラベルに近接する」ことを認識し、意図的に距離を確保した設計対応。**実際に座標を計算して重なりの有無を検算した**（既定値 MarginMm=20mm, Cell=9mm）:

| 項目 | 計算式 | 値 |
|---|---|---|
| 母線名/電圧ラベルY座標 | `Math.Max(2.5, 20-8.5)` | 11.5mm |
| 行0要素ラベルY座標（接点、dy=-1.5） | `YRow(0) - Cell*0.5 - dy = 24.5 - 4.5 + 1.5` | 21.5mm |
| 差 | | **10.0mm**（Cell=9mmより大） |

10mm離れており、Cell幅（9mm）より大きい距離が確保されているため、**この2つが視覚的に重なって「シンボルが隠れる」可能性は低い**（REFUTED寄りの判定）。ただし、この計算は`RenderOptions`の既定値（MarginMm=20, Cell=9）を前提にしたものであり、実際の実行時オプション値が異なれば結果も変わりうる（要素確認）。

## 静的解析の限界・不明点

- 「シンボル自体が描画されない」「別要素のラベルが誤表示される」という具体的な症状は、今回調査した範囲のコードロジックからは再現条件を特定できなかった。実際にコードが実行する座標値・描画順序が忍者の観測結果とどう対応するかは、静的読解だけでは確認しきれない。
- 忍者の観測手順（新規シート作成後、具体的にどのセルにどの順で要素を配置したか）の詳細が本調査では不明。「基準要素」という言葉がOR入力の基準行（縦コネクタのTopRow）を指すのか、単に「最初に置いた通常の要素」を指すのか、記録からは断定できない。
- 実機のスクリーンショット・座標値ログが無いため、「別要素のラベル」が具体的にどの要素のラベルで、どこに誤表示されたか（位置・内容）が分からない。

## 派生提案（範囲外の気づき、決定は家老）

根本原因を確定するには、以下のいずれかの動的検証が次善と考える：
1. 忍者に再現手順を詳細記録してもらう（新規シート作成→配置順序→症状発生タイミングのスクリーンショット付き）。
2. T-038/T-039の診断ログ方式を借りて、`DrawElement`/`DrawElementLabel`呼び出し時の実際の座標値（`X(lb)`, `YRow(row)`）を一時的に出力し、忍者に実測してもらう。

いずれも実装・実機操作を伴うため、隠密の裁量を超える（家老の采配事項）。
