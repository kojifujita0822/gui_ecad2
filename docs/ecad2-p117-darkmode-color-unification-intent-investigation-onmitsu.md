# P-117 ダークモード配色統一の意図確認調査（隠密）

日付: 2026-07-22
契機: `docs/proposed.md` P-117。ダークモードで空状態（要素配置前）と作業領域（要素配置後）の
背景色が同一色（`#FF202224`）になり視覚的区別が消えている件（T-077増分3陳腐化点検2026-07-21発見）。
仕様書3節はライトモード限定の古い記述（区別ありの前提）のまま。
依頼: T-083（ダークモード対応）当時のコミット・レビュー記録・実装コードを調査し、この統一が
「意図的な統一判断」か「単なる見落とし」かを判定する（殿裁定2026-07-22＝まずは意図確認のみ）。

## 結論

**見落としの可能性が高い**。ただし当時の実装者（侍）本人への直接確認は行っておらず、断定はできない。

## 根拠

### 1. 増分2着手前、既存コメントに「状態依存」という設計要件が明文化されていた

増分2直前の`App.xaml`（コミット`1235898`の差分元、削除された側）:

```
GX Works3実機調査(T-012)より: 配色は状態依存。
空状態(プロジェクト未作成)=濃紺、作業領域(ラダーエディタ内部)=白+黒グリッド。
グリッド色はEcad2.Core.Rendering.DrawingTheme.GridGray(#D2D2D2)と一貫させる。
```

「配色は状態依存」という設計意図が、増分2着手時点でコード上のコメントとして直接目に入る場所に
存在していた。

### 2. 増分2の采配・着手前調査の目的に「区別の維持」への言及がない

`docs/todo-archive.md`2398-2404行（本実装着手前調査完了、隠密、`ecad2-t083-honjissou-3layer-
design-survey-onmitsu.md`）：増分4分割案の記述は「増分2:UIクローム基盤構築+**シート0件時
キャンバス色固定バグ解消**(層C基盤)」。

この「バグ」の実体は`docs/todo-archive.md`2392-2395行（PoC完了時の忍者所見）：「シートが0件の
時のみトグルOn/Offに関わらずキャンバスが**固定の濃紺色になる**（シート追加後は正しく追従）」
——つまり元のバグは「ダークモードにしても空状態の色が変わらない」という**トグル無反応**の話で
あり、「空状態と作業領域を区別するか」とは別の関心事。増分2の目的定義・家老裁定文のいずれにも
「ダークモードでも状態依存の区別を維持する」という論点は見当たらない。

### 3. 実装コメントは「Core⇔App間の一貫性」のみ言及、「状態間の区別」への言及がない

`Theme.Dark.xaml`（コミット`1235898`導入、現行まで不変）:

```
T-083増分2(層C=UIクローム基盤、家老采配2026-07-16): ダークテーマの配色定義。
WorkAreaBackgroundBrush/EmptyStateBackgroundBrushは層A(Ecad2.Core.Rendering.DrawingTheme.Dark)
のBackground(#FF202224)・GridColor(#FF464A50)と値を揃え、テーマ全体の一貫性を保つ。
```

「テーマ全体の一貫性」は**Core層(作図キャンバス)とApp層(UIクローム)の間**の色の一貫性を指して
おり、「空状態⇔作業領域という状態間の区別」という別軸への言及・検討の形跡がコメント上ない。
実装者（侍）が「区別を消す」という選択そのものを意識した形跡は見当たらない。

### 4. 忍者の実機検証観点も「トグル反応」のみで「区別の有無」を検証観点に含んでいない

`docs-notes/verification-ninja-t083-zoubun2.md`観点2:「シート0件のままダークモードON/OFFを
切替えたところ、ON時は黒、OFF時は濃紺と、シートが無い状態でも正しく連動して切り替わることを
確認」「シート追加後（シート1）でも黒背景・グレーグリッド線で正しく表示されることを確認済み」。

空状態とシート追加後の色を別々に「黒である」とだけ確認しており、両者が**同一色**であることへの
言及・気づきは記録にない。検証観点自体に「区別が保たれているか」という項目が最初から設定されて
いなかった。

### 5. 静的レビュー（隠密）の専用記録が見当たらない

`docs-notes/`・`docs/`配下を横断検索したが、T-083増分2単独の隠密静的レビュー記録ファイルは
見つからなかった（増分3以降は「隠密の事前指摘的中」等の言及があるが、増分2については無し）。
検証パイプライン（侍実装→隠密静的レビュー→忍者実機確認）の隠密工程で本件が拾われた形跡がない。

### 6. 導入から5日後、別タスクの陳腐化点検で偶然発見された

増分2導入（2026-07-16）からT-077増分3陳腐化点検（2026-07-21）まで約5日間、本件は誰にも
気づかれていない。発見の経緯も「ダークモード対応の意図的判断か」を検討する作業からではなく、
`docs/spec`の陳腐化点検（仕様書とコードの整合確認）の副産物だった。

## 総合判断

上記6点はいずれも「意図的に区別を消す判断を行った」ことを示す記録・証跡ではなく、逆に
「区別維持という既存要件への言及が着手前調査・実装コメント・検証観点のどの段階にも一度も
現れない」という**不在の一貫性**を示している。最も自然な解釈は、実装者が「シート0件時の
トグル無反応バグを直す」「Core層と値を揃えてテーマの一貫性を保つ」という当面の目的に集中する
過程で、既存の「状態依存配色」という設計要件への抵触に気づかなかった、という**見落とし**。

ただし、以下の限界により完全な断定はできない：
- 当時の実装者（侍）本人への直接ヒアリングは行っていない（記録からの推測に留まる）
- 「口頭・チャットで意図的判断が議論されたが、コミットコメント・台帳に残らなかった」可能性は
  記録調査だけでは排除できない

## 副次的な気づき（範囲外、着手せず）

`Theme.Dark.xaml`のコメントは「層A(Core)のBackground(#FF202224)」と記載しているが、実際の
`src/Ecad2.Core/Rendering/DrawingTheme.cs`96行`Dark.Background = new(255, 32, 34, 38)`は
`Color(byte A, byte R, byte G, byte B)`（同ファイル`IRenderer.cs`9行の定義）に基づくと
`#FF202226`（B成分=38=0x26）であり、`Theme.Dark.xaml`側の実装値`#FF202224`（B成分=36=0x24）と
末尾1桁ズレている。視覚上は判別不能な誤差（実害なし）だが、「値を揃える」という意図に対して
実際には完全一致していない。GridColorは`#FF464A50`で両者完全一致しており、Backgroundのみの
転記時の誤差と見られる。要否は家老判断に委ねる。
