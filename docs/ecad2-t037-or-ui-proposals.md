# T-037 OR入力専用UI 画面案（侍・起草）

## 背景

殿裁定更新（2026-07-06）：T-037は当初案（部品選択リストへORa/ORb追加・固定7種化、隠密調査
`docs/ecad2-p010-or-fixed-parts-investigation-onmitsu.md`の案1）を**実施したうえで**、
加えて「OR入力の専用UI」を新設する。範囲は右パネル・ツールバーでの専用表記まで。インライン
バー（T-033浮動入力）でのOR表記は別途相談のため今回スコープ外。

本書はUI/UX分岐（画面配置・見せ方）のため、家老裁量で決めず殿へ選択肢として提示する
（`task-implementation`スキルの原則、過去T-026の教訓）。侍からの断定は控え、利点・コスト
を併記するに留める。

## 現状構造（整理）

- **ツールバー**：F5=a接点／F6=b接点／F7=コイル／F8=端子台、区切り後にShift+F5=OR a接点／
  Shift+F6=OR b接点（`MainWindow.xaml`149-209行目）。OR系は専用図形を持たず、通常のa接点/
  b接点と同一`PartFolderEntry`を使い、配置後に縦コネクタを自動生成する「配置操作の副作用」
  （`MainWindowViewModel.PlaceElementAtSelectedCell`、`isOr`引数）。アイコンはXAML `Path`の
  手描きで、通常アイコンに縦線を1〜2本追加しただけの区別（196・206行目）。
- **右パネル部品選択リスト**：`PartSelectionList`（`MainWindow.xaml`334-349行目）。1項目=
  [サムネイル24×24][カテゴリ(グレー小文字)][名前]の横並び。クリックは常に`isOr: false`固定
  （`PartSelectionItem_Clicked`）で、OR操作を発動する経路が現状リスト側に無い。
- **サムネイル同形課題**：`PartThumbnailRenderer`は`PartDefinition`の形状のみを描画し、OR/非OR
  の区別情報を持たない。ORa接点をリストに追加しても、サムネイル自体は通常のa接点と同じ絵になる
  （隠密調査所見）。

## 案A：リスト項目に「OR」バッジを重ね描画

**内容**：隠密案1どおりORa/ORbをリストへ論理エントリとして追加したうえで、サムネイル画像に
小さな「OR」バッジ（右下隅、並列線を模した小アイコンまたは"OR"の2文字）を合成描画する
（`PartThumbnailRenderer`側に軽微な拡張が要る）。ツールバーのShift+F5/F6ボタンは現状のまま
維持（変更なし）。

- **右パネル**：リストは現状と同じ1カラム構成のまま、サムネイルにバッジを重ねるのみ。区画の
  新設なし。
- **ツールバー**：変更なし（現行のPathアイコン＋キーラベルを維持）。
- **キーボード操作性**：リスト項目自体はT-026/T-016で確立済みの配置フローに乗るため追加コスト
  なし。ツールバーのShift+F5/F6も既存のまま。
- **実装コスト**：低〜中。サムネイル合成ロジックの追加のみで、新規UI区画・新規操作系列は発生
  しない。
- **利点**：変更範囲が最小。既存レイアウトへの影響がほぼ無い。
- **懸念**：「専用UI」と呼べるほどの見た目の変化は小さく、殿の意図する「専用表記」に対して
  控えめすぎる可能性がある。

## 案B：右パネルに「OR接続」トグルを常設

**内容**：部品選択リストの上部（見出し行の下）に「OR接続」チェックボックス/トグルボタンを
新設し、ONの状態で通常のa接点/b接点/自作パーツ等を選択すると、選択した部品がOR付きで配置
される。ORa/ORbを個別のリスト項目として追加する代わりに、既存項目＋トグルの組み合わせで
表現する（隠密案1のリスト項目追加とは別方式）。ツールバーのShift+F5/F6は維持しつつ、アイコン
に"OR"の文字ラベルを明示追加する（現状は縦線のみで識別しづらいため）。

- **右パネル**：部品選択リスト見出し（"部品選択"）の下に1行、トグルボタン「OR接続」を追加。
  ONの間は視覚的にハイライト（背景色変化等）し、状態を常時可視化する（design-brief 4節#6
  「モード可視化の多重化」と整合）。
- **ツールバー**：Shift+F5/F6のPathアイコンへ"OR"の小さな文字を追加描画し視認性を上げる。
- **キーボード操作性**：トグルへのキー割当（例: 部品選択リストフォーカス中にAlt+Oでトグル）
  を新設する必要がある。トグル→選択の2アクションになるため、単発選択に比べ操作数が1つ増える
  （ショートカットで補えるが、真の1ステップにはならない）。
- **実装コスト**：中。新規トグル状態（ViewModelプロパティ）・XAML区画・キー割当が要る。
  リストの項目数は増えない（T-015レビュー指摘のEntries/SelectionEntries 1:1前提を崩さない
  利点がある）。
- **利点**：GX Works3的な「モード切替＋常時表示」の作法に近い。リスト項目が増えないぶん一覧性
  を保てる。
- **懸念**：2ステップ操作になる点がキーボードファースト方針とやや緊張関係にある。

## 案C：ツールバー・右パネルにOR専用の小区画を新設

**内容**：ツールバーのShift+F5/F6を独立の「OR入力」専用ミニ区画（区切り線で囲った小さな
サブグループ）へ再配置し、「OR」という見出しラベルを常設する（現状は他の配置ボタンと同列の
Separator区切りのみ）。右パネルにも、部品選択リストとは別に「OR入力」専用の小区画（a接点/
b接点2ボタンのみの最小パレット）を新設し、部品選択リスト本体にはORa/ORbを追加しない
（サムネイル同形問題自体を回避する）。

- **右パネル**：部品選択リストの上または下に「OR入力」という見出し付きの小さな2ボタン区画
  （a接点/b接点のみ）を独立配置。既存リストへの項目追加は行わない。
- **ツールバー**：Shift+F5/F6を"OR"見出し付きの小グループとして視覚的に独立させる（区切り線＋
  ラベル）。
- **キーボード操作性**：新設区画のボタンにもF-key相当のショートカットを個別に割り当てれば
  単発操作を維持できる（design-brief 8節「アイコンにF-key併記」の作法を踏襲）。
- **実装コスト**：中〜高。新規XAML区画・新規子ViewModel相当の状態管理が要る。リストとは別の
  入力経路が並存するため、動作の一貫性（配置ロジックの共有）に注意が必要。
- **利点**：「OR入力は特殊操作である」という性格を視覚的に最も強く区別できる。GX Works3の
  「専用ツールバー切替」パターン（design-brief 9節）とも親和性が高い。サムネイル同形問題を
  そもそも発生させない（回避的解決）。
- **懸念**：入力経路が二重化し、部品選択リスト経由でOR配置ができない状態が残る（当初案1が
  目指した「リストとツールバーの選択肢一致」から外れる）。

## 侍所見

いずれも技術的に実現可能で、決定打はUI/UXの意匠判断（殿）に委ねる。強いて分類するなら、
案Aは最小変更、案Bは操作系列の統合を保ちつつモード切替方式、案Cは視覚的独立性を最も重視した
折衷案。「専用UI」という表現の重みをどう解釈するか（軽微な視覚強化で足りるか、独立区画まで
求めるか）が分岐の核だと考える。
