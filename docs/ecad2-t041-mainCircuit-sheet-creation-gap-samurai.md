# 主回路シート作成UIの現状把握・選択肢整理 — 2026-07-07（侍）

家老采配（忍者の範囲外検出「UIから主回路シートを作成する手段が現状無い」）に基づく調査。
**本段は現状把握と選択肢の整理のみ。実装はまだ行わぬ。**

---

## 1. 現状（実コード確認）

- シート追加はツールバー「＋」ボタン→`SheetNavigationViewModel.AddCommand`（唯一の入口）のみ。
- `AddCommand`は`new Sheet { PageNumber, Name, Grid = new GridSpec{Rows=10,Columns=20} }`を
  無条件生成する（`SheetNavigationViewModel.cs:62-73`）。`MainCircuit`は指定されず既定値`false`
  （制御回路）のまま。
- `Sheet.MainCircuit`（`Sheet.cs:27`）を参照する箇所はリポジトリ全体で`DiagramRenderer`内の
  描画分岐（母線描画・自動横配線のスキップ）と、T-041増分2/3で私が追加したガード
  （`ConfirmConnectorDraft`・`TryBeginConnectorDraft`・`TryPlaceWireBreak`の「制御回路シート
  限定」チェック）のみ。**シートの種別を選ぶ・切り替えるUIはApp層に一切存在しない。**
- つまり現状、`MainCircuit=true`のシートを作る唯一の方法は`.gcad`ファイルを手編集するか、
  テストコード内で`new Sheet{MainCircuit=true}`する以外に無い。

## 2. 影響

T-041増分5（`FreeLine`/`ConnectionDot`、主回路シート対象）を実装しても、主回路シートが
UIから作れない限り実機で使えず、忍者検証も成立しない。増分5着手前にこのUIギャップを埋める
必要がある（家老指摘の通り）。

## 3. 選択肢（いずれもUI/UX分岐、殿確認が必要）

### 案1: シート追加時に種別選択ダイアログを出す

既存の`RenameDialog`（`RenameSheetButton_Click`が使う小型モーダル）と同型の新規ダイアログを
「＋」ボタン押下時に出し、シート名に加えて種別（制御回路／主回路）を選ばせる。
- 長所: 既存のモーダルダイアログ規約（GuiEcad由来のパターン踏襲）にそのまま乗る。ボタン数は
  増えない。
- 短所: シート追加のたびに1クリック増える（現状は「＋」1発で追加完了）。

### 案2: 追加ボタンを種別ごとに2つに分ける

「＋制御」「＋主回路」のように、ボタン自体を種別別に用意する。
- 長所: クリック数は現状と同じ（1発）。ダイアログ不要で実装が単純。
- 短所: 左パレットのボタン数が増える（現状「＋」「－」「名前変更」の3つ→4つ）。頻度の低い
  操作（主回路シートは主回路図面のみで使う）のためのボタンが常時表示される。

### 案3: 追加後にシート種別を後付けで切り替える

現状どおり常に制御回路シートとして追加し、追加後に（右クリックメニュー、または
`RenameSheetButton`同様の「種別変更」ボタン／プロパティパネル等で）主回路へ切り替える。
- 長所: 追加操作自体は現状のまま変えない。GX Works3等でも「後からシート属性を変更する」操作は
  一般的。
- 短所: 制御回路用に配置済みの要素（`Elements`/`Connectors`等）がある状態で主回路へ切り替えると
  データ上の不整合（制御回路専用データが主回路シートに残る）が起こりうるため、切替時の扱い
  （警告／要素があれば切替禁止等）を追加で設計する必要がある。

## 4. 侍の所感（決定ではなく判断材料）

- 案1・案2は「作成時に確定」のため案3のようなデータ不整合の懸念が無く、実装も小さく済む。
- 案1・案2の優劣はクリック数と画面上のボタン密度のトレードオフであり、断定はしない。
- 案3は既存の「後から変更できる」利便性はあるが、データ不整合という新しい設計課題を持ち込む
  ため、T-041のスコープ（手動配線基盤）を超える検討が必要になりやすい。

## 5. 次のステップ（提案）

上記3案を家老経由で殿へ提示し、裁定後に侍が実装する。実装規模はいずれの案も小さく
（`SheetNavigationViewModel.AddCommand`まわりの変更＋案1のみ新規ダイアログ1点）、
増分5本体（`FreeLine`/`ConnectionDot`ヒットテスト・記入・消去）とは独立して先行実装できる。
