## 新規指摘(severity低、経過観察相当)

### 1. 二重異常条件(ページが最小サイズの2倍未満)では反転追従が無言で無効化される

**該当**: `MainWindowViewModel.cs:1327-1329`(`ClampResizeTarget`のフォールバック2文)

`anchorMm+MinSize>maxMm`と`anchorMm-MinSize<0`が同時に真になる極端に狭いページ(理論上`maxMm<10mm`)では、ドラッグ方向に関わらず最終的な伸長方向が`defaultGrowsPositive`(ハンドル本来方向)に固定され、往復2周目の眼目である反転追従がこのケースに限り効かない。クラッシュ・境界外はみ出しは起きない(安全側フォールバックとして機能)。現実的にどこまで想定すべきケースかは不明、意図的な度外視か見落としかは家老・侍判断が必要と各エージェントが評価。

### 2. `currentMm >= anchorMm`の等号がハンドル種別に関わらず正方向を優先する

**該当**: `MainWindowViewModel.cs:1327`

ドラッグ位置がアンカーにちょうど一致する1点で、TopLeft/BottomLeftハンドル(自然な伸長方向が負)でも正方向側へ倒れる。最小サイズ床処理により元々不連続なジャンプが存在する箇所(アンカー近傍)での帰属先が1点だけ直感と逆になる程度で、連続的なマウス移動でこの点に正確に乗る頻度は低い。

### 3. `HitTestImage`が同一右クリックハンドラ内で最大2回呼ばれる

**該当**: `MainWindow.xaml.cs:873`(行範囲外ガード)と`:907`(画像分岐)

副作用のない純粋関数のため誤動作は起きないが、行範囲外×画像ヒットのケースで同じ探索を2回行う。ロジックが2箇所に存在する形になり、将来ヒットテスト条件を変更する際に片方だけ更新して不整合を生むリスク(保守性)。性能上のコストは軽微。

### 4. `ClampResizeTarget`の2つの独立if文は1つの`hasRoom`判定にまとめられる

**該当**: `MainWindowViewModel.cs:1327-1329`

数式的に全分岐と等価な単一三項式(`hasRoom = growsPositive ? anchorMm+MinSize<=maxMm : anchorMm-MinSize>=0; if (!hasRoom) growsPositive = defaultGrowsPositive;`)が存在し、そちらの方が「ドラッグ方向に伸ばす余地があるか」という意図をそのまま読み取れる。現状の「フラグを逐次上書き」する書き方は、この関数が既に2回連続でパッチされている(往復1→2周目)経緯を踏まえると、次の変更で意図せず順序を崩すリスクがある。

## 派生提案(構造的、家老への申し送り推奨)

### 左右クリックのヒットテスト優先順位ロジックが2つの独立メソッドに手書きで重複している

**該当**: `MainWindow.xaml.cs:797-832`(左クリック側)と`:861-917`(右クリック側)

要素→縦コネクタ→(WireBreak→FreeLine→ConnectionDot、左クリックのみ)→画像、という優先順位判定が左右クリックそれぞれに個別実装されており、単一の情報源(single source of truth)が無い。**今回の往復2周目修正1自体が、この非対称実装が原因で生じた不具合**(右クリック側だけ画像の優先順位判定に行範囲ガードがかかっていた)。`rowInRange &&`を要素・コネクタの各条件式へ個別追加する今回の直し方も、将来別の要素種別(GroupFrame等、T-069コメントに将来追加の言及あり)を右クリックに追加する際、コピペ元によっては「グリッド非依存なのに`rowInRange`を誤って付ける／グリッド依存なのに付け忘れる」という同型の横展開漏れを再び招きうる設計になっている(Altitude角度が独立指摘)。

**パターン台帳との照合**: 既存PR-01(選択可能状態の横展開漏れ)・PR-07(コピペ重複、3箇所到達で確定)のいずれとも完全一致はしない(現状2箇所の重複でPR-07の「3箇所到達」要件には未達、対象も選択状態ではなくヒットテスト優先順位ロジック)が、「単一の入口・共通ヘルパーが無いことに起因する非対称性」という根はPR-01系と近い。**断定は避け「パターンの疑いあり」として申し送る**(2件目が別タスクで出れば正式パターン化を検討)。
