# T-069 右クリックメニュー残り4系統：着手前設計整理

調査者: 隠密2　最終更新: 2026-07-11

家老采配（殿しばし席を外されるにつき家老裁量で沙汰）。右クリックメニュー残り4系統（コピペ系・要素操作系・
縦コネクタ系・GroupFrame系）の内部機能有無・依存関係・先行着手可否を整理する。
**実装は行っていない、着手前調査のみ**。

## 前提: GuiEcad実物の右クリックメニュー全体像

`GuiEcad.App/MainPage.ContextMenu.cs`（全文Read済）の`ShowDrawingContextMenu`が全系統の実体。
ヒットテスト優先順位は「要素→縦コネクタ→GroupFrame→行」（`HitTest`→`HitTestConnector`→
`HitTestFrame`→行番号判定の順）。**ecad2の既存ContextMenu（T-055増分3、行操作のみ）は行番号のみ
判定しており、要素・コネクタ・枠のヒットテスト分岐が無い**——今回の4系統実装では、この
ヒットテスト分岐そのものを新設する必要がある（4系統共通の基盤整備、依存関係とは別枠の作業）。

GuiEcadの4系統の内訳（実物照合）：

| 系統 | メニュー項目 | 出現条件 |
|---|---|---|
| コピペ系 | コピー(Ctrl+C)／貼り付け(Ctrl+V) | 要素/選択範囲上でコピー、クリップボード保持時に貼り付け |
| 要素操作系 | 削除(Del)／コメント編集(F2)／機器名変更(Enter) | 要素上で右クリック |
| 縦コネクタ系 | 縦コネクタ削除 | 縦コネクタ上で右クリック |
| GroupFrame系 | ラベル編集／削除／線種(実線・破線・点線サブメニュー) | 枠上で右クリック |

## 各系統のecad2側 内部機能有無（DoD1: T-063同様の「器のみ」パターン確認）

### 1. コピー・ペースト系

`Clipboard`/`CopySelection`/`PasteSelection`関連をApp層全体でgrepし**0件**。todo.md記載どおり、
クリップボード機能自体が完全に未実装（棚卸しC区分・後日判断）。

**結論：器のみパターンではない。クリップボード機能（キー操作含め）そのものが丸ごと無いため、
右クリックメニュー単独では対応不可能——クリップボード機能自体の実装が前提条件。**

### 2. 要素操作系（3項目、内部機能の有無が項目ごとに異なる）

| 項目 | ecad2の現状 |
|---|---|
| 削除 | `MainWindowViewModel.DeleteSelectedElement()`既存（Deleteキー、T-041増分1、
  `MainWindow.xaml.cs:898`）。**器のみパターン、露出のみで対応可能** |
| コメント編集 | `RungComment`モデル自体はCore層に存在（`Sheet.cs:18`）し行シフト処理
  （`RowOps.cs:26-27,60-61,78-79`）も対応済みだが、**編集UI(App層)がecad2に一切存在しない**
  （GuiEcadの`ShowCommentEditor`/F2キー相当がApp層全体でgrep 0件）。**器のみパターンではない。
  コメント編集UIそのものの新規実装が必要（右クリックメニューだけの問題ではない）** |
| 機器名変更 | `DeviceNameBox`（右パネル常設プロパティ領域のテキストボックス、`MainWindow.xaml.cs:147-243`）
  が既存。ただしGuiEcadの「Enterキーでポップアップエディタを開く」方式とは異なり、ecad2は
  「選択時に常時表示されるプロパティパネルへ直接入力する」設計。右クリックメニューから選んだ場合、
  `DeviceNameBox`へフォーカスを移す程度の実装で足りると見込むが、UI/UX上の見せ方は要確認 |

**結論：3項目が均質ではない。「削除」のみ即座に器のみパターンで対応可能、「コメント編集」は
機能自体が新規実装（規模の見立てが必要）、「機器名変更」は既存機能はあるがフォーカス誘導の
実装確認が要る。3項目まとめて1系統として着手すると、コメント編集の新規実装が律速になる。**

### 3. 縦コネクタ系（1項目のみ）

`MainWindowViewModel.DeleteSelectedConnector()`既存（Deleteキー、既存の削除統合ロジックに含まれる、
`MainWindow.xaml.cs:898`）。

**結論：完全に器のみパターン（T-063型）。依存なし。**

### 4. GroupFrame系（3項目：ラベル編集・削除・線種変更）

T-067（GroupFrame作成・編集UI皆無、既起票）の確認内容と同一の結論に帰着：`new GroupFrame`・
`Frames.Add`ともsrc全体0件——**GroupFrameインスタンスがアプリ上で一切生成されない**ため、
編集・削除・線種変更いずれの対象も存在しない。Core層モデル（`GroupFrame.BorderStyle: LineStyle?`、
`Element.cs:150`）は用意されているが、App層に対応する編集/削除メソッドは存在しない。

**結論：器のみパターンではない。T-067（GroupFrame作成・編集UI皆無）に強く依存——枠を作る手段
自体が無いため、右クリックで編集する対象が存在しない。T-067が完了してからでないと着手不可。**

## DoD(2): コピペ系のクリップボード依存整理

前節のとおり、クリップボード機能（キーボードショートカット含め）は完全に未実装で、todo.mdの

のC区分）。**コピペ系の右クリックメニューは、クリップボード機能本体の実装なしには成立しない
（メニュー項目だけ先に作っても呼び出す先が無い）。着手順序としてはクリップボード機能本体
（後日判断枠）の裁定が先。**

## DoD(3): 依存なし系統の先行着手可否

| 系統 | 依存 | 先行着手可否 |
|---|---|---|
| コピペ系 | クリップボード機能本体（未実装・後日判断枠） | **不可** |
| 要素操作系「削除」 | なし（既存機能） | **可** |
| 要素操作系「コメント編集」 | コメント編集UI本体（未実装、規模別途調査要） | **不可**（新規実装が前提） |
| 要素操作系「機器名変更」 | なし（既存機能、フォーカス誘導のみ） | **可**（UI/UX確認要） |
| 縦コネクタ系 | なし（既存機能） | **可・最有力候補** |
| GroupFrame系 | T-067（GroupFrame作成・編集UI皆無） | **不可**（T-067完了が前提） |

**提案**：依存なしで即座に着手できるのは「縦コネクタ系（単独1項目）」と「要素操作系のうち削除・
機器名変更」——いずれもT-063と同型の「既存機能への露出」で完結する。コメント編集・GroupFrame系・
コピペ系の3系統はそれぞれ別のタスク（新規UI実装・T-067完了・クリップボード裁定）が前提条件のため、
**T-069を「先行着手できる部分（縦コネクタ削除＋要素の削除/機器名変更＋右クリックヒットテスト基盤）」
と「後続待ちの部分（コメント編集・GroupFrame・コピペ）」に分割する**のが規模・依存の実態に合う。
分割の是非・呼称は家老采配または殿確認事項として申し送る。
