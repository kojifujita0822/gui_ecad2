## 2. 「使い方」メニュー項目新設方式

既存の`AboutMenuItem_Click`と同型パターンで実装可能：

```xml
<MenuItem Header="使い方(_G)" Click="UsageMenuItem_Click"/>
```
```csharp
private Views.UsageWindow? _usageWindow;
private void UsageMenuItem_Click(object sender, RoutedEventArgs e)
{
    if (_usageWindow is null)
    {
        _usageWindow = new Views.UsageWindow { Owner = this };
        _usageWindow.Closed += (_, _) => _usageWindow = null;
        _usageWindow.Show();
    }
    else
    {
        _usageWindow.Activate();
    }
}
```

多重起動防止（既存インスタンスがあれば`Activate()`のみ）は、非モーダル別ウィンドウのGX Works3的な使い勝手（メインウィンドウと並べて常時参照）から見て必要な配慮と考える。ショートカットキー割当（F1等）は殿確認事項として別途提示が必要（本プランでは技術調査の範囲外、着手時に確認）。

## 3. コンテンツ表示技術方式：3案比較（UI/UX分岐・殿確認要）

| 観点 | 案A: Markdown直接表示（サードパーティ） | 案B: FlowDocument変換（自作） | 案C: WebView2埋め込み |
|---|---|---|---|
| 新規依存 | NuGetパッケージ1件追加（例: MdXaml等） | **なし**（CLAUDE.md原則に最も合致） | NuGetパッケージ+WebView2 Runtime（Windows環境依存） |
| 実装コスト | 低（ライブラリがMarkdown→表示変換を内包） | 中〜高（Markdown→FlowDocument変換を自作、対応構文の範囲次第） | 低〜中（Markdown→HTML変換に別ライブラリが要る場合あり、WebView2初期化・エラー処理が必要） |
| ダークモード連動 | ライブラリのテーマ切替API次第（要検証） | 容易（既存`DynamicResource`機構をそのまま利用可能） | CSS側で作り込みが必要 |
| 見た目の表現力 | Markdown標準的な見た目 | WPFネイティブ、既存UIとの統一感が高い | 最も高い（ブラウザレンダリング、表・画像等に強い） |
| 保守性・リスク | 依存ライブラリの継続メンテ状況・.NET 10対応状況に依存（未検証） | 自社管理、依存リスクなし。変換ロジックの実装漏れ（未対応構文）はリスク | 依存が最も重い（SDK+ランタイム2層）。ランタイム未導入環境での動作不良リスク |

**隠密所見**：CLAUDE.mdの「不要な外部依存を追加しない」「シンプルさこそ正義」という原則に照らせば案Bが最も整合するが、実装コスト（Markdown変換ロジックの自作）とのトレードオフがある。案Aは実装コストが低く現実的な折衷案になりうるが、ライブラリの.NET 10/WPF最新版との相性は未検証（着手時に候補ライブラリの動作確認が必要）。**技術選定はUI/UX上のトレードオフを伴う判断のため、断定はせず3案を殿へ提示することを推奨する。**

## 4. ナビゲーションUI設計案（UI/UX分岐・殿確認要）

殿確定事項「GX Works3のヘルプウィンドウに近い形」を踏まえた3案：

**案1（推奨）: 左目次（TreeViewまたはListBox）+ 右コンテンツ表示**
11領域を左側に一覧表示、クリックで右側のコンテンツ領域が切り替わる。GX Works3を含む一般的なヘルプツール（Windows CHM viewer等）の標準構成に最も近く、殿確定事項に直接合致する。

**案2: 上部タブ切り替え（11タブ）**
タブストリップに11領域を並べる。11個は画面幅を圧迫しやすく、折りたたみ/スクロール可能なタブUIが必要になる。並列情報の切替には向くが、階層的なヘルプ目次としてはGX Works3的な体験からやや外れる。

**案3: 単一ページ+目次アンカーリンク**
11領域を1つの長いコンテンツへ連結し、冒頭の目次リンクからスクロールジャンプする。実装はシンプル（切替UIのステート管理不要）だが、長大化・スクロール位置ジャンプの実装（技術方式によっては手間）というデメリットがある。

**隠密所見**：案1が殿確定事項「GX Works3のヘルプウィンドウに近い形」に最も忠実であり、家老への一次推奨としてよいと考えるが、最終選択は殿確認を推奨する。

- クラス名・メソッド名・ファイル:行番号・コード引用（開発者向け出典、ユーザーには不要）

### 見積り（推測、断定不可）

行数だけでの単純な比例見積りは実態と乖離しうる（質的な書き換え作業のため）が、目安として、開発者向け150-220行のファイルは、実装詳細を削り操作説明に絞ることで**ユーザー向け50-80行程度への圧縮**になると推測する。11ファイル合計で、平易版は概算700-900行程度と見込む（**推測であり、実際に1-2ファイルを試作してから精度を上げることを推奨**）。

### 誰がどう作業するか

`docs/spec`自体の作成はT-075として隠密が担当している。ユーザー向け平易版への変換も文書コンテンツの編集作業であり、実装（UI表示機構）とは性質が異なる。**作業分担は家老の采配判断事項のため本プランでは決め打ちしない**が、選択肢として：(a) 隠密が全11件を文書編集タスクとして担当、(b) 侍がUI実装と並行してコンテンツ変換も担当、(c) 両者で分担、のいずれかを家老に判断いただきたい。
