### (1) テスト設計どおりに実装されたか

観点1(ClickCount値×ヒット領域×シート種別の同値分割・境界値)は忠実に実装されている。
`RungCommentDoubleClickTests`(`ShouldOpenRungCommentEditor`純粋関数、Theory5件)+
`RungCommentHitTestTests`(`HitTestRungCommentRow`直接、Fact4件)の2層分割は、隠密の設計書が

重複なくカバーする合理的な実装判断と評価する。

観点2(状態遷移)・観点3(F2対称性・ドラッグ分岐優先順位)・観点4(既存4分岐回帰)は、View層

### (2) RED証明の整合(家老が特に吟味を求めた点)

**家老の懸念は正当と判断する**。`ShouldOpenRungCommentEditor`はDown/Upどちらのイベントに
配線されていても同じ入出力を返す純粋関数であり、そのテストは「判定条件(`clickCount==2`)」の
正しさのみを保証し、「その判定がDown側イベントに配線されていること」自体は一切保証しない。
仮に将来誰かがこの呼び出しをUp側へ書き戻す退行が起きても、`RungCommentDoubleClickTests`・
`RungCommentHitTestTests`はいずれも変更を検知できずグリーンのまま通過する。

**(ア)(イ)への所見**:
- 単体テストで「配線位置」自体を直接検証する現実的な策は、`MouseButtonEventArgs.ClickCount`の
  setアクセサがinternalで直接構築できないというWPFのAPI制約(家老裁定3で確認済み)により乏しい。
  リフレクションでの強制注入やソースコード文字列検査は、脆さ・過剰実装の割に得るものが少なく
  推奨しない
- 現実的な落としどころは、(i)実機確認を最終防波堤として明示的にDoD化する(家老裁定2の方向性を
  支持)、(ii)現状既に厚く書かれているコード内コメント(Down側・Up側双方に経緯説明あり)を維持する、
  の2点セットで運用上カバーするのが妥当。**(ア)「現状で許容」の判断自体は追認するが、「実機確認で
  補う」を単なる一般論に留めず、次回リリース前の定例チェック項目として`docs/`のどこかに明文化する
  ところまでやって初めて『補った』と言える**、という条件付きの追認としたい

## 要注意事項(ブロッキングではないが実機確認DoDへの反映を推奨)

### 要注意1: Up側の対応ガード欠落(独立2角度=line-by-lineスキャン・言語落とし穴、一致検出)

- **file/line**: `src/Ecad2.App/MainWindow.xaml.cs:404-409`(Down側新規早期return)、
  `:613-618`(Up側、旧ガード削除・コメントのみ残存)、`:656-657`(Up側フォールスルー先)
- **事実**: 往復1周目まではUp側に`if (e.ClickCount==2 && ...) { OpenRungCommentEditor(...); return; }`
  という早期returnがあり、コメントエディタを開いた同じクリックのUp処理(656-657行目
  `_viewModel.SelectedCell = ...; TryPlaceActiveTool();`)への到達を防いでいた。本diffはこの
  early returnをDown側へ移設し、Up側は「移設した」という説明コメントのみが残り、**代替の明示
  ガードが無い**。F2キー経路には`Window_PreviewKeyDown`冒頭に`_rungCommentEditingRow`による
  明示ガードが存在する(`MainWindowViewModel.cs:70`コメント「経路はWindow_PreviewKeyDown冒頭の
  ガードが対で塞ぐ」)のと対称的に、マウスUp側には同種の明示ガードがない
- **現状これを防いでいる仕組み**: `OpenRungCommentEditor`内の`_viewModel.IsRungCommentEditorVisible
  = true;`(同ファイル1722行目)が`SetProperty`経由で`OnPropertyChanged(nameof(IsMainContentEnabled))`
  を同期発火し(`MainWindowViewModel.cs:71-79`)、WPFバインディングにより`MainContentArea.IsEnabled`
  へ即時反映される(非同期ディスパッチは介在しない、標準的なINotifyPropertyChangedバインディング
  機構)。理論上はDownハンドラの処理が完了する前にIsEnabled=falseが反映されているため、後続の
  物理Upイベントは無効化された`LadderCanvasHost`をヒットテストの対象外として素通りするはず
  (=(b)調査で確認済みの「無効化要素はヒットテスト対象外」という挙動と整合)
- **判定**: 実害の発生確率は理論上低い(タイミング上、通常は防がれるはず)が、**これはコードによる
  保証ではなく暗黙の副次効果**であり、隠密のテスト設計書「不明点」・侍のコミットメッセージ・家老
  裁定2のいずれもが「実機確認で確認する」としてこの経路を未検証のまま残していることと符合する。
  新規の見落としというより、**既に認識されていた残課題の具体的な発生機序が今回のコードレビューで
  明確になった**という位置づけ。忍者の実機確認DoDに「エディタ表示中に選択中の配線プリミティブ
  (縦コネクタ等)が巻き添えでクリアされないか」「配置ツールarmed中にダブルクリックした場合、
  コメント編集直後に意図しない配置・ステータスメッセージ上書きが起きないか」を具体的な確認項目
  として追加することを推奨する

### 要注意2: 短絡評価の消失によるヒットテスト実行頻度の増加(4角度が独立して一致検出)

- **file/line**: `src/Ecad2.App/MainWindow.xaml.cs:405`
- **事実**: `ShouldOpenRungCommentEditor(e.ClickCount, LadderCanvasHost.HitTestRungCommentRow(position, rcSheet))`
  はC#の引数評価順により、`HitTestRungCommentRow`が`ShouldOpenRungCommentEditor`本体の判定より
  **先に**無条件実行される。旧Up側実装は`e.ClickCount == 2 && ... is int rcRow`という`&&`の短絡
  評価で、ヒットテストはダブルクリックの2回目でのみ実行されていた。本diffの純粋関数抽出により、
  ClickCount=1の通常クリック(セル選択・要素配置・ドラッグ開始等の全操作)でも毎回ヒットテストが
  走るようになった
- **増幅要因**: `HitTestRungCommentRow`内部で`DiagramRenderer.TotalRows(sheet)`
  (`Ecad2.Core/Rendering/DiagramRenderer.cs`)を呼んでおり、これは`sheet.Elements`全件を舐める
  O(n)の線形スキャン(キャッシュなし)。大規模な制御盤設計でElements数が多いシートでは、右母線
  右側の単発クリックのたびに全要素走査のコストを払うことになる
- **判定**: 現状は実害が顕在化するほどの要素数ではない可能性が高く、機能バグではないため
  ブロッキングにはしない。ただし4つの独立角度が一致して検出した確度の高い指摘であり、修正は
  「`e.ClickCount == 2`を先に評価してからヒットテストを呼ぶ」という1行の変更で解消できる
  (「ツールモードを問わず優先判定」という仕様自体は維持したまま)。次の増分または軽微修正の
  タイミングでの解消を推奨する
