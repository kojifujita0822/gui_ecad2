# T-098 実機検証（忍者）

**対象コミット**: `0a12585`（シート追加時のPageNumber採番方式見直し、`DetermineNextPageNumber`新設）
**検証日**: 2026-07-21
**手法**: `ecad2-ui-automation`スキル。PageNumberはUI上に直接表示されない内部値のため、`.gcad`ファイル（JSON、`pageNumber`キー）を保存しPowerShellの`ConvertFrom-Json`で実測する方式を採用。
**参照**: 隠密静的レビュー`docs/ecad2-t098-review-onmitsu.md`（指摘なし）

## 前提確認：PageNumberの画面表示手段

コード確認の結果、`Sheet.PageNumber`はUIのどこにも直接表示されない（シートタブの表示名は`Sheet.Name`、PDF出力のページ番号`PdfPage.PageNumber`も物理ページの単純連番でモデルのPageNumberとは無関係）。家老指定の検証観点「PageNumber表示とSheets.Count（表示順序）の対応」を確認するため、`.gcad`ファイル保存→JSON実測という手段を用いた。

## DoD別結果

### (1) 欠番状態から新規シート追加→既存最大PageNumberの次番号が正しく付くこと — OK

手順：
1. シート1・2・3を作成（PageNumber=1,2,3、`.gcad`保存でJSON実測し基準値を確認）
2. 中間のシート2を削除（欠番が生じる）
3. 新規シート追加
4. `.gcad`保存しJSON実測

**実測結果**：残存シート1(PN=1)・シート3(PN=3)に加え、新規追加シートのPageNumberは**4**（`DetermineNextPageNumber`＝既存最大3+1）。旧実装（`Sheets.Count+1`＝2+1=3）であれば既存のシート3(PN=3)と重複していたはずのバグが解消されていることを実測で確認した。

### (2) 欠番なし状態での通常の連続追加→従来どおり連番になること（回帰なし） — OK

新規ドキュメントでシートを3回連続追加し`.gcad`保存・JSON実測、シート1=1、シート2=2、シート3=3と正しく連番。回帰なし。

### (3) T-084の欠番警告メッセージが引き続き適切なタイミングで出る／出ないこと — OK

- **中間シート削除（真に新規追加分の削除、欠番が生じるケース）**：シート2（PageNumber=2、他にPageNumber=3のシートが残る）を削除→ステータスバーに「シート削除によりページ番号に欠番が生じました。」と正しく表示（`ecad2-t098-middle-deleted.png`）。
- **末尾シート削除（欠番が生じないケース）**：シート3（末尾、他に自分より大きいPageNumberのシートが存在しない）を削除→警告メッセージなし（クリーンな新規ドキュメントで再現し確認、`ecad2-t098-fresh-last-deleted.png`）。

両ケースとも`docs/todo.md`の問題意識（「末尾シートを消しただけなのに欠番警告が出る」という直感に反する挙動の解消）どおりの結果。

## 検証観点（家老指定）の結論：表示順序とPageNumberの対応にズレなし

削除・追加を繰り返した後もJSON実測で確認した限り、シートタブの表示順序（`Sheets`配列順）とPageNumber数値は常に整合していた。DoD1のシナリオ（中間削除→追加）でも、新シートはタブ上は末尾に追加され、PageNumberも最大値+1（表示順序で最後＝数値上も最大）となり、直感どおりの対応が保たれている。

## 範囲外の気づき（実機確認手法上の教訓、T-098のバグではない）

検証中、忍者の操作起因で以下2点に遭遇した。いずれも`ecad2-ui-automation`スキルの既存トラブルシュートに記載の罠と同型または関連するため、次回以降の参考として記録する。

1. **「上書き保存」ボタンの連続Invokeでモーダルダイアログが多重化**：初回保存時、`Send-Ecad2Keys`でのファイル名入力・Enter確定後にダイアログが閉じたことを確認せず次の操作（タブ切替→再度保存ボタン）を重ねてしまい、「名前を付けて保存」ダイアログが3枚重なった。既知の罠（本スキル6節）と同型。今回はUIAツリーが著しく不安定化し（要素数が数個しか取得できない等）、EscキーやWM_CLOSEメッセージも効かず、最終的にプロセス再起動で仕切り直した。
2. **保存先フォルダが直前の別ダイアログのカレントフォルダを引き継ぎ、アクセス拒否が発生**：T-105検証で画像挿入時に使った`C:\Windows\Web\Screen`が「名前を付けて保存」ダイアログの既定フォルダとして残っており、そのまま保存しようとして「アクセス許可がありません」という確認ダイアログ（はい/いいえ）が追加で挟まった。**対策**：保存操作では`%n`でファイル名欄へフォーカス後、ファイル名のみでなく`C:\ECAD2\sample\<name>`のようにフルパスを明示的に入力すると、フォルダの引き継ぎ問題を回避できる（今回はこの方式で確実に保存できた）。
3. **「新規作成」ボタンは未保存の変更がある場合、確認ダイアログ（はい/いいえ/キャンセル）を挟む**：これ自体は正しい既定動作だが、`Invoke-Ecad2Button`実行後にダイアログの有無を確認せず直後にステータス確認を行うと、ダイアログが応答待ちのまま残っていることに気づかず「新規作成が反映されていない」と誤診断しかけた（実際は`StatusMessage`クリア処理自体は正しく実装されていた）。ダイアログを開く可能性のある操作は、本スキル6節の教訓どおり毎回`EnumWindows`で確認する運用を徹底する必要性を再認識した。

いずれも侍実装のバグではなく忍者の検証手順起因のため`docs/todo.md`化は不要と判断する。ecad2-ui-automationスキルへの追記候補として家老の判断に委ねる。
