# T-104増分1(基本機能タブPoC・ダミー表示)実機確認記録

忍者作成、2026-07-20。家老委譲(DoD4点)の実機確認結果。
対象: `MainWindow.xaml`配置ツールバー領域、`基本機能`/`配置ツール`の2タブ構成(ContentId="MainToolBar"新設)。

## DoD観点別結果

### (1) タブが2枚現れクリックで正しく切り替わるか → OK

UIA `SelectionItemPattern.Select()`でTabItem[0](基本機能)⇔TabItem[1](配置ツール)を相互に選択し、
その都度`IsSelected`が正しく入れ替わることを確認。選択中タブはBounds高さ13px・Y=222、非選択は
高さ10px・Y=223(選択タブが1px上・3px高く描画される一般的なタブスタイル)という一貫した視覚差も
確認できた。

### (2) CanFloat="False"でフロート化操作が実際に無効化されるか → **NG(重大)**

配置ツールタブのヘッダー「メニュー」ボタン(HelpText="メニュー")からコンテキストメニューを開くと、
**「フローティング」メニュー項目が`IsEnabled=True`のまま**(隠密の懸念どおり、グレーアウトも
無反応化もされていない)。実際にクリックすると**本当にフロート化が成立してしまった**。

**フロート化後の状態**:
- 新規ウィンドウ(タイトル無し、`GetWindowRect`で579x101程度)が生成され、中身は「配置ツール」
  パネルのみ(タイトルバーに「配置ツール」と表示、配置ツールのボタン群一式)
- メインウィンドウ側は「基本機能」パネルのみが残り、**2タブが同一ウィンドウ内で共存する形には
  ならず**、フロート化直後は「基本機能」パネルのタブヘッダー行自体が非表示になった(1タブのみに
  なりタブUIが消えたとみられる)
- 殿より「フロート操作でタブは表示されるか？」とのご質問があったが、上記のとおり**2タブが同じ
  フロートウィンドウに一緒に表示されることはない**(選択中だった片方のパネルのみが単独で
  切り離される、AvalonDockの一般的な挙動)

**結論**: `CanFloat="False"`設定の意図(恐らく「配置ツール／基本機能はフロートさせない」)が
実装上機能していない。隠密の静的レビューでXAML側の設定漏れ・伝播不備を確認されたい。

### (3) ドッキング復帰(Ctrl+Alt+R)で2タブとも正常復元 → OK

上記(2)のフロート化状態から`Ctrl+Alt+R`を送信したところ、ステータスバーに
「パネルレイアウトを既定に戻しました」と表示され、フロートウィンドウは消滅、メインウィンドウの
`TabItem`数が4(基本機能・配置ツール・シート・出力)に復元された。Bounds も元の2タブ構成
(1976,222,40,13 / 2015,223,40,10)と一致。

### (4) キーボードナビゲーション(Tab/Shift+Tab)でタブ間移動が破綻しないこと → **NG**

`AutomationElement.SetFocus()`でTabItemへフォーカスを設定し、`Send-Ecad2Keys`でTab/Shift+Tabを
送信し`AutomationElement.FocusedElement`で移動先を確認。

- **TabItem[0](基本機能)からTab(前進)** → フォーカスが`ControlType.Window`(メインウィンドウ自体)
  へ抜けてしまう(2回試行し再現)
- **TabItem[1](配置ツール)からTab(前進)** → 同様にWindowへ抜ける(再現)
- **TabItem[1](配置ツール)からShift+Tab(後退)** → `SheetNavList`(シート一覧)へ正しく移動

**結論**: Tabキー(前進方向)でのタブ間ナビゲーションが機能しておらず、2枚のタブ間を相互に
Tabキーで移動することができない(どちらからも「次」がWindow自体になり行き止まりになる)。
Shift+Tab(後退)側は正常に見えるため、フォーカス順序(TabIndex等)の設定が前進方向のみ
欠落している可能性がある。

## DoD(4)追加実測(2026-07-20、侍の診断ログ計装版・GotFocus/PreviewKeyDown計装)

侍が`GotFocus`(FrameworkElement全体)・Tab押下時のフォーカス元を`%TEMP%\ecad2-diag.log`へ記録
する診断ログを計装したビルドで再実測。

### 実測結果

- **「基本機能」タブへフォーカス設定→Tab送信(1回目)**: `GotFocus: type=LayoutAnchorableTabItem`
  を記録。UIAの`FocusedElement`は依然`ControlType.Window`を報告(WPF内部イベントとUIA報告が
  食い違う)。**視覚的なフォーカス破線枠は「基本機能」タブに表示されたまま変化なし**。
- **さらにTab連打(計7回)**: `focusedBefore.type`が`LayoutAnchorSideControl`⇔`ItemsControl`の
  間を往復するのみを実測。「基本機能」「配置ツール」いずれのタブヘッダーへも
  `GotFocus`イベントは一度も記録されなかった。
- **「配置ツール」タブへフォーカス設定→Shift+Tab送信**: `GotFocus: type=ListBox
  name=SheetNavList`を記録(前回のUIA実測と一致、後退方向は正常)。

### 結論

Tab(前進)方向のフォーカス順序は、AvalonDockの内部コントロール階層
(`LayoutAnchorSideControl`⇔`ItemsControl`、恐らくAutoHideエリア管理用の非表示要素)を
巡回するのみで、「基本機能」⇔「配置ツール」という**ユーザーに意味のあるタブ間移動には
一度も到達しない**。加えて、視覚的なフォーカスインジケータ(破線枠)がWPF内部の実際のフォーカス
遷移に追従せず「基本機能」タブに固定表示され続ける点も、ユーザー体験上の食い違いとして
記録しておく。UIAの`FocusedElement`(常にWindow報告)は本問題の実測には使えず、侍の
GotFocus計装が必須だった。DoD(4)は**確定的にNG**、原因はTabIndex/KeyboardNavigation設定の
不備、またはAvalonDockのAutoHideエリア関連コントロールへフォーカスが迷い込む実装上の問題と
推察される(原因特定は侍・隠密に委ねる)。

## 隠密仮説の実測裏付け・影響範囲確認(2026-07-20)

隠密より「DockingManager既定ControlTemplateが常設する4方向AutoHideサイド領域
(`LayoutAnchorSideControl`)にFocusable制御が無く、これがTab巡回の迷子先になっている疑い」との
仮説が提示され、UIA実測での裏付けと影響範囲確認を依頼された。

### 仮説裏付け: 強く支持

`ClassName="LayoutAnchorSideControl"`ではUIAツリーに1件もヒットせず(**UIA非対応要素、
AutomationPeer自体を持たないと見られる**)。一方`ClassName="ItemsControl"`で検索したところ
**16件すべてが`IsKeyboardFocusable=True`**、かつBoundsが軒並み幅6px/高さ6pxという極端に細い
矩形(画面四辺の境界領域、AutoHideサイド領域のヒットテスト用と見られる)であることを確認。
これらがTab巡回時にフォーカスを吸い込む「見えない罠」になっているという隠密仮説と完全に整合。

### 影響範囲確認: **T-104固有、既存パネルには影響なし**

「シート」タブ・「出力」タブから同様のTab連打を実施したところ、いずれも
`TabItem`→`LayoutDocumentTabItem`(キャンバスのドキュメントタブ)→
`Button name=DocumentCloseButton`という**正常な経路**を辿り、AutoHideサイド領域への迷子は
一度も発生しなかった。「基本機能」「配置ツール」(いずれも配置ツールバー領域の
`PlacementToolBarDockingManager`配下、T-104新設)からのTab送信でのみ迷子現象が再現し、
既存の「シート」「出力」(メイン`DockingManager`配下)では再現しなかった。

**推察**: `PlacementToolBarDockingManager`はT-099(c)/T-100調査でも「PrintWindow撮影・UIA探索
双方が正しく捕捉できないことがある」特殊な構造(`AnchorablePaneControlStyle`のローカル値
カスタム設定)と既に指摘されており(本スキル6節既存記載)、メインの`DockingManager`とは異なる
テンプレート構成・フォーカス順序を持つ可能性が高い。右パネル(機器表・プロパティ)はそもそも
タブ化されておらず単独`AnchorablePane`のため、今回のTab巡回試験の対象外(通常操作でも
タブ切替UIが存在しない)。

### 忍者からの申し送り

対策(`Focusable="False"`化)の適用範囲は、**メイン`DockingManager`側ではなく
`PlacementToolBarDockingManager`側のAutoHideサイド領域に限定してよい**可能性が高い実測結果。
ただし「なぜ配置ツールバー側だけ迷子が起きるか」の根本原因(テンプレート差異の詳細)は
UIA実測の範囲外につき、隠密・侍の静的解析に委ねる。

## DoD(4)対策実装版の実機確認(2026-07-20、侍対策=LayoutAnchorSideControlのFocusable無効化)

侍が`MainWindow.xaml`1071-1072行目へ
`<Style TargetType="{x:Type avalonDock:LayoutAnchorSideControl}"><Setter Property="Focusable"

### 実測結果: **部分改善、ただしタブヘッダー間の直接移動には未到達**

- UIA `ClassName="ItemsControl"`で検索した16件の境界要素は、対策後も**全て
  `IsKeyboardFocusable=True`のまま変化なし**(`LayoutAnchorSideControl`へのFocusable設定が、
  UIA上`ItemsControl`として現れるこれら16要素には反映されていない可能性)。
- 一方、`AutomationElement.Current.HasKeyboardFocus`でTab連打の実際の遷移先を追跡したところ、
  **経路が変化した**——「基本機能」タブからTab連打すると、今回は「シート」パネルの
  `＋`→`名前変更`→`設定`ボタンを順に巡回し、8回目付近で一瞬追跡不能(タイミング起因の可能性)、
  9回目で`＋`に戻るという**円環**を確認。以前の「Windowへ即座に抜ける」症状からは改善したが、
  **「基本機能」⇔「配置ツール」というタブヘッダー間の直接移動には一度も到達しなかった**。
- 「配置ツール」タブからのShift+Tab(後退): 変わらず`SheetNavList`へ正常移動(対策前と同一、
  回帰なし)。
- 「シート」タブからのTab(他パネルへの影響確認): ボタン要素(Bounds=2012,271,15,15、位置的に

### 結論

対策は症状を変化させた(即座にWindowへ抜ける現象は解消)が、**DoD(4)の本来の目標(基本機能⇔
配置ツールタブ間の直接Tab移動)はまだ未達成**。新たな迷子先は「シート」パネルのボタン群
(`＋`/`名前変更`/`設定`)の円環になっており、`LayoutAnchorSideControl`以外にも何らかの

適用範囲自体は安全と見てよい。原因の追加調査・対策は侍・隠密に委ねる。
