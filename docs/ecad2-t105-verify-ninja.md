## DoD別結果

### (1) GroupFrame選択中、無修飾矢印キー（上下左右）で移動できること — OK

配置直後は自動的に選択状態になる仕様（`ConfirmFrameDraft`実装どおり）。無修飾の上下左右キーそれぞれ1回ずつ送信し、キャンバス上でGroupFrameが1セル分ずつ正しく移動することを確認（`ecad2-t105-move-right.png`〜`ecad2-t105-move-up.png`、4方向とも往復させ元位置へ復帰することも確認）。

### (2) 既存要素（Element/Image等）の矢印キー移動に回帰がないこと — OK

**重要な前提確認**：コード確認の結果、既存要素の矢印キー割当はGroupFrameと同一ではなく、要素種別で異なる：
- `Element`（接点等）：**Ctrl+矢印キー**（`MoveSelectedElement`、T-088）。無修飾矢印キーは`SelectedCell`のカーソル移動（フォールバック）に使われる。
- `Image`：無修飾矢印キー（`MoveSelectedImage`、T-064、GroupFrameと同一キー）。

この前提を踏まえて両方を確認：
- M3（a接点）を選択し**Ctrl+右矢印キー**を送信 → 要素シンボル自体が1セル右へ移動、プロパティパネルにも正しく「種別: a接点、デバイス名: M3」を表示（`ecad2-t105-elem-ctrlmove2.png`）。回帰なし。
  - 補足：無修飾矢印キーのみを送信した際は`SelectedCell`のカーソルだけが移動し要素自体は動かなかったが、これは前述のキー割当仕様どおりであり不具合ではない（範囲外の気づき、後述）。
- 画像を選択し無修飾矢印キー（GroupFrameと同一キー）を送信 → 左方向で画像が正しく移動（`ecad2-t105-image-moveleft.png`）。右方向は画像サイズが大きく境界に既にクランプされていたため無反応だった（`MoveSelectedImage`の`Math.Clamp`仕様どおり、後述DoD3と同型の正常挙動）。回帰なし。

### (3) グリッド境界（端）で移動がクランプされること（4方向とも） — OK

GroupFrame（4行×4列）を各方向へ十分な回数（`{RIGHT 40}`/`{DOWN 30}`等）矢印キー移動させ、境界到達後にさらに同方向キーを送っても位置が変化しないことを4方向個別に確認：
- 左：初期配置時点で既に列0（左端）、左矢印キーで無反応（`ecad2-t105-frame-boundary-left.png`）
- 上：同じく初期配置時点で既に行1（上端相当）、上矢印キーで無反応（`ecad2-t105-frame-boundary-up.png`）
- 右：`{RIGHT 40}`後、追加の右矢印キー1回で位置変化なし（`ecad2-t105-frame-rightedge.png`→`rightedge2.png`）
- 下：`{DOWN 30}`+追加`{DOWN 20}`後も位置変化なし（`ecad2-t105-frame-bottomedge.png`→`bottomedge2.png`）

4方向とも全否定方式のクランプが正しく機能、回帰なし。

### (4) Undo/Redoで正しく巻き戻る・やり直せること（移動操作後） — OK

右下端へクランプ後、上矢印キーを2回送信して2行分移動 → `Ctrl+Z`（Undo）を1回送信し1行分だけ正しく巻き戻ることを確認（`ecad2-t105-frame-undo1.png`）→ `Ctrl+Y`（Redo）を1回送信し正しくやり直されることを確認（`ecad2-t105-frame-redo1.png`）。

**重点確認（隠密指摘(c)の実機裏取り）**：Undo実行後、プロパティパネルが「要素を選択してください」に戻り、GroupFrameの表示も選択ハイライト（オレンジ実線相当）から通常色（黒点線）に変化——`SelectedFrame`が正しく`null`クリアされ、幽霊参照が残っていないことを実機で確認。Redo後も同様に選択状態はクリアされたまま（`ApplyUndoRedoSnapshot`内`SelectedCell`再代入経由の設計どおり、異常なし）。

### (5) GroupFrame移動中、他要素との重複が許容されること（エラーにならないこと） — OK

GroupFrameを画像・M3要素と重なる位置まで矢印キーで移動させ、出力パネル（DRC等）にエラー・警告が一切出力されないことを確認（`ecad2-t105-frame-overlap.png`）。コード上「GroupFrameは他要素との重複を許容するため占有判定は不要」との設計どおり、実機でも例外・意図しない拒否は発生しなかった。

## 範囲外の気づき

Element（接点等）とImage/GroupFrameで矢印キーの修飾キー割当が異なる（前者はCtrl+矢印キー、後者は無修飾）という仕様に検証序盤で気づかず、無修飾矢印キーだけでM3要素を動かそうとして「動いていないように見える」状態に一時遭遇した。コード確認（`MainWindowViewModel.cs`のXMLコメント）で意図的な既存仕様と判明、忍者の誤操作であり実装側の不具合ではない。範囲外のため`docs/todo.md`化・修正提案はしない。
