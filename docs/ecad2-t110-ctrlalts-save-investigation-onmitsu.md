# T-110 「Ctrl+Alt+S保存が旧4ファイル形式のまま」独立調査（隠密）

調査日: 2026-07-22 03:12頃（JST）　担当: 隠密　委任元: 家老（Wチェック原則、侍の修正と並行）
現象報告（忍者→家老経由）: Ctrl+Alt+S保存で`main-layout.xml`が生成されず、旧4ファイル
（left-palette.xml等）が「保存操作と同一秒で更新」され、中身も旧形式のまま。

## 結論（先出し）

1. **新コード（a78b802/e1c8f73）に旧4ファイルを書く経路は存在しない**（侍の自己申告と独立に
   一致確認）。
2. **旧4ファイルは本日の検証時間帯に一切更新されていない**（ファイルシステム実測: 4本とも
   最終更新2026-07-21 18:30:43=昨日夕刻）。「保存操作と同一秒で更新された」という観測は
   現存する物理証拠と矛盾する——**4本が同一秒を共有する（昨日の旧バイナリWindow_Closing
   一括保存の痕跡）のを、本日の保存によるものと誤読した可能性が高い**（推測、忍者へ
   日付込みの再確認を要請）。
3. 「忍者が旧ビルドを使っていた」仮説（家老追加情報）も**現物証拠と不整合**——旧ビルドで
   Ctrl+Alt+Sを押していたなら旧4ファイルに本日の時刻が付くはずだが、付いていない。また
   現在実行中のEcad2.Appプロセス（PID 14328、03:09:42起動）は03:09:41ビルドの新バイナリである。
4. **真の異常は「Ctrl+Alt+Sを押しても何も書かれない」こと**（main-layout.xml不存在が唯一の
   実害）。原因候補2つを§3に提示、いずれも忍者の1操作で切り分け可能。

## 1. 確定事実（出典付き）

- **静的**: `src/`全域grepで旧ファイル名（left-palette/output-panel/right-panel/
  placement-toolbar.xml）の出現は`MainWindow.xaml.cs`129行・503行の**コメント2箇所のみ**。
  `SaveDockingLayoutAsDefault`（.cs:445-464）の書き込み先は
  `DockingLayoutFilePath`=`%AppData%\Ecad2\docking-layout\main-layout.xml`（定数、131-134行）
  ただ1つ。旧`GetDockingLayoutFileName`は消滅済み（テストも定数検証へ書き換え済み）。
- **ファイルシステム実測**（2026-07-22 03:12:04時点）:
  - `left-palette.xml`/`output-panel.xml`/`placement-toolbar.xml`/`right-panel.xml`:
    最終更新**2026-07-21 18:30:43（4本とも同一秒）**
  - `main-layout.xml`: **不存在**
  - 増分1の検証は本日02:52頃以降に開始（家老の引き継ぎ時刻より）。検証時間帯に旧ファイルへの
    書き込みは物理的に発生していない。
- **プロセス実測**: 実行中Ecad2.Appは PID 14328、起動03:09:42、バイナリ
  `src\Ecad2.App\bin\Debug\net10.0-windows\Ecad2.App.exe`（exe/dllとも03:09:41ビルド=増分1
  最終形）。旧バイナリのプロセスは現存しない。

## 2. 観測の再解釈（推測、要忍者再確認）

4本のファイルが**同一秒（18:30:43）**を共有するのは、旧バイナリの`Window_Closing`自動保存が
4Managerをループで連続書き込みした痕跡として自然（昨日夕刻のセッション終了時刻とも整合）。
ディレクトリ一覧で「4本が揃って同じ秒に更新されている」のを見ると、直前の保存操作の結果と
誤読しやすい——**日付部分の確認が抜けると特に**。忍者へ「観測時にタイムスタンプの日付まで
確認したか」を照会されたい。

## 3. 真の異常（main-layout.xml不生成）の原因候補と切り分け手順

- **候補1（本命寄り）: OverlayWindow残存バグとの連関——フォーカスゲートで保存ハンドラが
  黙って弾かれた**。`OnGlobalDockingLayoutShortcut`（.cs:158）は
  `sender is not (MainWindow or LayoutFloatingWindowControl)`なら無言でreturnする。同時報告の
  「十字型ドロップターゲット残存」状態では、残存OverlayWindow（独立したWindow、上記2型の
  いずれでもない）がアクティブ/フォーカス保持のままCtrl+Alt+Sを受けた可能性があり、その場合
  **ステータスバーに何も表示されず、何も書かれない**（現象と完全一致）。
- **候補2: `HasExpectedContent`スキップ**。保存前検証（.cs:449-453）が偽なら
  「パネルレイアウトが不完全な状態のため保存をスキップしました」を表示して何も書かない。
  Float/Dock往復（検証項目6、OverlayWindow残存が起きた操作）後にレイアウトモデルが汚損して
  いれば成立しうる。
- **切り分け手順（忍者へ、計2分）**:
  (1) アプリ再起動直後（Float/Dock等の操作前）にCtrl+Alt+S→ステータスバー文言と
  `main-layout.xml`生成有無を確認。**生成されれば静的欠陥なし**（保存機構自体は健全）と確定し、
  残るは「OverlayWindow残存後にのみ保存が死ぬ」動的問題（候補1/2いずれかは文言で判別:
  無言=候補1、スキップ文言=候補2）。
  (2) 生成されなければ静的欠陥の筋が復活——その場合は拙者が再調査する。

## 4. 副次所見（本件と独立、侍へ申し送り推奨）

**起動時読込の退行**: 旧`LoadDockingLayoutFromFileIfExists`は`if (savedXml is null) continue;`
——**ファイル無し時は一切Deserializeしない**設計だった（a78b802^:506-527実確認）。新実装
（.cs:477-494）はファイル無し時も既定XMLの`TryDeserializeDockingLayout`へフォールスルーし、
XAML初期状態から構築済みの全モデルを起動のたび無意味に再構築する。実害は軽微と見られるが、
(a)不要なモデル差し替え、(b)E-1（起動時選択タブが「基本機能」になる件）へSerialize/Deserialize
往復の変数を1つ加える、の2点で旧ガード（`savedXml is null`なら即return）の復元を推奨する。

# 第2報（2026-07-22 03:20頃、忍者増分1検証記録の精読+再実測を受けて）

## 追1. 「旧4ファイルが保存で更新される」は起きていない（物理証拠で確定）

忍者報告（`docs/ecad2-t110-increment1-verification-ninja.md`2.3）の記述を精読すると、実測内容は
「保存操作と同一秒（**実測: 4ファイルとも同一タイムスタンプ**）」——括弧内が示すとおり、
**実測されたのは「4本が互いに同一秒」であり、「その秒＝保存操作の時刻」は実測されていない**。

拙者の2回の実測（03:12:04と03:17:12）で、旧4ファイルは**2026-07-21 18:30:43（昨日夕刻）から
不変**、本日の検証時間帯の更新はゼロと確定。「4本が互いに同一秒」は旧バイナリの
`Window_Closing`一括保存（昨日の終了時）の痕跡であり、本日の保存によるものではない。
「中身が旧形式」も当然（昨日のファイルそのものゆえ）。

**帰結: 「実行時に旧4ファイルを更新する経路」は存在しない**（コード面=grep確定、物理面=
タイムスタンプ確定の両面）。忍者の反証（新バイナリ使用の物証）とも何ら矛盾しない——
「ビルドは新しい」と「旧ファイルは今日書かれていない」は両立し、真の異常は一貫して
**「保存が何も書かないこと」**である。

## 追2. 終了時自動保存についての新事実と留保

03:17:12時点でEcad2.Appプロセスは消滅済み（03:09:42起動の新バイナリインスタンスが終了）だが
`main-layout.xml`は生成されていない。`Window_Closing`配線は健在（`MainWindow.xaml:16`
`Closing="Window_Closing"`・`.cs:1193-1202`で`SaveDockingLayoutAsDefault()`呼出、旧コードと同一）
ゆえ、終了時保存の不発は次のいずれか:
- (a) `ConfirmDiscardIfDirty()`偽→`e.Cancel`（ただしプロセスは消えているため該当しない見込み）
- (b) **検証自動化でのプロセス強制終了（Stop-Process等）**——`Window_Closing`自体が走らない
  （忍者の後片付け手順次第、要確認）
- (c) `HasExpectedContent()`偽→スキップ（下記追3）
**忍者へ「アプリをどの方法で終了したか」の確認を要請されたい**（(b)なら終了時保存の不発は
バグではなく手順由来）。

## 追3. 残る本丸=「Ctrl+Alt+Sで何も書かれない」の候補2択と最短切り分け

- **候補A: `HasExpectedContent()`偽→黙ってスキップ**。成立させうる唯一の静的経路は§4の退行
  ——起動時、保存ファイル無しでも既定XMLを`TryDeserializeDockingLayout`で丸ごと再構築する
  フォールスルー。この往復（Serialize→Deserialize→RebindDockingContent）で7要素のうち
  1つでもContent再バインドに失敗していれば、**以後の保存は全て（Ctrl+Alt+S・終了時とも）
  無言でスキップされ続ける**（現象と完全一致する決定論的な筋）。
  ※Ctrl+Alt+S時はステータスバーへ「…保存をスキップしました」が出るはず——忍者報告に
  文言への言及が無いのは「見ていなかった」可能性と「出ていなかった」可能性の両方がある。
- **候補B: senderゲート無言return**（`.cs:158`、フォーカスがMainWindow/フロート窓以外——
  第1報候補1）。この場合は文言も出ない。
- **最短切り分け（忍者、1分）**: アプリ再起動直後（他操作前）にCtrl+Alt+S→
  (1)ステータスバー文言 (2)`main-layout.xml`生成有無。
  文言「スキップしました」→候補A確定（→侍の診断ログは`HasExpectedContent`の戻り値と
  偽時の欠落ContentId列挙を仕込むのが最短）。文言なし・生成なし→候補B系。生成あり→
  静的欠陥なし（操作起因の動的問題へ切替）。
- **侍の計装への提案**: `HasExpectedContent`へ「偽の場合、`_expectedContentIds`のうち
  presentIdsに無いIDをログ出力」の1行を足せば、候補Aの真偽と欠落要素の特定が同時に済む。

# 第3報（2026-07-22 03:25頃、忍者の最小条件切り分け結果「無表示・不生成」を受けて）

## 追4. 「ステータスバー無表示」の含意——SaveDockingLayoutAsDefault未到達が確定

`SaveDockingLayoutAsDefault`（.cs:445-464）は**全分岐が必ずステータス文言を出す**
（スキップ/成功/失敗の3種）。「無表示」はすなわち**同メソッドが呼ばれてすらいない**ことの
証左であり、疑いは`OnGlobalDockingLayoutShortcut`（.cs:149-164）の入口〜ゲート4段
（修飾キー判定/キー判定/senderゲート/MainWindow取得）に限定される。候補A
（HasExpectedContentスキップ）は**Ctrl+Alt+S経路については棄却**（スキップなら文言が出る）。

## 追5. 家老仮説（senderゲートの恒常的問題)への検証結果——コード比較では支持されない

`OnGlobalDockingLayoutShortcut`は増分1で**一字も変わっていない**（B-4で新旧一致確認済み）。
senderゲートを含む4ゲートの判定材料（フォーカスの所在＝MainWindow、修飾キー、
Application.Current.MainWindow）も、起動直後の最小条件では旧実装と同一の条件であり、
旧実装時代にCtrl+Alt+R/Sが機能していた実績（T-058増分4忍者確認・T-099(c)殿実測）と
「新コードでのみ恒常的にゲートが弾く」は両立しない。**恒常的なコード欠陥説は現時点の
静的証拠では支持されない**。

代わって有力なのは**合成キー入力の配送問題**（忍者手法起因の疑い）:
- 忍者自身がPoC検証2.2で確立した「UIA合成操作は実イベントを発火させないことがある」罠
  （TogglePattern.Toggle()がClickを発火せず）の**キーボード版**の可能性。
- 技術的裏付け: Alt絡みの合成送出は順序次第でWPF側が`Key.System`扱いにする
  （Ctrl成立前にAlt+Sが届くとWM_SYSKEYDOWN→`e.Key==Key.System`となり、
  `e.Key != Key.R && e.Key != Key.S`ゲートで**無言return**——症状と完全一致）。
  物理キーボードのCtrl+Alt+SはCtrl押下がAltのシステムキー化を抑止するため通常起きない。

## 追6. 決定打の切り分け実験（優先順）

   - 生成OK→保存機構・HasExpectedContentとも健全と証明。問題はキーイベント配送のみに限定され、

   - 無表示・不生成→Click配線レベルの異常（可能性低、XAML配線は確認済み）
2. **物理キーボードでCtrl+Alt+S**（殿または忍者の物理操作）: 成功すれば合成入力起因と確定
