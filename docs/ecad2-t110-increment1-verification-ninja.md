## 2. 重大所見

### 2.1 【重大・要注意】ドラッグ&ドロップ経由のDockで十字型オーバーレイがハングする

配置ツールバーをFloat()後、フロートウィンドウのタイトルバーをドラッグしてメインウィンドウ上部
へ戻そうとしたところ、十字型ドロップターゲットのオーバーレイが表示された状態のまま消えなく
なった。以下5パターンを試したがいずれも解消せず:
1. マウスLEFTUPの再送
2. その場での左クリック(LEFTDOWN+LEFTUP)
3. メインウィンドウへのフォアグラウンド化後Escキー送信
4. 離れた位置(キャンバス中央付近)でのクリック
5. オーバーレイ中央のドッキングアイコンへの正確なクリック

UIA自体は応答していた（ステータスバー取得は成功）が、**CopyFromScreen(実画面キャプチャ)で確認
したところ、オーバーレイは実際に画面上へ表示され続けていた**(PrintWindowの撮影アーティファクト
ではなく実態)。プロセス再起動で解消し、以降の検証は支障なく継続できた。

合成マウスドラッグ（`SetCursorPos`+`mouse_event`合成）特有の限界（AvalonDock内部の`DragService`
が正規のドラッグ終了を検知できない）か、隠密レビュー観2で言及された「T-103ドロップ枠の判定
範囲がManager全体へ拡大」する仮実装自体の不備かは、忍者の手法だけでは判別不能。家老采配により
侍への診断ログ計装を依頼済み（対応中）。

### 2.2 (観1)配置ツールバー上段のラベル非表示が発火しない

配置ツールバー2タブ(基本機能/配置ツール)は帯(タイトルバー相当)自体は表示される。ただし
Float()させた際、フロートウィンドウのタイトルバーには「配置ツール」というラベルが表示された
まま消えなかった。PoC増分0検証（`docs/ecad2-t110-poc-verification-ninja.md`2.5）と同一現象。
フロートウィンドウのボタン構成(`SinglePaneContextMenu`/`PART_PinMaximize`/`PART_PinClose`)を
確認したところ、ドッキング時単独ペイン(シート等)で使われる統合タイトルスタイル
(`ContentId`分岐を含む)とは異なる別のチェコンテナが使われている模様——本実装でも
`Model.ContentId=="PlacementToolBar"`のラベル非表示トリガーが実際に発火する場面は今回の
操作範囲内では確認できなかった。隠密レビュー観1の懸念どおり、実害はない（ラベルが出ても
機能に支障なし）が、設計意図どおりに機能していない点は申し送る。

### 2.3 【重大バグ、訂正版】レイアウト保存でmain-layout.xmlが生成されない

**訂正（隠密の独立調査で発覚、2026-07-22）**: 初報では「旧4ファイルが保存操作と同一秒で更新
されていた」と記載したが、これは**日付を見落とした誤判定**だった。旧4ファイルの実際の最終更新は
2026-07-21 18:30:43（検証セッションより前の時刻）であり、今回のCtrl+Alt+S操作とは無関係。
訂正の上、以下が正しい所見。

Ctrl+Alt+Sでレイアウト保存を実行しても、`%AppData%\Ecad2\docking-layout\main-layout.xml`は
**生成されない**。旧4ファイルも更新されない（無反応）。**切り分け実験**（アプリ再起動直後、
他の操作を一切行わずCtrl+Alt+Sのみ実行）でも同様に再現——main-layout.xml不生成、旧4ファイルも
無変化、ステータスバーにも保存関連の文言表示なし（無表示）。この結果は、ドラッグ&ドロップ等の
動的要因を経由せずとも再現するため、**保存機構自体が静的に壊れている可能性**を示す
（隠密所見、詳細は`docs/ecad2-t110-ctrlalts-save-investigation-onmitsu.md`参照）。

隠密静的レビューのB-1判定（`DockingLayoutFileName="main-layout.xml"`定数の存在を確認、OK）は
定数の存在確認に留まり、実際の保存処理が呼び出されているか（イベントハンドラの購読漏れ等）
までは追えていなかった可能性がある。侍のコード確認要、家老采配で対応中。
