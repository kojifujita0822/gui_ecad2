## 1. GuiEcadの画面構成（現状把握）

### 1.1 ウィンドウ構造

`MainWindow`（タイトルバー＋`Frame`のみの薄いシェル）→ `RootFrame.Navigate(MainPage)` で単一ページ
`MainPage` に遷移し、以後Navigateは呼ばれない。**画面遷移の仕組み（Frame/Navigate）は用意されているが
実質使われておらず、設定画面等も別ページ化せずダイアログで実装されるシングルページアプリ**。
独立ウィンドウは自作パーツエディタ`PartEditorWindow`のみ。

### 1.2 メインエリアのレイアウト（実機確認で裏取り済み）

```
[メニューバー: ファイル|編集|図面|表示|図形|ヘルプ]
[標準TB: テスト|Undo/Redo|削除|接続検査|ズーム|列±|行±|PDF]
┌──────┬───────────┬────────────────────┬────────────────┐
│縦    │ナビツリー  │ Win2D CanvasControl │[タブ]機器表    │
│パレット│(シート一覧)│ （ラダー図作図エリア）│    /プロパティ │
│(Auto) │(180px)    │        (*)          │  (220px, 開閉) │
└──────┴───────────┴────────────────────┴────────────────┘
[下部出力パネル: 回路エラー|検索結果|接続検査タブ、40px+折りたたみ]
[ステータスバー: 作画/テストモード | 行列位置 | シート番号 | 未保存表示]
```

右パネルは幅0⇔220pxのアニメーションで開閉（右端28px常設トグルバー）。列幅は180px/220px固定、
可変幅（`*`）は作図領域のみ。**GridSplitter等のリサイズ可能なスプリッターは使われていない。**

実機のUI Automationツリーで確認した限り、**作図キャンバス（Win2D CanvasControl）はツリー上に
要素として一切現れない**（Total 120要素中、名前付きコントロール以外は列挙不可）。これは後述の
「悪かった点」および要件R9と直結する。

### 1.4 ツールパレット・ドック機構

`enum PaletteDock { Left, Float, Top, Bottom }` による自前実装の可変ドッキング機構。
Left（ドック）／Float（自由移動、既定）／Top・Bottom（画面端40px以内へドロップで吸着、横並びに
変形）の4状態を持ち、位置は `palette-pos.txt` へ永続化・起動時復元。外部ライブラリなしで
Jw_cad風の可動パレットを実現している（`docs/float-palette-plan.md`）。

### 1.5 ダイアログ一覧

すべて`ContentDialog`。WinUI3の「同時に複数ContentDialogを開けない」制約への対策として、
`SemaphoreSlim`ラッパーで直列表示を保証（`ShowDialogAsync`）。未保存確認／ドキュメント情報／
シート設定／部品リスト(BOM)／PDFプレビュー／使い方／バージョン情報／エラー表示／テンプレート関連／
オートセーブ設定／ショートカットキー設定／シート名変更／自作図形管理、等。

## 2. 操作フロー（現状把握）

- **新規/起動→編集→保存**: 起動時`XamlRoot`確定待ちのポーリング実装あり（オートセーブ復元ダイアログの
  サイレント失敗対策）。新規/開くは`ConfirmDiscardIfDirtyAsync`を経由（ただし後述の抜け穴あり）。
- **配置**: 単一の`ToolState`（`ToolMode` enum、record struct）でツール選択を集約→キャンバスクリックで
  配置。同一行の隣接要素は自動横配線。
- **機器名・コメント編集**: 配置直後/選択中にEnterで機器名、F2でコメント、右母線ダブルクリックで
  行コメント。いずれもキャンバス座標に合わせて動的配置する重ね合わせTextBoxで実現。
- **プロパティ編集**: 選択時に右パネル「プロパティ」タブが自動アクティブ化。
- **モード切替（作画⇔テスト）**: `ToggleButton`一つで切替。テストモード中はツールバー色変化・
  右クリックメニュー分岐（接点強制ON/OFF）・タイマ実時間計時パネル表示。
- **キーボード配置モード**: 矢印キーでセル移動・数字キーでツール切替・Enterで配置・Escで終了、という
  設計だったが、**WinUI3のフォーカス制御バグ（Issue #6179）により2026-07-01付けで入口を非表示化**
  （ロジックは残置、実質使用不可）。
- **検索・置換／DRC／接続検査**: いずれも下部出力パネルの対応タブと連動する独立フロー。
- **シート管理**: `NavTree`（TreeView）で切替、＋/－/名前ボタンで追加・削除・改名。
- **自作パーツ・図形**: 独立ウィンドウ`PartEditorWindow`で作成・編集、`Saved`イベントで即メニュー反映。

## 3. 良かった点

1. **業務ソフト踏襲のレイアウト**: GX Works3・Jw_cad等、対象ユーザーが慣れた既存CADの操作感を
   意図的に踏襲（左パレット・中央キャンバス・右機器表/プロパティ・下部出力パネル・ステータスバー）。
2. **可動ツールパレット**: ドック/フロート/上下吸着の4状態＋位置永続化を外部ライブラリなしで実現。
3. **モードの視認性**: 作画/テストモードでツールバー色・右クリックメニュー・ステータス表示が明確に
   切り替わる。
4. **状態集約の実例**: ツール選択状態を単一`ToolState`（enum）に集約済み（後述のGodクラス問題の中で
   数少ない成功したリファクタ）。
5. **フレームワーク制約への丁寧な回避策**: ダイアログ多重表示防止、`Closing`非同期処理回避策などが
   コード・コメントとして文書化されている。
6. **選択種別ごとのヒント表示、DRC/検索結果のハイライトジャンプ**等、細部のフィードバックUIが
   継続的に改善されてきた形跡（CHANGELOG参照）。

## 4. 悪かった点

1. **未保存変更の破棄確認に抜け穴がある**（コードレビュー指摘・優先度「高」）: `New`/`Open`は
   Dirty判定できるのに確認なしで文書を差し替えている経路があり、×ボタンクローズも同様のリスクを
   非同期回避策でようやく塞いでいる状態。
2. **エラー表示にスタックトレース全文が出る箇所がある**（一般ユーザ向けに不適切）。
3. **「Godクラス」化**: `MainPage`本体+partial群で計約4000行。数十個のprivateフィールドをpartial間で
   共有する設計で、状態管理の一元化・テスト容易性に課題（3段階リファクタ計画のうちフェーズ1のみ実施）。
4. **Escapeキーが7つの意味を持ち、優先順位がif-else連鎖の順序に依存**（ユーザーから見て挙動予測が
   困難になりうる典型パターン）。
5. **UI状態同期の細かいバグが繰り返し発生**: ダイアログ多重表示、配置ツール→選択ツール切替時の
   ゴーストプレビュー残留、Undo直後の選択枠クリア漏れ、等。
6. **Win2D CanvasControlはUI Automation/アクセシビリティツリー上で単一の画像要素にしかならない**
   （忍者の実機確認でも追認: ツリー上にキャンバス要素自体が現れない）。作図結果の自動検証もキーボード
   操作の自動化検証も及ばず、支援技術対応の構造的制約になっている。
7. **キーボード配置モードが実質使用不能**: WinUI3のフォーカス制御バグ（Issue #6179、Microsoft
   "not planned"）に阻まれ、マウスレス操作という当初目標を達成できないまま入口を隠して据え置き。
8. **右パネルがタブ排他式**（機器表/プロパティを同時参照不可）、**リサイズ不可の固定幅**
   （180px/220px）。

## 5. R1〜R10との突き合わせ

| 要件 | GuiEcadでの実態 | ecad2への示唆 |
|---|---|---|
| R1 フォーカス一元性【最重要】 | Issue #6179によりPointerReleased後の暗黙フォーカス移動を制御できず、キーボード配置モード・枠ラベル編集の2箇所で実害。技術選定側はWPFで対策済み（T-002/T-006 PoCで`FocusManager`＋`PreviewLostKeyboardFocus`のe.Handled、タブ切替のブロックまで実証） | UI/UX設計面でも「編集中に別要素へフォーカスが移りうる操作（タブ切替・パネル切替・ダイアログ表示）」を最初から意識して設計する。GuiEcadは後付け対応で苦労した教訓を活かす |
| R2 グローバル/ローカル分離 | 機器名/コメント編集は重ね合わせTextBoxで実現、数字キーでのツール切替は配置モード限定。テキスト編集中とグローバルショートカットの共存は概ね機能していた | 同様の「重ね合わせ編集UI＋グローバルショートカットは別モード」というパターンは踏襲候補 |
| R3 押下/解放の両方を検知 | Spaceパンを結局PageUp/PageDown離散スクロールに変更し回避 | ecad2では技術選定段階でKeyDown/KeyUp両方が確実に取れることを確認済み（WPF）。連続操作系UIを最初から設計に含めてよい |
| R4 キャンバスへの確実なフォーカス取得 | Win2D CanvasControlは既定でフォーカス非取得、対応忘れが不具合の温床に | WPFの`DrawingVisual`/`FrameworkElement`ベースなら`Focusable=true`の明示のみで済み、設計時にチェック項目化する |
| R5 パネル間フォーカスナビゲーション | 明確な設計記述は見当たらず、右パネルのタブ切替・下部出力パネルとのTab順は未検証 | ecad2では主要パネル間のTab順序を明示的に設計する（GuiEcadで手薄だった領域） |
| R6 Enter/Escの一貫した意味論 | Escapeが7つの意味を持ち順序依存という悪い実例が存在 | ecad2ではコンテキスト（配置中/編集中/選択中/ダイアログ表示中）ごとにEnter/Escの意味を表で明文化してから実装する |
| R7 モーダルUIの入れ子制約 | ContentDialogの多重表示不可という制約に直列化ラッパーで対応済み（GuiEcadの数少ない「解決成功」例） | WPFの`Window`はモーダル入れ子に強いが、同様の直列化ガードパターンは流用候補 |
| R8 無効化とフォーカス可能性の分離 | GuiEcad側で明示的な言及は見当たらず | ecad2では`IsEnabled=false`によるフォーカス喪失を避ける設計（フラグ管理での多重防止等）を最初から採用 |
| R9 テスト容易性・自動化親和性 | Win2D Canvasは単一画像要素化、SendInput/SendKeysも届かずUI Automation経由のボタン操作止まり（忍者実機確認でも同様の制約を別途確認） | WPFの`DrawingVisual`は個別Automation Peerを実装する選択肢もある。テスト自動化の観点からアクセシビリティツリー露出を設計時に検討する価値がある |
| R10 既知バグデータベースの成熟度 | Issue #6179を選定段階で把握できず9ラウンドの手戻りが発生 | ecad2はWPF選定前にPoC（T-002/T-006）で先回り検証済み。同種の手戻りは回避できている |

## 6. 残すべき操作パターン

- 全体レイアウトの基本形（左パレット・中央キャンバス・右機器表/プロパティ・下部出力パネル・
  ステータスバー）。対象ユーザー（既存CAD経験者）の学習コストを下げる効果が実証済み。
- 作画/テストモードの明確な視覚的フィードバック（色変化・右クリックメニュー分岐）。
- ダイアログの直列化ガード（多重表示防止）という設計パターン自体。
- 選択対象種別ごとのヒントテキスト切替、DRC/検索結果からキャンバスへのハイライトジャンプ。
- 可動ツールパレット（ドック/フロート/上下吸着）の**機能自体**は評価が高いため継承候補
  （実装は自前実装ではなくWPFの標準機構で代替できないか技術検討の余地あり）。
- 機器名・コメントの「重ね合わせ編集UI」パターン。

## 7. 変えるべき点

- **未保存確認は全経路（New/Open/×ボタン/アプリ終了）で抜け漏れなく実装**する。後付けでなく設計の
  最初から「Dirty判定→確認」を横断的な共通処理として組み込む。
- **状態管理はenum/ステートマシンで一元化**し、フラグ変数の束を避ける。GuiEcadの`ToolState`成功例を
  出発点として、Godクラス化を防ぐ責務分割を初期設計に含める（MVVM等）。
- **キーボード配置モードは正式機能として最初から設計に組み込む**。WPF選定＋T-002/T-006 PoCにより
  フォーカス制御の技術的な実現性は確認済みのため、GuiEcadのような「後から追加して頓挫」ではなく
  最初からR1〜R10を満たす設計で臨める。
- **キャンバスのアクセシビリティ**: WPFの`DrawingVisual`で個別要素にAutomation Peerを持たせるか、
  最低限テスト自動化に必要な情報（選択状態・座標）を別経路で公開するか、設計時に検討する。
- **右パネルは可変幅（GridSplitter等）・複数同時参照可能なレイアウト**を検討する。タブ排他式・
  固定幅という制約を無条件に踏襲しない。
- **エラー表示はユーザー向けメッセージとログを分離**し、スタックトレース全文は開発者向けログのみに
  出す。
- **Escapeキー等、複数モードで意味を持つキーは、コンテキスト×意味の対応表を設計時に明文化**してから
  実装する（if-else連鎖の順序依存を避ける）。

## まとめ

GuiEcadの画面構成・操作フローは「既存CADの操作感を踏襲する」という方針自体は成功しており、
ecad2でも基本レイアウトは継承候補となる。一方、**状態管理の分散（Godクラス化）とWinUI3の
フォーカス制御バグの2点が、UI/UXの機能追加・安定性の両面で最も足を引っ張った**。ecad2では
技術選定（WPF）によりフォーカス制御の課題は解消の見込みが立っており（T-002/T-006 PoCで実証済み）、
残る課題は状態管理の設計を初期段階からenum/ステートマシン・責務分割で作り込むことに集約される。
