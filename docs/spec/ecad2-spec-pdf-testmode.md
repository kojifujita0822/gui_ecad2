## 1. PDF出力（T-060）

### Core層のAPI（追加実装不要）

- `PdfRenderSurface`（`src/Ecad2.Pdf/`）：ページ単位の描画面。**Document/Sheetを跨ぐページ走査
  オーケストレーションを担うクラスは存在しない**——呼び出し側（App層）がループを書く必要がある。
- `DiagramRenderer`は複数ページ機能を完全実装済み：`RenderPageCount`（枠あり時の行分割ページ数）・
  `RenderCrossRefPage`（クロスリファレンス専用ページ）・`RenderBomPage`（部品表専用ページ）・
  `CrossRefPageCount`。
- `CrossReference.cs`・`Document.cs`はGuiEcad対応ファイルと**全文一致**（T-007移植時のまま）。
  `CrossReferenceBuilder.Build`・`DeviceTable`・`DocumentSettings.EnableBorder`/`PaperSize`も
  同シグネチャ・同既定値で完全一致。

### 【重要】P-001「PdfExporterが1ページで打ち切られる」問題は解消済み

該当`PdfExporter`クラスは`poc/wpf-focus-poc/`配下のみに存在する簡易実装（独立スパイク）で、本番の
`Ecad2.Pdf`/`DiagramRenderer`とは無関係。**現行アーキテクチャには当てはまらず実質解消済み**。

### GuiEcadの実物結線パターン（移植の参考）

プレビュー→エクスポートの2段階UI：`OnMenuPreviewPdf`→`PdfPreviewDialog`表示→OK→
`OnMenuExportPdf`が全シート走査（`foreach sheet`）＋枠ありページ分割＋クロスリファレンス専用
ページ＋BOM専用ページ→`PdfRenderSurface`書き出し。**出力範囲は「全シート常に」決め打ち**
（現在シートのみの選択肢はGuiEcadに存在しない）。

### App層で新規実装が必要な箇所（4点）

1. PDFエクスポートのオーケストレーションコード新設（Core層API自体は流用可能）
2. 保存ダイアログ（既存の`SaveDocumentAs`と同型パターンで足りる見込み）
3. メニュー/ツールバーへのCommand結線（両方ともXAML要素はあるが`Click`/`Command`が未設定）
4. **プレビュー機能**——GuiEcadはWinUI3の`ContentDialog`＋Win2D `CanvasControl`で自前描画するが、
   WPFにはWin2Dが存在せず単純移植では済まない（開かれた論点2参照）

### 開かれた論点（着手時に殿確認【MUST】、4点）

1. 出力範囲：GuiEcad同様「全シート常に」か、現在シートのみの選択肢を設けるか
2. プレビュー機能の要否・実装時期（v1は直接エクスポートのみに絞る余地あり）
3. 枠あり/枠なし切替のUI（`EnableBorder`はモデル層に既存だがUI操作手段は未確定）
4. エラー時の見せ方（既存パターン踏襲で足りる見込み、実装着手時に侍が確認）

### 裁定根拠

起票元はP-046（隠密、T-055調査時の範囲外の気づき「PDF出力機能がUIに未結線の疑い」）。殿裁定
（2026-07-11）で承認・T-060起票。ロードマップ優先順位3。

## 2. テストモード（T-061）

### Core層のAPI（追加実装不要）

`TestSession.cs`・`Netlist.cs`はGuiEcad対応ファイルとnamespace以外**全文一致**。

| メソッド | 役割 |
|---|---|
| `TestSession(Sheet, PartLibrary?)` | シート単位でセッション生成 |
| `Evaluate()` | 現在の入力＋持ち越した励磁状態で再評価 |
| `SetInput(device, on)` | 入力設定＋即時評価 |
| `ToggleInput(device)` | 入力トグル＋即時評価 |
| `SetPosition(device, notch)` | セレクトSWノッチ位置設定＋即時評価 |
| `Tick(dt)` | 実時間dt秒経過、タイマ加算・評価 |
| `IsEnergized`/`IsOscillating`/`IsCyclic` | 評価結果参照 |

`DiagramRenderer.Render(...)`は`SimState? sim = null`引数を既に持ち、非nullなら通電ネット集合・
機器励磁状態を算出し描画に反映する経路が**既に実装済み**（具体的な配色は未確認、実装着手時に
確認要）。

### GuiEcadの実物結線パターン（移植の参考）

- モード切替：ツールバーの`ToggleButton`（キーボードショートカット割当なし、マウス専用）。切替時に
  `_testSessions`（シートごとの辞書）をクリアして現在シート分のみ再生成、タイマパネル表示切替、
  ツールバー背景色変化。**テストモードOFFで全シート分クリア**（再ON時は全シートまっさら）。
- 入力操作：押しボタンはモーメンタリ（押している間だけON）、セレクトSWはクリックでノッチ順送り、
  その他機器はクリックでトグル。
- 右クリック（接点限定）：「手動でON（強制閉路）」「手動を解除」の2択、実体は`SetInput`呼び出し。
- タイマ実時間計時：`DispatcherTimer`（100ms間隔）＋`Stopwatch`。
- ステータス表示：テストモード中は評価更新のたびにDRCも自動再実行。

### 【重要な設計指針の既存先例】ToolMode enumによる状態一元化

GuiEcadは配置ツールの状態を「複数のboolフラグの束」として実装し、「フラグの取りこぼしバグの
温床になった」（design-brief 3節#1）。ecad2はこれを教訓に`ToolMode` enum + `ToolState` record に
最初から一元化する設計を既に採用している。**GuiEcadのテストモードも`private bool _testMode`という
単純フラグ**であり、この一元化以前の設計思想のまま。ecad2で結線する際、design-briefの既存原則
（状態管理の一元化）に照らすと、独立の上位enum（例：`AppMode{Drawing,Test}`）が既定方針と整合的
と考えられる（隠密所感、断定ではない、設計判断は着手時に確定）。

### 【重要・既存の表示矛盾】F5キー重複の実態と現在の裁定状況

メニュー「テストモード(_M)」には`InputGestureText="F5"`と表示されているが、**F5は既にツールバーの
「a接点配置」で実利用中**。GuiEcad自体はテストモード切替にキーボードショートカットを持たない。
`docs/spec/ecad2-spec-menu-toolbar.md`4節で確認済みのとおり、**「重複」は表記上のみで実行時の
機能競合は起きない**（F5を押すと常にa接点配置が起き、テストモードには絶対に入らない）。

**この問題はP-053として起票済みだが、殿裁定は「pending（T-061着手時に論点1として殿裁定）」のまま
——本仕様書執筆時点で追加裁定・decision化はされていない。** T-061着手時に必ず解消すべき表示上の
矛盾として明記されている。

### 通電表示の配色（GuiEcad比較調査）

`docs/ecad2-ladder-reference-systems-survey-onmitsu.md`によれば、業界で通電表示の配色は割れている：
LDmicro（通電=明るい赤/非通電=グレー）、CODESYS（TRUE=青太線/FALSE=黒太線、赤字「Simulation」
ステータス表示）、OpenPLC（通電=緑ハイライト）、CLICK PLC（true=青ハイライト）。**統一基準はなく、
ecad2独自の配色を決める必要がある**（開かれた論点3）。design-brief63行は「作画/テストモードの
明確な視覚的フィードバック（色変化・右クリックメニュー分岐）」をGuiEcadから残すべき良い操作
パターンとして明記——踏襲が既定方針と一致する。

### App層で新規実装が必要な箇所（7点）

1. モード切替の状態管理（enum/プロパティ新設）
2. モード切替UIの結線
3. 通電表示の色分け設計（DrawingTheme拡張の可能性）
4. 入力操作の結線（ポインタイベントハンドラへの新規分岐）
5. 右クリックメニュー（接点の手動強制、`ContextMenu`新設）
6. 実時間タイマパネル（`DispatcherTimer`は.NET標準機構、WPFでもそのまま使える、移植コスト低）
7. ステータスバー連携（既存の`OutputPanel.RunDrcCommand`との関係整理）

### 段階導入の叩き台（隠密2所感、着手順は家老・侍判断）

第一歩＝モード切替の状態＋Command結線のみ（通電表示・入力操作・右クリックは未実装でも「モードに
入って抜けられる」ことは確認可能）→第二歩＝通電表示色分け→第三歩＝入力操作→第四歩＝右クリック
強制ON/OFF→第五歩＝実時間タイマパネル（後回し候補）。

### 開かれた論点（着手時に殿確認【MUST】、6点）

1. **F5キー重複の解消方針**（メニュー文言修正のみか、実際にキー割当を新設するか）
2. モード管理の設計方式（GuiEcad同型の単純boolか、上位enum新設か）
3. 通電表示の配色
4. モード切替の場所・見た目
5. シートまたぎの状態保持方針（GuiEcadは「OFF→全シート分クリア」を踏襲するか）
6. タイマパネルの要否・実装時期
