# T-021 筆頭PoC（スパイク）— Enter配置のフォーカス復帰検証

作成: 2026-07-04（侍）。家老采配「筆頭PoC（poc/隔離、増分i+iii最小結線）」に対する成果。

## 目的

案X（フル・キーボードファースト）の最大リスク＝**増分(iii)「命名確定後のフォーカス復帰」**が
WPFで成立するかを、本実装（src/）に入る前に最小構成で検証する。GuiEcad(WinUI3)が Issue #6179 で
9ラウンド格闘の末に断念した「ダイアログ/Visual Tree変化後のフォーカス所在制御」がWPFでは
アプリ側でコントロールできるか、を確かめる。

## 検証シナリオ

「選択セルEnter→配置→モーダルダイアログ命名→Enter確定→キャンバスへフォーカス復帰→
続けて矢印→Enterで次配置」の一気通貫ループを3回連続で回し、フォーカスが迷子にならないかを見る。

- **比較条件**: ダイアログクローズ後の `Keyboard.Focus(canvas)` 明示復帰の「あり」「なし」2通り。
  暗黙委譲だけで戻るのか、明示Focusが必須かを判定する。
- **判定基準**: 連続3回の配置ループで (a)フォーカスがキャンバスから外れない (b)命名ダイアログに
  毎回フォーカスが入る、を両方満たせばPASS。

## 実行方法

- **手動（GUI）**: `dotnet run --project T021EnterPlacementPoc` で起動。キャンバスをクリック→
  矢印で移動→Enterで配置→命名→Enter確定、を手で試す。上部チェックボックスで明示Focus復帰を切替。
  右パネルにフォーカス所在のライブ表示と検証ログ。
- **無人（自動判定）**: `dotnet run --project T021EnterPlacementPoc -- --auto --out <ログ出力先>`。
  明示復帰なし→ありの両条件を自動で走らせ、結果を指定ファイルへ書き出して終了する。

| 条件 | 配置成功 | ダイアログ入焦 | 復帰後の矢印移動 | 判定 |
|------|---------|--------------|----------------|------|

## 考察（侍。「確認事実」と「推測」を区別）

- **確認事実**: 案C（モーダル`ShowDialog()`据え置き）の前提では、**明示Focus復帰なし（暗黙委譲）でも
  フォーカスはキャンバスに戻り、一気通貫ループが成立した**。これはWPFのモーダルダイアログの標準挙動
  （`ShowDialog()`が閉じると呼び出し元ウィンドウの直前フォーカス要素へフォーカスが戻る）による。
- **推測**: GuiEcad(WinUI3)が断念した「フォーカス迷子」は、案Cのモーダル前提では**そもそも発生
  しにくい**（モーダルのフォーカス退避・復元がフレームワーク側で保証されるため）。真の非モーダル
  浮動インライン化（T-033）に進んだ時に初めて、隠密が指摘した`PreviewLostKeyboardFocus`+Handled化の
  出番になる可能性が高い。案Cの範囲では明示Focus復帰は「保険」として入れておけば十分と考える。
- **限界**: 本PoCの自動ループはWPFイベント注入（`InputManager.ProcessInput`）ベースの補助計器であり、
  実キー入力とは経路が異なる。隠密調査が警告する「UI Automation偽結果の罠」と同種の懸念があるため、
  **最終判定は忍者の物理キー入力＋T-023 Automation Peerでの二重確認が必要**。

## 侍の推奨（本実装への引き継ぎ案）

1. 本実装（src/）の増分(iii)では、`TryPlaceElement`のダイアログクローズ後に
   `Keyboard.Focus(LadderCanvasHost)` を**保険として明示追加**する（PoCで害がないことは確認済み、
   暗黙委譲が万一効かない環境差への備え）。
2. 増分(i)のEnter配置入口は、プラン素案どおり`Window_PreviewKeyDown`にcase追加＋既存
   `TryPlaceElement`流用で実装する。
3. 本PoCは検証目的達成のため `poc/` に隔離したまま残す（本体src/とは分離）。
